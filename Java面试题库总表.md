# 高并发微服务设计题库

## 1. 如何设计一个读多写少的高并发系统？

### 30秒结论

- 我通常按 `CDN -> Nginx/OpenResty -> 本地缓存 -> Redis -> MySQL` 做分层，核心是多级缓存、热点治理、限流熔断降级和一致性策略。

### 3分钟回答

- 先看流量模型，把绝大部分读请求挡在缓存层，减少数据库直接承压。
- 架构上一般做入口层、应用层、缓存层、数据库层。热点优先本地缓存，其次 Redis，数据库只处理少量未命中请求。
- 一致性上按业务容忍度选旁路缓存、异步刷新或延迟双删，不盲目追求强一致。
- 稳定性要补热点治理、限流熔断降级和监控，至少盯住 QPS、P99、缓存命中率、连接数和错误率。

### 追问答案

- 为什么要加本地缓存，直接 Redis 不行吗？
  本地缓存能进一步降低网络开销和 Redis 压力，特别适合极端热点场景。
- 缓存一致性怎么保证？
  常见做法是更新数据库后删缓存，再配合延迟双删或异步刷新，目标是控制最终一致窗口。
- 如果 Redis 挂了怎么办？
  先限流和降级，再用本地缓存、静态数据或只读模式兜底，避免流量直接打穿数据库。

## 2. 缓存穿透、击穿、雪崩分别是什么？怎么解决？

### 30秒结论

- 穿透是查不存在的数据，击穿是热点 Key 失效瞬间大量回源，雪崩是大批 Key 同时失效或缓存整体故障。常见方案是布隆过滤器、空值缓存、互斥锁、随机过期和限流降级组合。

### 3分钟回答

- 穿透本质是无效请求绕过缓存直接打数据库，通常用参数校验、布隆过滤器、空值缓存处理。
- 击穿发生在热点 Key 上，通常用单飞、互斥锁或逻辑过期加后台刷新控制同一时刻只有一个线程回源。
- 雪崩是大范围缓存同时失效或 Redis 故障，常见应对是随机过期、多级缓存、预热和降级兜底。

### 追问答案

- 布隆过滤器有什么误判？
  可能把不存在的数据误判为存在，但不会把存在的数据误判为不存在。
- 逻辑过期和互斥锁怎么选？
  更看重可用性用逻辑过期，更看重新鲜度用互斥锁。
- Redis 整体不可用时怎么处理？
  先止血做限流降级和静态兜底，再排查故障并补预热。

## 3. 如何处理热点 Key？

### 30秒结论

- 热点 Key 的核心不是查得快，而是不能在失效时把数据库打穿。常见手段是热点探测、逻辑过期、单飞、本地缓存和预热。

### 3分钟回答

- 热点 Key 访问极高、集中度高，一旦失效就容易引发瞬时回源风暴。
- 治理上常见做法是本地缓存、逻辑过期、互斥锁或 singleflight、请求合并和缓存预热。
- 如果热点极端，还可以做 Key 拆分、静态化或异步构建，并且一定要和限流配合。

### 追问答案

- 逻辑过期会不会读旧数据？
  会，本质是用短暂旧数据换系统可用性。
- 热点探测怎么做？
  通过网关日志、Redis 统计、应用埋点或监控平台识别短时间高集中访问。
- 本地缓存会不会有一致性问题？
  会，所以更适合热点读和弱一致场景。

## 4. 缓存一致性怎么保证？

### 30秒结论

- 缓存一致性没有绝对完美方案，核心是在一致性和性能之间做平衡。工程上最常见的是旁路缓存加删除缓存，再配合补偿机制。

### 3分钟回答

- 大多数业务优先用旁路缓存：先查缓存，未命中查库并回填；写时先更新数据库，再删除缓存。
- 这个方案仍然有并发窗口，所以一致性要求更高时会加延迟双删、MQ 异步刷新或 binlog 订阅。
- 关键不是口头说强一致，而是把最终一致窗口、补偿、重试和监控讲清楚。

### 追问答案

- 为什么不直接更新缓存？
  因为缓存不是最终数据源，直接双写数据库和缓存更容易不一致。
- 延迟双删时间怎么定？
  参考业务 RT、缓存回填耗时和线程调度，通常要压测校准。
- 删除缓存失败怎么办？
  要有重试、告警和异步补偿机制，不能假设删缓存必然成功。

## 5. 线程池参数怎么设置？

### 30秒结论

- 线程池参数不能背固定数字，要结合任务类型、接口 RT、下游能力、队列长度和拒绝策略一起定。

### 3分钟回答

- CPU 密集型线程数通常接近 CPU 核数；IO 密集型可以适当放大，但要看下游承载能力。
- 队列不能无限大，否则会把问题从拒绝请求变成排队超时。
- 参数最终要结合压测和监控校准，核心看活跃线程数、队列堆积、任务等待时间、接口 RT 和 CPU。

### 追问答案

- 队列长度怎么定？
  要结合峰值流量、任务 RT、可接受等待时间和内存容量综合定。
- 为什么不把最大线程数设很大？
  线程太多会增加上下文切换、内存占用和下游压力。
- CallerRunsPolicy 适合什么场景？
  适合希望通过反压让上游变慢、且调用线程可承受执行成本的场景。

## 6. 限流、熔断、降级分别怎么理解？

### 30秒结论

- 限流是控制进入系统的流量，熔断是下游异常时主动切断调用，降级是在资源不足时优先保核心功能。

### 3分钟回答

- 限流常用令牌桶、漏桶和滑动窗口，重点是阈值要结合压测和容量定。
- 熔断用于持续失败的下游，避免流量继续冲击依赖。
- 降级偏业务层，核心是保主流程、牺牲非核心功能，比如返回旧缓存、默认值或静态页。

### 追问答案

- 限流阈值怎么设？
  根据历史峰值、压测结果、下游容量和冗余来定。
- 熔断和重试是什么关系？
  重试是单次调用策略，熔断是连续失败后的全局保护。
- 降级后如何保证用户体验？
  先保主流程，再给可接受的兜底结果，而不是直接报错。

## 7. 超时和重试怎么设计？

### 30秒结论

- 原则是先设合理超时，再谨慎重试，并且只对幂等操作重试，同时限制重试次数和总时长。

### 3分钟回答

- 超时一般参考下游 TP95 或 TP99，再留一定冗余。
- 重试只适用于短暂性失败，最好配合指数退避和随机抖动。
- 要控制整条链路的总超时预算，避免多个层级重试叠加放大故障。

### 追问答案

- 超时时间怎么定？
  参考下游 TP95 或 TP99，再结合整条链路总预算来定。
- 哪些场景不能自动重试？
  非幂等写操作，比如扣款、发货、库存扣减。
- 重试和熔断如何协同？
  先限制重试次数和总时长，再让熔断监控整体失败率。

## 8. 如何做服务降级？

### 30秒结论

- 降级的核心是保核心链路、牺牲非核心功能，在依赖异常或资源紧张时优先让主流程活下来。

### 3分钟回答

- 先完成业务分级，不是所有接口都同等重要。
- 常见做法包括返回默认值、返回旧缓存、关闭次要功能、改同步为异步。
- 降级必须可观测、可开关、可恢复，通常结合配置中心动态控制。

### 追问答案

- 哪些功能适合降级？
  推荐、评论、埋点、排行榜、消息通知这类非核心功能。
- 降级和熔断有什么区别？
  熔断偏技术保护，降级偏业务取舍。
- 降级时如何避免用户看到脏数据？
  提前定义哪些场景能接受旧数据，不能接受就返回默认值或明确提示。

## 9. 如何回答线上故障处理？

### 30秒结论

- 我会按 `发现 -> 止血 -> 定位 -> 修复 -> 复盘` 回答，重点体现优先级和系统性修复能力。

### 3分钟回答

- 先看监控和告警，确认是错误率、RT、流量还是资源问题。
- 止血阶段优先限流、降级、熔断、扩容、切流量，先收住影响面。
- 定位阶段结合日志、链路追踪、线程池、连接池、缓存和数据库指标找根因。
- 修复之后要补监控、压测、演练和复盘，避免再次发生。

### 追问答案

- 如果是缓存雪崩你怎么处理？
  先限流降级和本地缓存兜底，再排查 TTL 同时失效还是 Redis 故障。
- 如果数据库 CPU 飙高你先看什么？
  先看慢 SQL、连接数、活跃会话、锁等待和请求量变化。
- 复盘最重要的三项是什么？
  根因、防再发措施和责任到人的落地计划。

## 10. 如何保证接口安全又不影响性能？

### 30秒结论

- 性能优化不能突破安全基线。至少要覆盖鉴权、限频、防重放、签名校验和风控，并把通用能力前置到网关层。

### 3分钟回答

- 常见做法是网关层做认证鉴权、限流和基础校验。
- 防重放通常用 nonce、timestamp 和签名组合。
- 高成本校验要分层做，不是所有接口都一刀切上最重的安全策略。

### 追问答案

- JWT 有什么问题？
  优点是无状态，缺点是签发后不好主动失效，通常要配合短 TTL 或黑名单。
- 防重放为什么不能只靠 timestamp？
  因为在有效时间窗口内仍可能被重复利用。
- 安全和性能冲突时怎么取舍？
  安全基线不能破，但可以分层做，把高成本校验集中到高风险接口。

## 11. 微服务和单体架构怎么选？

### 30秒结论

- 不是所有系统都该上微服务。业务复杂度、团队规模和协作成本不高时，单体通常更稳；当边界清晰、团队拆分和独立演进需求很强时，微服务更合适。

### 3分钟回答

- 单体的优势是开发、测试、部署和排查链路简单，早期推进速度更快。
- 微服务的优势是独立部署、独立扩缩容和团队自治，但代价是分布式复杂度明显上升。
- 所以更合理的思路通常是先单体，复杂度到了再拆。

### 追问答案

- 什么场景更适合单体？
  业务还在快速试错、团队小、系统规模可控时。
- 微服务最大的代价是什么？
  是分布式复杂度，包括调用链变长、治理、运维和一致性问题。
- 为什么不建议一开始就拆微服务？
  因为早期需求和边界不稳定，过早拆分容易把简单问题复杂化。

## 12. 微服务拆分边界怎么定？

### 30秒结论

- 拆分边界核心看业务能力，而不是数据库表怎么分。原则是高内聚、低耦合、独立演进和数据自治。

### 3分钟回答

- 更推荐按业务能力和领域模型拆，比如下单、履约、支付、营销，而不是按技术层拆。
- 拆分时要看职责是否独立、变更频率是否不同、是否需要独立扩容、团队能否独立负责。
- 一旦拆分出去，数据所有权也要明确，不能多个服务随意跨库直连。

### 追问答案

- 为什么不能按数据库表拆？
  因为容易变成数据驱动设计，最后服务之间为了拼业务流程频繁互调。
- 数据边界怎么定义？
  谁对这份数据负责创建、修改和一致性，就应拥有主权。
- 如果拆错了怎么办？
  保留演进空间，必要时合并、重构或先逻辑拆分后物理拆分。

## 13. 服务注册发现是怎么工作的？

### 30秒结论

- 核心是服务启动注册、定期心跳续约、注册中心维护实例列表、消费方订阅并做本地缓存。

### 3分钟回答

- 服务提供者启动后把地址、端口、元数据注册到注册中心。
- 运行过程中通过心跳或租约告诉注册中心自己还活着，超时没续约就会被剔除。
- 消费方通常订阅变更并缓存服务列表，再结合负载均衡选实例。

### 追问答案

- 为什么消费方要做本地缓存？
  降低注册中心压力，提高调用可用性。
- 心跳机制解决什么问题？
  解决实例存活性判断，避免流量命中已挂节点。
- 注册中心挂了会怎样？
  短期内有本地缓存还能调用，但长期会影响实例变更同步。

## 14. RPC 调用的完整流程是什么？

### 30秒结论

- RPC 从业务代码看像本地方法，底层实际经历了服务发现、负载均衡、序列化、网络传输、服务端反序列化和执行、结果返回。

### 3分钟回答

- 调用方先通过代理对象拦截调用，把方法名、参数和服务名封装成请求。
- 根据注册发现和负载均衡选实例。
- 请求序列化后发送到服务端，服务端反序列化并执行目标方法，再把结果回传。
- 真正难的是超时、重试、熔断、连接池、线程池和可观测性治理。

### 追问答案

- 为什么 RPC 不能真当本地调用？
  因为底层有网络，就有超时、抖动、失败和序列化成本。
- 序列化协议怎么选？
  看性能、可读性、跨语言兼容和团队生态。
- 负载均衡起什么作用？
  决定一批可用实例里流量怎么分配。

## 15. Dubbo 和 Spring Cloud 的区别怎么答？

### 30秒结论

- Dubbo 更偏高性能 RPC 治理框架，Spring Cloud 更偏微服务全家桶和生态整合。

### 3分钟回答

- Dubbo 的优势在于 RPC 性能、服务治理和 Java 生态沉淀。
- Spring Cloud 的优势在于和 Spring 体系整合度高，基础设施组件更完整。
- 选型重点不是谁替代谁，而是团队技术栈、治理成熟度和场景。

### 追问答案

- Dubbo 的优势是什么？
  高性能 RPC 和成熟的服务治理能力。
- Spring Cloud 的优势是什么？
  生态完整，和 Spring Boot 配合紧密。
- 现在怎么选？
  看团队栈和场景，纯 Java 内部 RPC 很重时 Dubbo 很合适。

## 16. 接口幂等性怎么保证？

### 30秒结论

- 幂等性的核心是同一个请求重复提交，结果只能生效一次。常见手段是唯一业务单号、幂等表、状态机控制和唯一索引。

### 3分钟回答

- 幂等性最常见在支付、下单、消息消费和重试场景。
- 解决思路是在服务端建立唯一标识，比如订单号、支付流水号、消息 ID。
- 真正难的是并发下不能重复执行，所以经常要配合唯一索引、状态流转和事务。

### 追问答案

- 唯一索引为什么常用来做幂等？
  因为能把只能插一次的约束交给数据库。
- 分布式锁能不能替代幂等设计？
  不能，锁只能控制并发窗口，不能替代业务层重复判断。
- 消息消费怎么做幂等？
  按消息 ID 去重，处理成功后落幂等记录。

## 17. 分布式事务有哪些方案？怎么选？

### 30秒结论

- 分布式事务没有银弹，核心是在一致性、性能和复杂度之间做取舍。常见方案有 XA、TCC、消息最终一致性和 Saga。

### 3分钟回答

- XA 偏强一致，但阻塞和性能代价大，互联网高并发场景通常不是首选。
- TCC 适合核心链路，但业务侵入强。
- 最终一致方案更适合高吞吐和能接受延迟的场景。
- Saga 适合长流程，把大事务拆成多个本地事务加补偿。

### 追问答案

- 为什么很多互联网业务不直接用 XA？
  因为阻塞和性能成本高，运维复杂度也大。
- TCC 最大的代价是什么？
  业务侵入强，还要处理空回滚、悬挂和幂等。
- 最终一致适合什么场景？
  通知、积分、报表同步这类能接受短暂延迟的场景。

## 18. TCC 事务怎么理解？适合什么场景？

### 30秒结论

- TCC 把分布式事务拆成 Try、Confirm、Cancel 三步，适合库存、支付这类对一致性要求高且能抽象出预占语义的场景。

### 3分钟回答

- Try 先做资源预留或校验，比如冻结库存、冻结余额。
- Confirm 在全链路成功后正式提交，Cancel 在失败时释放预留资源。
- 真正难点不是三段名字，而是幂等、空回滚、悬挂和超时补偿。

### 追问答案

- TCC 为什么适合库存和支付？
  因为天然有预占和确认/取消的业务语义。
- 什么是空回滚？
  就是 Try 没真正成功，但 Cancel 先到了。
- 什么是悬挂？
  Cancel 已完成后，迟到的 Try 又执行了，导致状态错乱。

## 19. 基于消息的最终一致性怎么做？

### 30秒结论

- 核心思路是本地事务先成功，再通过消息把后续动作异步推进，失败靠重试和补偿纠正。

### 3分钟回答

- 常见做法是本地消息表或事务消息，确保业务成功了，消息也一定不会丢。
- 下游消费者完成自己的本地事务，比如加积分、发通知、同步报表。
- 关键不是 MQ 本身，而是幂等、去重、重试和补偿链路完整。

### 追问答案

- 为什么不能先发消息再写数据库？
  因为容易出现消息发出去了，但本地事务失败了。
- 消息重复怎么办？
  消费方必须做幂等。
- 如果补偿一直失败怎么办？
  进入死信或人工处理队列，避免无限重试拖垮系统。

## 20. 如何保证分布式系统中的数据一致性和可用性平衡？

### 30秒结论

- 核心不是追求绝对强一致，而是先分清哪些链路必须强保证，哪些链路可以最终一致，再按业务层级做方案取舍。

### 3分钟回答

- 关键交易、支付、库存、账户余额这类链路更偏一致性优先。
- 通知、积分、推荐、报表更偏可用性和最终一致。
- 技术上常见做法是关键链路缩小强一致范围，非核心链路异步、补偿和兜底。

### 追问答案

- 为什么不能所有链路都做强一致？
  因为强一致往往意味着更高延迟和更重的资源占用。
- 一致性边界怎么定义？
  看业务是否能接受短暂延迟、旧数据或异步补偿。
- 如果可用性和一致性冲突怎么办？
  先看这条链路的业务优先级，再明确牺牲哪一边和窗口大小。
