## 高频易混点

- 负载高不等于 CPU 一定高，负载里还可能包含 IO 等待和不可中断任务。
- JVM 问题不等于一定在 JVM，很多时候根因在下游依赖、数据库、磁盘或网络。
- 线程池打满不等于线程数一定要继续调大，先看任务耗时和下游阻塞更关键。
- 内存上涨不等于一定内存泄漏，也可能是缓存膨胀、请求堆积或对象生命周期变长。
- RT 抖动不等于一定代码慢，也可能是 GC、数据库、锁竞争、网络或磁盘问题。
- 容器里看到的 CPU 和内存视角不一定等于宿主机真实资源边界，K8s/Cgroup 限制会影响判断。
- `OOMKilled` 不等于 JVM 一定抛了 `OutOfMemoryError`，前者可能是容器被内核直接杀掉。
- CPU 利用率不高，不代表没有 CPU 资源问题；容器场景下还要警惕 `CPU Throttling`。
- `TIME_WAIT` 多和 `CLOSE_WAIT` 多不是一回事，前者常见于短连接多，后者更要警惕连接没正确关闭。
- 堆内存正常，不代表进程总内存就正常；直接内存、元空间、线程栈和代码缓存也会把进程撑大。

## 云原生视角补充

- 现在很多 Java 服务运行在容器和 K8s 里，排障时要同时区分：
  - 应用自己看到的资源
  - 容器被分配到的资源
  - 宿主机真实资源
- 容器场景里常见但容易误判的问题有：
  - `CPU Throttling`
  - `OOMKilled`
  - 探针误杀
  - 节点层网络或磁盘问题映射到容器服务
- 如果你只会用物理机思维排障，很容易在高级面里被追问卡住。

## 现代诊断工具补充

- `Arthas`：在线观察方法入参、返回值、线程状态、类加载和热点方法，非常适合 Java 服务临时诊断。
- `Async-profiler`：适合生成 CPU/锁/内存分配火焰图，定位热点比只看线程栈更直观。
- `JFR (Java Flight Recorder)`：更像低开销黑匣子，适合抓取线上短时抖动、锁竞争、GC 和线程事件。
- `Grafana + Prometheus`：更适合作为第一观察入口，不要等到 ssh 上机器才开始排障。
- `SkyWalking / Jaeger / Zipkin`：分布式链路追踪更适合先判断“哪一跳慢了”，再决定要不要登录机器。

## 网络深坑补充

- 网络问题不只是“通不通”，很多时候是“抖不抖、丢不丢、重不重传”。
- 如果 RT 抖动但应用层看不出明显热点，要顺手想到：
  - TCP 重传
  - 丢包导致的指数退避
  - `TIME_WAIT` 过多
  - `CLOSE_WAIT` 堆积
- 对高级排障来说，`ss` 只是入口，必要时还要结合 `netstat -s`、抓包和链路指标一起判断。

## JVM 进阶内存补充

- 排内存问题时，不要只盯堆：
  - 堆内存
  - 直接内存
  - 元空间
  - 线程栈
  - 代码缓存
- 如果堆看起来还好，但进程总内存很大，可以顺手想到：
  - `NMT (Native Memory Tracking)`
  - Netty/Direct Memory
  - 类动态生成导致的元空间膨胀

## 排障主线

- 先看现象：
  CPU 高、内存高、RT 抖动、错误率上升、连接数异常、磁盘打满，到底最先暴露在哪一层。
- 再看资源：
  机器资源、JVM 资源、线程池、数据库连接池、网络连接、磁盘和文件句柄。
- 再看链路：
  是业务代码慢、下游依赖慢、缓存回源、锁竞争，还是队列积压。
- 最后做动作：
  先止血，再定位，再复盘，再沉淀监控和预案。

## 全链路观察主线

- 第一入口优先看：
  监控大盘、告警面板、链路追踪，不要默认第一步就是登录机器。
- 第二步再看：
  是单机局部问题，还是某个服务实例、某条调用链、某个下游依赖在抖。
- 第三步才落到：
  Linux 命令、JVM 工具、线程栈、火焰图、连接状态这些细粒度定位动作。
- 高级排障更强调：
  `Metrics -> Trace -> Logs -> 命令行细查`
  而不是一上来 `top + jstack` 全靠猜。

## 设计取舍

- 为什么先看 Linux 资源，不先直接看应用日志：
  很多线上问题会先暴露在 CPU、负载、磁盘、连接数这些系统指标上，先定层比先看局部日志更稳。
- 为什么先缩小范围，不先猜 GC 或 SQL：
  线上问题往往跨层传导，先猜根因容易把自己带偏，先缩范围才能少走弯路。
- 为什么线程池打满不能只靠调大线程数：
  如果根因是慢依赖、慢任务或重试放大，线程数越大只会把问题进一步放大。
- 为什么止血优先于定位：
  线上问题是实时影响业务的，先保核心链路、收敛影响面，比先找最深根因更重要。
- 为什么很多问题要和数据库、缓存、网络一起看：
  Java 服务常常只是问题放大的位置，不一定是问题最初产生的位置。
- 为什么很多时候先看监控和链路，而不是立刻登录机器：
  因为高级排障更强调先判断问题范围，再决定细查入口。
- 为什么容器场景要特别关注资源限制：
  因为 Java 服务看到的“进程状态正常”，不代表没有被容器 quota、内存限制或探针策略影响。

## 能力自测

### 自测题 1

- 某接口 RT 抖动明显，但应用日志没有明显报错。
- 你先回答：
  第一层先看什么，为什么不能一上来就翻业务代码。
- 自测重点：
  你能不能自然想到 CPU、负载、磁盘、线程池、数据库和下游依赖这些分层入口。

### 自测题 2

- 某 Java 进程 CPU 飙高，但整机其他服务基本正常。
- 你先回答：
  这时候第一条排障主线应该怎么走。
- 自测重点：
  你能不能用“进程 -> 线程 -> 栈”这条线把排查顺序讲出来。

### 自测题 3

- 线程池队列越来越长，接口超时越来越多，团队有人建议先把线程池调大。
- 你先回答：
  为什么这个动作不一定对，第一观察点应该是什么。
- 自测重点：
  你能不能主动补任务耗时、下游依赖、重试放大和拒绝策略。

### 自测题 4

- 机器负载很高，但 CPU 利用率看起来并不夸张。
- 你先回答：
  这种现象通常意味着什么，应该优先看哪类指标。
- 自测重点：
  你能不能想到 IO 等待、磁盘、网络和阻塞任务。

### 自测题 5

- 某服务 CPU 利用率不高，但容器 `RT` 周期性抖动，平台侧监控显示 `CPU Throttling` 很严重。
- 你先回答：
  这类问题为什么不能只看 Java 进程 CPU，第一批排查动作应该是什么。
- 自测重点：
  你能不能想到容器 quota、线程池规模、日志输出、锁竞争和 Cgroup 限制。

### 自测题 6

- Java 进程突然消失，没有明显业务异常日志，也没有明显 `hs_err_pid` 文件。
- 你先回答：
  你会先去哪些地方找线索。
- 自测重点：
  你能不能想到 `dmesg`、`journalctl`、容器事件、`OOMKilled`、脚本误杀和平台探针。

### 自测题 7

- 某服务连接数没有明显爆炸，但接口 RT 明显抖动，网络监控显示偶发丢包。
- 你先回答：
  你会如何判断是不是 TCP 重传、短连接过多，还是连接没有正常关闭。
- 自测重点：
  你能不能想到 `TIME_WAIT`、`CLOSE_WAIT`、`netstat -s`、重传和连接生命周期。

### 自测题 8

- Java 进程 RSS 持续上涨，但堆 dump 看起来不算大，GC 也没有明显异常。
- 你先回答：
  这时候为什么不能只盯堆，要往哪几类非堆内存方向看。
- 自测重点：
  你能不能想到直接内存、元空间、线程栈、代码缓存和 `NMT`。

## 实战案例

### 案例 1：接口 RT 抖动，为什么根因最后不是代码本身？

#### 背景

- 某核心接口在高峰期 RT 明显抖动，应用日志没有明显异常，团队第一反应是“某段业务代码退化了”。

#### 暴露出来的问题

- 机器负载上升，线程池排队增加，但 CPU 并没有持续拉满。
- 数据库慢查询增多，下游依赖耗时也被一起拉长。

#### 处理思路

- 先看系统层，确认是不是磁盘、网络、连接或负载异常。
- 再看 Java 进程和线程池状态，确认是否存在排队和阻塞。
- 最后再落到数据库和下游依赖，确认是不是外部慢把应用拖住。

#### 学习重点

- 很多 RT 抖动问题，根因并不在“代码执行慢”，而在依赖变慢和资源争用。

#### 复杂分布式视角补充

- 如果单机指标不算特别坏，但线程池持续排队，要警惕“下游慢但未完全超时”的灰色故障。
- 这类问题常见传导链路是：
  下游变慢 -> 线程占用时间变长 -> 线程池排队 -> RT 抖动 -> 重试放大 -> 整体雪崩。

### 案例 2：线程池打满后，为什么服务会越来越慢？

#### 背景

- 某服务高峰期线程池活跃线程和队列同时逼近上限，请求超时越来越多。

#### 暴露出来的问题

- 线程池看起来“很忙”，但真正根因是下游接口变慢，导致任务迟迟出不了队。
- 部分重试请求进一步放大压力，形成恶性循环。

#### 处理思路

- 先判断是流量太大、任务太慢，还是下游变慢。
- 先止血：暂停非核心任务、限流、必要时熔断慢依赖。
- 再定位：看任务耗时、拒绝次数、下游 RT 和重试放大。

#### 学习重点

- 线程池问题通常是链路问题，不只是线程参数问题。

#### 现代诊断工具补充

- 这类问题可以优先通过：
  - 监控看线程池活跃数、队列长度、拒绝数
  - Trace 看哪一跳最慢
  - 再用 Arthas 或线程栈看卡在什么方法
- 如果只靠“多打几次 jstack”，很容易看到现象却看不到调用链上下文。

### 案例 3：容器 OOMKilled 和 JVM OOM，为什么不能混为一谈？

#### 背景

- 某 Java 服务在 K8s 里频繁重启，业务同学说“是不是 JVM 内存溢出了”，但应用日志里没有明显 `OutOfMemoryError`。

#### 暴露出来的问题

- Pod 被平台拉起又拉起，但 JVM 本身来不及完整打印异常。
- 监控显示容器内存接近 limit，平台事件里出现 `OOMKilled`。

#### 处理思路

- 先区分是 JVM 自己抛 OOM，还是容器被内核或平台直接杀掉。
- 再看堆内存、非堆内存、直接内存、线程栈和容器 limit 的关系。
- 最后结合平台事件、`dmesg`、容器事件、JVM 参数一起判断是不是资源设置或内存模型不匹配。

#### 学习重点

- `OOMKilled` 和 JVM OOM 不在同一层。一个偏容器/内核层，一个偏 JVM 层。高级排障里一定要先分层。

### 案例 4：为什么 CPU 不高，但服务还是被限速了？

#### 背景

- 某服务平均 CPU 利用率并不夸张，但接口 RT 经常抖动，K8s 侧提示容器 CPU 被明显限制。

#### 暴露出来的问题

- 应用层看不出明显热点线程。
- 平台侧监控里 `CPU Throttling` 比例很高，说明容器虽然没“跑满宿主机 CPU”，但已经在容器配额层被限制。

#### 处理思路

- 先确认容器 CPU request/limit 和线程池规模是否匹配。
- 再看是不是存在高频日志、锁竞争、短时突发计算，把有限配额打满。
- 最后判断是资源配置不合理，还是应用层需要减小并发度、降低不必要 CPU 消耗。

#### 学习重点

- 容器场景下，CPU 利用率和“是否被限速”不是一回事。高级开发一定要有 quota 和 throttling 视角。

### 案例 5：为什么一个下游慢调用，最后会拖垮整条链路？

#### 背景

- 服务 A 调 B，B 调 C。C 的响应从平时几十毫秒抖到 2 秒左右，但没有完全超时。

#### 暴露出来的问题

- A、B 两层线程池逐渐排队，请求堆积，RT 和错误率同时上升。
- 从单点看，每个服务都“还能工作”，但整条链路的吞吐越来越差。

#### 处理思路

- 先通过 Trace 看是链路哪一跳先变慢，而不是直接在入口服务里盲猜。
- 再看线程池、连接池、重试策略有没有把慢调用进一步放大。
- 最后决定是快速失败、限流隔离、摘掉慢依赖，还是临时降级掉非核心功能。

#### 学习重点

- 高级排障不只是看单机现象，还要识别“慢调用放大成雪崩”的链路传导。

## 一页速记

- 排障顺序：
  `先现象 -> 再机器资源 -> 再 Java 进程 -> 再 JVM -> 再线程栈和依赖`
- 常见现象：
  `CPU 高看线程和栈，内存高先分类，RT 抖动看跨层传导，负载高别只盯 CPU`
- 止血顺序：
  `先保核心，再限流降级，再切慢依赖，最后深挖根因`

## 高频答题口诀

- CPU 高：
  `先进程，再线程，最后看栈`
- 内存高：
  `先分层，再分类型，别急着说泄漏`
- RT 抖动：
  `先缩范围，再看资源，后看依赖`
- 线程池打满：
  `先看耗时和下游，不先调线程数`

## 典型业务场景

- Java 服务 RT 抖动，应用日志没明显报错，但 CPU 和负载突然升高。
- 应用接口大量超时，最后发现根因不是代码，而是磁盘 IO 或连接数异常。

## 最小实践建议

- 自己找一台开发机，跑一次 `top`、`vmstat`、`iostat`、`ss`、`lsof`，先熟悉输出。
- 练习把“现象”和“应该看哪条命令”对应起来，而不是孤立记命令。

## 1.1 Linux 线上排障时，一般先看哪些指标？

### 核心讲解

- 一般先看 CPU、内存、负载、磁盘 IO、网络连接和进程状态。
- 这些指标能快速告诉你问题更像资源打满、依赖阻塞，还是局部进程异常。
- 线上排障第一步通常不是立刻翻日志，而是先判断机器层有没有明显异常。

### 深入理解

- 这题的核心不是背指标名，而是形成“先看资源画像”的习惯。很多线上问题一开始并不会在业务日志里直接写出根因，但会先在 CPU、负载、内存、磁盘或网络上暴露出异常趋势。
- 所以第一眼先看系统层，能帮你快速把问题缩到更小范围：是整机资源紧张，还是只有一个 Java 进程在异常，还是数据库、网络、磁盘把应用拖慢了。
- 高级 Java 面试里，这题其实是在看你有没有系统排障视角，而不是只会看应用代码。

### 追问答案

- 为什么不先看日志？
  因为很多问题的根因并不先体现在日志里，而是先体现在资源异常和链路阻塞上。

### 学习提示

- 记住一句话：先看整机，再看进程，再看 JVM，再看线程和代码。

## 1.2 CPU、内存、负载、磁盘、网络分别代表什么问题？

### 核心讲解

- CPU 高通常代表计算密集、线程空转、锁竞争或频繁 GC。
- 内存高通常代表对象堆积、缓存膨胀、请求堆积或泄漏风险。
- 负载高通常代表 runnable 线程多、IO 等待高或系统存在阻塞。
- 磁盘和网络异常更常见于日志刷盘、慢 IO、连接异常或下游调用问题。

### 深入理解

- 这题最重要的是把“现象”和“可能的根因层级”对应起来。CPU 高不是一律代码有 bug，内存高也不一定就是泄漏，负载高更不等于 CPU 一定高。
- 高级排障能力，很多时候就体现在你能不能把指标看成“缩小排查范围的入口”，而不是直接等于根因。
- 所以这题建议按“指标 -> 常见原因 -> 典型误判”三层去记。

### 追问答案

- 为什么负载高但 CPU 不一定高？
  因为负载还可能包含 IO 等待和不可中断状态的任务。

### 学习提示

- 这题建议和 [11-操作系统与计算机网络面试题库.md](E:\学习记录\11-操作系统与计算机网络面试题库.md) 对照着学。

## 1.3 `top`、`vmstat`、`iostat`、`ss`、`lsof` 分别适合看什么？

### 核心讲解

- `top` 适合快速看整机资源、进程 CPU 和内存占用。
- `vmstat` 适合看运行队列、上下文切换、系统态和 IO 等待。
- `iostat` 适合看磁盘利用率和 IO 延迟。
- `ss` 适合看网络连接、端口状态和连接堆积。
- `lsof` 适合看文件、端口、句柄占用和进程打开资源。

### 深入理解

- 这题不是工具大全，而是排障链路。通常你不会只用一个命令，而是先 `top` 看大盘，再用 `vmstat`、`iostat`、`ss`、`lsof` 去缩小范围。
- 真正的能力不是记住命令参数，而是知道在什么现象下优先看哪一个。
- 比如 RT 高但 CPU 不高时，看 `vmstat` 和 `iostat` 往往比死盯 `top` 更有效。

### 追问答案

- 为什么 `top` 不够？
  因为它更像总览入口，很多 IO、网络、连接和句柄问题需要其他命令补足。

### 学习提示

- 复习时不要背参数大全，先把每条命令的“最主要用途”记住。

## 1.4 为什么线上排障不能只盯 Java 日志？

### 核心讲解

- 因为很多线上问题根因并不在业务日志里直接出现，而是先体现在系统资源、网络、磁盘、线程和下游依赖上。
- 只盯日志容易把问题看得太窄。
- 更稳的做法是“日志 + 指标 + 线程栈 + 系统状态”一起看。

### 深入理解

- 这题本质上是在纠正一个常见误区：应用日志只是排障信息的一部分，不是全部真相。
- 比如数据库慢、磁盘抖动、线程池堆积、连接数打满、下游依赖超时，这些问题常常不会直接在应用日志里告诉你“根因就是我”。
- 所以高级 Java 的排障方式，一定是跨层的，不会停留在单一日志视角。

### 追问答案

- 什么情况下日志最容易误导？
  当日志只记录结果异常，但真正根因在资源争用、依赖抖动或系统层阻塞时。

### 学习提示

- 记一句话：日志看细节，指标看趋势，线程栈看阻塞，系统命令看资源。

# 2. Java 服务排障主线

## 知识地图

- `机器异常 -> Java 进程 -> JVM -> 线程栈 -> 代码与依赖`

## 学习目标

- 建立 Java 服务的标准排障顺序，不靠拍脑袋乱查。
- 能把 Linux 指标、JVM 指标、线程栈、数据库和下游依赖串起来。

- JVM 基础
- Java 并发基础
- 数据库与缓存基础

## 本模块总结

- 这一组最核心的，不是记命令，而是建立排障主线。
- Java 服务变慢、CPU 高、内存涨，往往都不是单点问题，而是跨层传导。
- 真正成熟的排障能力，关键在于能快速区分问题更像系统层、JVM 层、线程层还是业务依赖层。

## 典型业务场景

- 接口 RT 抖动，应用日志没异常，但用户感知明显变慢。
- 某个 Java 进程 CPU 飙高，机器上其他服务也被带慢。
- 堆内存持续上涨，最后发生 OOM。

## 最小实践建议

- 自己做一次“CPU 高”和“内存高”的排障演练，练习把系统层和 JVM 层串起来。
- 复习时把“现象 -> 第一观察点 -> 第二观察点 -> 常见根因”写成链路。

## 2.1 Java 服务线上排障，一般从哪些主线展开？

### 核心讲解

- 更稳的顺序通常是：先看现象，再看机器资源，再看 Java 进程，再看 JVM，再看线程栈和业务依赖。
- 也就是先缩小问题范围，再深入到代码。
- 不建议一开始就直接猜 GC、猜 SQL、猜网络。

### 深入理解

- 这题其实是在问你有没有“排障框架”。高级 Java 面试官通常更看重你有没有稳定的方法论，而不是某个具体工具。
- 如果没有主线，排障很容易变成“看什么都像问题”。有了主线后，你会更自然地先排除大面，再钻细节。
- 这套顺序的本质，是让你先判断问题在哪一层暴露，再决定用什么工具。

### 追问答案

- 为什么不能一开始就猜根因？
  因为线上问题常常是多层耦合，先缩范围比先猜更稳。

### 学习提示

- 这题建议背成一句话：先现象，后资源，再 JVM，再线程，最后代码和依赖。

## 2.2 Linux 指标和 JVM 指标怎么串起来看？

### 核心讲解

- Linux 指标先告诉你机器层有没有资源异常，JVM 指标再帮助你判断是不是 GC、堆内存、线程或对象分配问题。
- 两者不是替代关系，而是先后关系。
- 更成熟的方式是“先看系统资源，再确认 JVM 是否是根因放大器”。

### 深入理解

- 很多 Java 工程师一看到问题就直接打开 GC 日志或堆 dump，但其实先看机器层会更快。
- 如果整机没问题、只有 Java 进程异常，再深入 JVM 就更有针对性；如果整机已经磁盘抖动、网络阻塞或 IO 等待高，那 JVM 很可能只是被拖慢的结果。
- 所以 Linux 和 JVM 的关系更像“总盘面”和“细化透视”。

### 追问答案

- 什么情况下先看 JVM 更有价值？
  当整机资源没有明显异常，但 Java 进程表现明显异常时。

### 学习提示

- 记住：Linux 指标先定层，JVM 指标再定因。

---

## 2.3 Java 进程 CPU 飙高时，一般怎么排？

### 核心讲解

- 一般先确定是哪个进程 CPU 高，再看线程级 CPU，最后结合线程栈定位具体热点代码或阻塞点。
- 常见根因有死循环、线程空转、频繁 GC、锁竞争和高频序列化计算。
- 排障时不要只看到 CPU 高，就条件反射说“扩机器”。

### 深入理解

- CPU 高这类问题最怕“只看到结果，不去拆线程来源”。真正有效的排法，通常是进程级定位后，再深入到线程级，再落到线程栈。
- 高级岗位更看重你能不能说出这条链路，而不是只说一个 `top`。
- 另外，CPU 高不一定全是业务代码算得多，也可能是线程空转、自旋、GC 或锁竞争。

### 追问答案

- 为什么 CPU 高还要看线程栈？
  因为真正的热点通常要落到具体线程和调用栈，才能知道是业务计算、GC 还是锁竞争。

### 学习提示

- 这题面试时建议用“进程 -> 线程 -> 栈”三层去答。

---

## 2.4 Java 进程内存持续上涨时，一般怎么排？

### 核心讲解

- 一般先区分是堆内存涨、非堆内存涨，还是系统层缓存和连接资源涨。
- 再看对象是不是持续堆积、缓存是不是失控、请求是不是积压、对象生命周期是不是异常变长。
- 不建议一看到内存涨就直接下结论说“内存泄漏”。

### 深入理解

- 这题最关键的是先分类。很多内存问题不是传统意义上的泄漏，而是对象堆积、缓存膨胀、线程本地变量未释放、请求积压导致的暂时性上涨。
- 所以更成熟的排法，是先确定涨在哪一层，再看对象结构和生命周期，而不是上来就 dump 一把。
- 真实线上里，很多“内存高”最后会和缓存策略、消息积压、下游变慢一起出现。

### 追问答案

- 为什么内存上涨不等于泄漏？
  因为很多对象仍然是可达的，只是生命周期异常变长或请求积压导致没及时释放。

### 学习提示

- 面试里最好主动补一句“先分堆内、非堆、系统层”，会显得更稳。

## 2.5 接口 RT 抖动时，一般怎么区分是系统层、JVM 层还是业务层问题？

### 核心讲解

- 一般先看机器资源和 JVM 是否有明显异常，再看线程池、数据库、缓存、下游依赖和锁竞争。
- RT 抖动常见根因包括 GC、数据库慢、缓存回源、线程池堆积、下游超时和网络抖动。
- 真正重要的不是列举所有原因，而是先把范围快速缩小。

### 深入理解

- RT 抖动最典型的难点，就是原因很多，而且常常会跨层传导。所以这题真正考的是“区分能力”。
- 如果机器和 JVM 都平稳，那就更像业务依赖或下游问题；如果机器层已抖，那就先别急着钻代码；如果线程池队列开始堆，那又是另一条线。
- 高级 Java 需要能把 RT 抖动看成一条链，而不是一个孤立指标。

### 追问答案

- 为什么 RT 抖动比 CPU 高更难查？
  因为 RT 是最终用户感知，背后可能叠了系统层、JVM 层和业务层多个原因。

### 学习提示

- 建议把 RT 抖动和 [09-高并发系统设计面试题库.md](E:\学习记录\09-高并发系统设计面试题库.md) 的排障表一起看。

# 3. 常见线上问题定位

- `3.4` 文件句柄过高时，一般怎么理解？
- `3.5` 为什么有时候机器负载很高，但 CPU 并不高？

## 知识地图

- `症状 -> 第一观察点 -> 常见根因 -> 止血动作`

## 学习目标

- 把最常见的线上问题拆成熟悉的排查模板。
- 形成“先看什么、先做什么、什么不能先做”的习惯。
- 把机器指标、Java 进程、线程池、连接池和业务依赖串起来。

## 先修知识

- Linux 基础指标
- Java 服务排障主线

## 本模块总结

- 这一组最重要的是看到症状后别慌，先抓第一观察点。
- 很多问题的第一动作不是修代码，而是先保住服务。
- 只要你能把每类问题的“第一观察点”练熟，排障能力就会明显提升。

## 典型业务场景

- 高峰期线程池打满，核心接口开始超时。
- 机器负载高，但看起来 CPU 利用率并不高。
- 磁盘突然打满，日志写入和应用行为一起异常。

## 最小实践建议

- 每个题都试着自己写一版“第一观察点”。
- 练习把问题分成：资源打满、依赖阻塞、配置失衡、链路放大。

## 3.1 线程池打满时，一般先看什么？

### 核心讲解

- 先看活跃线程数、队列长度、任务耗时、拒绝次数和下游依赖 RT。
- 线程池打满不一定是线程数太小，也可能是任务执行变慢、下游卡住或重试放大。
- 所以第一步不是盲目调大线程池，而是先看任务为什么堆。

### 深入理解

- 高级排障里，线程池问题非常高频，因为它常常只是“上游流量”和“下游变慢”之间的受害者。
- 如果只看线程数，很容易误判。真正关键的是任务为什么不能及时出队。
- 这题很适合和 Java 并发、微服务、高并发题库一起联动。

### 追问答案

- 为什么不能一上来就把线程池调大？
  因为如果根因在慢依赖或慢任务，线程数越大只会把问题放得更大。

### 学习提示

- 记一句话：先看耗时和队列，再看线程数。

---

## 3.2 连接数异常升高时，一般先看什么？

### 核心讲解

- 先看连接数升高的是数据库连接、HTTP 连接还是 socket 连接，再看是不是连接泄漏、慢依赖、重试放大或请求堆积。
- 连接数问题常常和慢请求、线程池堆积一起出现。
- 所以这类问题最好和 RT、线程池、下游依赖一起看。

### 深入理解

- 这题的关键是不要把“连接数高”当成一个孤立指标。连接数高，往往只是请求没结束、连接没释放、依赖变慢或重试放大的结果。
- 高级岗位里，这类题往往不是问你连接池参数，而是问你有没有排障思路。

### 追问答案

- 为什么连接数高经常伴随 RT 高？
  因为请求处理变慢，连接生命周期被拉长。

### 学习提示

- 先分清是什么连接，再谈原因。

---

## 3.3 磁盘打满时，一般怎么排查？

### 核心讲解

- 先看是日志膨胀、临时文件堆积、dump 文件未清理，还是业务落盘异常。
- 再看磁盘利用率、目录大小、写入热点和应用是否受影响。
- 止血时通常先释放空间、控制日志增长，再追根因。

### 深入理解

- 磁盘问题很容易被忽略，但它会直接影响日志写入、数据库、消息落盘甚至应用稳定性。
- 这题真正要理解的是：磁盘问题通常不是“只有磁盘问题”，它会连带放大 RT、错误率和依赖异常。

### 追问答案

- 为什么磁盘打满会影响 Java 服务？
  因为日志、临时文件、dump、甚至下游存储行为都可能依赖磁盘写入。

### 学习提示

- 这题回答时别停在“删日志”，要补“为什么会涨这么快”。

---

## 3.4 文件句柄过高时，一般怎么理解？

### 核心讲解

- 文件句柄过高通常意味着进程打开了太多文件、连接或 socket，常见原因是连接未释放、日志文件过多、文件流泄漏或高并发连接堆积。
- 句柄问题很容易和连接数、线程池、网络调用问题一起出现。
- 所以它本质上也是资源泄漏或资源生命周期异常的问题。

### 深入理解

- 很多 Java 服务线上故障，最后都能追到“资源没关”或“请求一直卡着没结束”。文件句柄只是一个结果指标。
- 高级岗位更想看你能不能把句柄问题和连接、文件、socket 生命周期联系起来。

### 追问答案

- 为什么文件句柄问题不只是文件问题？
  因为 socket、连接、管道本质上也会消耗句柄。

### 学习提示

- 记住：句柄问题通常是资源生命周期管理问题。

---

## 3.5 为什么有时候机器负载很高，但 CPU 并不高？

### 核心讲解

- 因为负载不只看正在用 CPU 的任务，还包括等待 CPU、等待 IO 或处于不可中断状态的任务。
- 所以磁盘慢、网络阻塞、锁等待都可能让负载高，但 CPU 利用率并不一定高。
- 这类问题最容易误导只会盯 CPU 的人。

### 深入理解

- 这题是高级面试里的高频判断题。很多人一看负载高就默认“CPU 不够”，其实 IO 等待和阻塞任务更常见。
- 真正会排障的人，会立刻联想到磁盘、网络、锁和下游阻塞。

### 追问答案

- 负载高时第一步为什么不一定先扩 CPU？
  因为如果根因是 IO 或阻塞，扩 CPU 并不能解决问题。

### 学习提示

- 这题建议和 `vmstat`、`iostat` 一起记。

# 4. 止血、复盘与工程化

## 本模块题目

- `4.1` 线上出问题时，为什么先止血再定位？
- `4.2` Java 服务线上问题复盘，一般怎么写？
- `4.3` 怎么把一次线上故障沉淀成长期治理项？

## 知识地图

- `故障暴露 -> 先止血 -> 再定位 -> 复盘 -> 长期治理`

## 学习目标

- 理解技术排障里为什么止血优先于深挖，以及不同止血动作适合什么场景。
- 学会把一次故障从“查到了”推进到“技术治理掉”。
- 建立监控、预案、容量和演练这些工程治理意识。

## 先修知识

- 常见线上排障主线
- 监控与告警基础

## 本模块总结

- 这一组仍然站在技术排障视角看问题，重点是“先怎么救、再怎么查、最后技术上怎么补”。
- 如果只是想练事故表达和复盘话术，可以把这组和 [10-项目表达与故障排查面试题库.md](E:\学习记录\10-项目表达与故障排查面试题库.md) 对照着看。
- 复盘在这里更偏技术治理动作，而不是面试表达方式本身。

## 常见误解

- 以为止血就是“扩容和重启”。很多时候更重要的是保核心链路、限流、摘流量和收敛重试。
- 以为监控只要有 CPU、内存、RT 就够。高级治理还要补业务语义监控。
- 以为故障恢复后就结束了。真正的长期价值在治理项、自愈和机制沉淀。
- 以为探针一定越严格越好。探针阈值和 Full GC、启动时间、下游依赖抖动不匹配时，反而会误杀实例。

## 典型业务场景

- 核心接口在活动期间大量超时，需要先决定保哪些、停哪些。
- 某次故障已经恢复，但团队不知道后续应该补监控、补压测还是补架构治理。
- 某个实例明明还活着，但探针和流量摘除策略没有及时工作，导致故障持续放大。

## 最小实践建议

- 至少准备一个“止血优先级”案例和一个“复盘后治理”案例。
- 练习用“现象 -> 第一观察点 -> 临时止血 -> 技术根因 -> 长期治理”去复述一次线上事故。

## 4.1 线上出问题时，为什么先止血再定位？

### 核心讲解

- 因为线上第一目标是恢复核心服务、收敛影响面，而不是立刻找到最深层根因。
- 如果不先止血，用户损失和故障扩散会持续放大。
- 所以高级岗位强调的是“先救火，再查因，再复盘”。

### 深入理解

- 这题本质上在看你有没有业务优先级意识。很多人技术上会查问题，但缺少线上判断：什么时候先限流，什么时候先降级，什么时候先切依赖。
- 真正成熟的止血，往往是业务和技术一起做取舍，不是只谈技术动作。

### 追问答案

- 为什么止血优先于定位？
  因为线上损失是实时发生的，定位越晚越要先控影响面。

### 学习提示

- 面试里这题一定记得补“核心链路优先”。
- 如果是容器和网关体系，还要顺手补一句“有没有办法自动摘掉异常实例、降低扩散速度”。

## 4.2 Java 服务线上问题复盘，一般怎么写？

### 核心讲解

- 更稳的复盘结构通常是：现象、影响面、时间线、临时止血、根因分析、改进项。
- 复盘重点不是写故事，而是让团队下次更早发现、更快止血、不再重复犯。
- 所以复盘一定要落到具体治理动作。

### 深入理解

- 复盘的价值，不是证明谁查得快，而是把一次故障变成团队能力。
- 高级 Java 面试里，如果你能把复盘讲成“时间线 + 关键决策 + 治理动作”，会很加分。

### 工程化补充

- 高级团队的复盘，不只补技术监控，还会补：
  - 业务语义指标
  - 容量阈值
  - 探针和摘流量策略
  - 自动化回滚或降级预案

### 追问答案

- 为什么复盘不能只写根因？
  因为只有根因没有治理动作，团队还是会重复踩坑。

### 学习提示

- 复盘一定补三类动作：监控、预案、架构或配置治理。

## 4.3 怎么把一次线上故障沉淀成长期治理项？

### 核心讲解

- 通常要落到监控补齐、告警优化、容量评估、压测、预案、架构优化和流程改进。
- 治理项不能停在“以后注意”，而要能被执行、验证和复用。
- 真正有价值的故障治理，是把问题从个案变成机制改进。

### 深入理解

- 这题是在看你有没有工程化思维。真正高级的地方，不是一次排障有多快，而是能不能让系统和团队因此变强。
- 所以长期治理一般会分成两类：技术层补监控、补限流、补资源治理；流程层补预案、演练、上线检查和容量评估。

### 自愈与治理补充

- 常见长期治理动作可以再细分成三层：
  - 观察层：监控、Trace、火焰图、业务指标
  - 保护层：限流、隔离、摘流量、探针、自适应降级
  - 恢复层：自动拉起、自动摘除、自动回滚、预案和演练
- 这三层越完整，系统越不依赖“人肉救火”。
- 如果是 K8s 体系，还要顺手问自己：
  - 探针阈值是否和服务启动时间、GC 行为匹配
  - 自动摘流量是否会误伤短时抖动实例
  - 恢复动作是不是只会“重启”，还是能真正保护核心链路

### 追问答案

- 为什么很多团队故障老是重复发生？
  往往不是没查到原因，而是没有把原因变成机制和动作。

### 学习提示

- 记一句话：排障解决当下，治理解决未来。
