# 数据库面试题库

## 1. MySQL 索引为什么能加速查询？

### 30秒结论

- 索引本质上是帮助数据库缩小扫描范围的数据结构，InnoDB 默认用 B+ 树。它能把全表扫描变成有序查找，减少磁盘 IO 和回表成本。

### 3分钟回答

- 不加索引时只能大范围扫描，数据量一大 IO 和 CPU 开销都会明显上升。
- InnoDB 的索引底层通常是 B+ 树，适合分页读取和范围查询。
- 聚簇索引叶子节点直接存整行数据，二级索引叶子节点通常存主键值，所以有时还会回表。

### 追问答案

- 为什么是 B+ 树，不是红黑树？
  因为数据库更关注磁盘 IO，B+ 树多叉、树高更低，更适合页式存储。
- 什么是回表？
  走二级索引查到主键后，再去聚簇索引取整行数据。
- 什么是覆盖索引？
  查询需要的列都在索引里，不需要回主表。

## 2. 聚簇索引和二级索引有什么区别？

### 30秒结论

- 聚簇索引叶子节点直接存整行数据；二级索引叶子节点通常存主键值。前者更适合主键定位，后者更适合按其他字段查询。

### 3分钟回答

- InnoDB 表一定有聚簇索引，通常就是主键索引。
- 二级索引基于普通字段建立，不直接存整行，所以查不全字段时要回表。
- 主键建议尽量短、稳定、递增，因为所有二级索引叶子节点都带主键值。

### 追问答案

- 主键为什么建议自增？
  更接近顺序写，页分裂少，索引维护成本低。
- 没有主键会怎样？
  InnoDB 会选非空唯一索引或生成隐藏列作为聚簇索引。
- UUID 为什么不适合做主键？
  因为长且随机，会导致页分裂更多、索引更大。

---

## 3. 联合索引为什么要遵守最左前缀原则？

### 30秒结论

- 联合索引底层按字段组合顺序排序，只有从最左边开始连续使用索引列，数据库才能有效利用这棵索引树。

### 3分钟回答

- `(a,b,c)` 在物理上会先按 a 排，再按 b 排，再按 c 排。
- 所以查询如果直接从 b 或 c 开始，通常没法高效利用索引有序性。
- 最左前缀不是必须三个字段全用，而是从最左边开始连续命中即可。

### 追问答案

- 范围查询会影响后面的索引吗？
  通常会，一旦前面用了范围，后面列继续高效利用索引的能力会下降。
- 联合索引字段顺序怎么定？
  看高频过滤条件、区分度、排序需求和覆盖需求。
- `where a = ? and c = ?` 能用到 `(a,b,c)` 吗？
  通常只能稳定用到 a，因为中间缺了 b。

## 4. 什么情况下索引会失效？

### 30秒结论

- 常见原因有函数操作、隐式类型转换、前导模糊匹配、计算表达式、不符合最左前缀，以及优化器认为全表扫描更便宜。

### 3分钟回答

- 对索引列做函数或计算，比如 `date(create_time)`，通常会破坏索引利用。
- 隐式类型转换也可能导致索引失效。
- `like '%xx'` 无法利用有序前缀。
- 最终是否走索引要以执行计划为准。

### 追问答案

- `like 'abc%'` 为什么能走索引？
  因为保留了左侧前缀。
- 隐式类型转换为什么危险？
  本质上相当于在列上套函数。
- 怎么判断 SQL 有没有走索引？
  看 `explain` 的 `type`、`key`、`rows`、`extra`。

## 5. MySQL 事务的四大特性和隔离级别怎么答？

### 30秒结论

- 事务四大特性是 ACID。常见隔离级别有读未提交、读已提交、可重复读、串行化，InnoDB 默认是可重复读。

### 3分钟回答

- 原子性是要么都成功要么都失败；一致性是事务前后数据状态合法；隔离性是多个事务互不干扰；持久性是提交后的数据不会丢。
- 读未提交可能脏读；读已提交解决脏读但仍可能不可重复读；可重复读进一步保证事务内多次读取一致；串行化最严格但并发最低。
- InnoDB 默认可重复读，并通过 MVCC 和锁机制解决大部分并发问题。

### 追问答案

- 什么是脏读、不可重复读、幻读？
  脏读是读到未提交数据；不可重复读是同一行前后读结果不同；幻读是范围查询前后条数不同。
- MySQL 默认为什么选可重复读？
  因为它在一致性和性能之间比较折中。
- 一致性和隔离性有什么区别？
  一致性强调结果合法，隔离性强调并发事务之间影响受控。

## 6. MVCC 是什么？解决了什么问题？

### 30秒结论

- MVCC 是多版本并发控制，核心是让普通读读到历史可见版本，从而减少读写冲突。

### 3分钟回答

- InnoDB 每行有事务 ID、回滚指针等隐藏字段，旧版本通过 undo log 串成版本链。
- 查询时通过 Read View 判断哪个版本可见，所以很多普通查询在 RC 或 RR 下不需要加锁也能看到一致结果。
- MVCC 主要作用于快照读，当前读仍然要配合锁。

### 追问答案

- 快照读和当前读有什么区别？
  快照读读历史可见版本；当前读读最新版本并可能加锁。
- undo log 在这里起什么作用？
  保存旧版本数据，支撑回滚和版本链读取。
- MVCC 能解决幻读吗？
  快照读下能大幅弱化幻读影响，但当前读还需要间隙锁、临键锁。
- MVCC 是如何解决 RC 和 RR 的读写冲突？
  RC 每次快照读生成新的 Read View，只能看到当时已提交版本；RR 第一次快照读生成 Read View 后整个事务复用，所以同一事务多次读取结果更稳定。

## 7. MySQL 里的行锁、表锁、间隙锁、临键锁怎么理解？

### 30秒结论

- 行锁锁单行，表锁锁整张表；间隙锁锁的是记录之间的区间；临键锁是行锁加间隙锁的组合。

### 3分钟回答

- 行锁是 InnoDB 最常见的锁粒度，锁的是索引命中的记录，所以没走索引时可能退化成更大范围的锁。
- 从实现上说，很多所谓行锁本质上都是加在索引记录或索引区间上的。
- 间隙锁和临键锁的核心目标是防止范围并发插入，避免可重复读下的幻读。

### 追问答案

- 为什么没走索引会锁很多行？
  因为 InnoDB 行锁基于索引实现，不能精确命中索引时会扫描并锁住更大范围。
- 间隙锁为什么会影响插入？
  因为它锁住的是区间，目的是防止别人在该范围内插入新记录。
- 什么情况下会发生死锁？
  多个事务互相等待对方释放锁时就会死锁。
- 为什么 InnoDB 的行级锁要依赖索引实现？哪些锁是基于索引加的？
  因为只有通过索引定位到具体记录或区间，才能实现细粒度加锁。记录锁、间隙锁、临键锁本质上都和索引项相关。

## 8. 慢 SQL 应该怎么排查和优化？

### 30秒结论

- 排查顺序一般是 `先定位 -> 看执行计划 -> 查索引和扫描行数 -> 看是否回表、排序、临时表 -> 再决定改 SQL、补索引还是调数据模型`。

### 3分钟回答

- 先通过慢查询日志、APM 或业务日志定位具体 SQL。
- 再看 `explain`，重点关注有没有走索引、扫描行数多不多、有没有 filesort、temporary、回表和全表扫描。
- 然后结合 SQL 结构判断是索引、分页、排序、join 还是模型设计问题。

### 追问答案

- `explain` 重点看哪些字段？
  常见重点是 `type`、`key`、`rows`、`filtered`、`extra`。
- filesort 一定不好吗？
  不一定，小数据量下也可能完全可接受。
- 优化 SQL 和加缓存怎么选？
  优先把数据库查询本身做对，缓存是放大器不是替代品。
- `explain` 里 `type` 从好到差怎么理解？
  通常 `const`、`eq_ref`、`ref` 较好，`range` 通常可接受，`index` 和 `all` 要重点警惕。
- 为什么建了索引还是可能慢？
  可能是索引选择性差、顺序不合理、回表多、深分页、排序分组没利用上索引，或者优化器判断全表扫描更划算。
- 慢 SQL 和锁等待怎么区分？
  SQL 本身慢通常表现为扫描行数大、执行计划差；锁等待则是语句本身没问题，但时间耗在等待锁。
- 常见导致慢 SQL 的根因有哪些？
  常见有没合适索引、联合索引顺序不合理、索引失效、回表过多、深分页、排序分组不走索引、`select *`、join 字段没索引。
- join SQL 怎么排查？
  先看驱动表和被驱动表选得是否合理，再看 join 字段有无索引，然后结合 `explain` 看访问类型、扫描行数和连接代价。

## 9. 深分页为什么慢？怎么优化？

### 30秒结论

- 深分页慢的核心原因是 `limit offset, size` 在 offset 很大时，数据库通常要先扫描并丢弃前面大量记录。

### 3分钟回答

- 比如 `limit 100000, 20`，数据库往往先扫描前面很多数据，再丢掉。
- 如果还伴随排序，性能会更差。
- 常见优化是基于主键或有序字段做游标翻页、先走覆盖索引查主键再回表，或限制翻页深度。

### 追问答案

- 游标分页有什么缺点？
  更适合连续翻页，不适合随机跳页。
- 为什么先查主键再回表会更快？
  因为先走覆盖索引可显著减少扫描列和 IO。
- 哪些场景不适合 offset 分页？
  数据量大、翻页深、排序复杂的后台列表、日志检索、消息记录等场景。

## 10. MySQL 主从复制和读写分离有哪些问题？

### 30秒结论

- 主从复制主要解决读扩展和高可用，但会带来主从延迟、一致性窗口和路由复杂度问题。

### 3分钟回答

- 主库负责写，从库负责读，主库把 binlog 传给从库，从库重放日志完成同步。
- 这样能分摊读流量，但代价是复制延迟，所以写后立刻去从库读可能读到旧数据。
- 工程上通常通过关键读走主库、短时间会话读主、缓存兜底等方式处理一致性场景。

### 追问答案

- 什么是主从延迟？
  主库已经提交成功，但从库还没来得及重放对应 binlog。
- 写后立刻读怎么保证一致？
  关键读请求强制走主库，或者短时间路由到主库。
- 主从复制是同步还是异步？
  默认是异步，工程上也有半同步方案，但都不能简单理解成主从绝对实时一致。

## 11. 为什么会出现幻读？MySQL 是怎么处理的？

### 30秒结论

- 幻读是同一事务里两次按范围查询，结果集条数发生变化。InnoDB 在可重复读下，快照读主要靠 MVCC 保持一致视图，当前读则靠间隙锁和临键锁防止范围内再插入。

### 3分钟回答

- 幻读更偏范围内突然多了或少了几行，不可重复读更偏同一行的值变了。
- 普通 `select` 这种快照读下，事务会读到一致快照，所以很多场景下你感知不到幻读。
- 但 `select ... for update`、`update`、`delete` 这种当前读要读最新数据，所以要靠锁防止范围并发插入。

### 追问答案

- 幻读和不可重复读有什么区别？
  不可重复读通常是同一行前后值不同；幻读通常是范围查询前后条数不同。
- MVCC 能完全解决幻读吗？
  对快照读影响被大幅弱化，但当前读下的范围并发问题还得靠间隙锁和临键锁。
- 为什么当前读还要加间隙锁？
  因为不锁区间的话，别的事务可能在该范围内插入新记录。

## 12. 可重复读下，普通 `select` 和 `select ... for update` 为什么表现不同？

### 30秒结论

- 因为普通 `select` 是快照读，读的是历史可见版本；`select ... for update` 是当前读，读的是最新版本并加锁。

### 3分钟回答

- 普通 `select` 在可重复读下会基于第一次快照读取历史可见版本，所以别的事务提交了更新，你后面再查也可能看不到。
- `select ... for update` 是当前读，它要读最新数据并锁住相关记录或区间，保证后续更新安全。
- 所以表现不同，不是可重复读失效，而是读类型不同。

### 追问答案

- 哪些语句属于当前读？
  常见有 `select ... for update`、`select ... lock in share mode`、`update`、`delete`。
- 普通 `select` 一定不加锁吗？
  一般是快照读，不主动加锁，但不能脱离隔离级别和具体语义绝对化。
- 为什么更新前常用 `for update`？
  为了先把目标记录锁住，避免多个事务同时读写同一批数据。

## 13. 读已提交和可重复读怎么选？

### 30秒结论

- 核心看业务对一致性和并发性能的平衡。RC 更接近最新已提交结果，并发更友好；RR 更强调事务内结果稳定。

### 3分钟回答

- RC 每次快照读都会生成新的 Read View，所以更容易读到别的事务最新提交的数据。
- RR 第一次快照读生成 Read View 后，整个事务复用，所以同一事务多次读取更稳定。
- 核心交易链路通常更倾向 RR，后台管理和报表类场景有时会考虑 RC。

### 追问答案

- RC 的优点是什么？
  更接近最新已提交结果，锁冲突通常更少。
- RR 的代价是什么？
  一致性更强，但某些场景下锁范围和排查复杂度会更高。
- 为什么 InnoDB 默认选 RR？
  因为它在一致性和性能之间比较折中，适合大多数 OLTP 场景。

## 14. 大事务为什么危险？

### 30秒结论

- 大事务会放大锁持有时间、undo/redo 压力、主从延迟和回滚成本，影响面远不只是执行时间长一点。

### 3分钟回答

- 大事务首先会让锁持有更久，其他事务容易排队。
- 其次会产生大量 undo 和 redo，占用更多资源。
- 回滚时也更痛苦，因为要撤销的改动更多。
- 在主从架构下，大事务还会放大 binlog 重放压力。

### 追问答案

- 大事务会带来哪些直接现象？
  锁等待增多、RT 抖动、主从延迟升高、undo 膨胀、连接占用时间变长。
- 为什么回滚大事务更痛苦？
  因为回滚本质上也是执行补偿操作，事务越大撤销越慢。
- 怎么避免写出大事务？
  拆批提交、缩小事务边界、避免在事务里做 RPC、文件 IO 和重计算。

---

## 15. 长事务会带来哪些问题？

### 30秒结论

- 长事务最大的问题是资源长期不释放，会让锁占更久、旧版本回收不了、undo 膨胀、主从延迟变大。

### 3分钟回答

- 长事务即使没改太多数据，只要不提交，也会让事务快照长期存在。
- 在 MVCC 下，只要旧事务没结束，相关旧版本就不能及时清理。
- 如果还持有锁，就会阻塞后续写请求，造成排队和超时。

### 追问答案

- 长事务和大事务有什么区别？
  大事务强调改动量大，长事务强调持续时间长，两者可能重合但不是一个概念。
- 长事务为什么会影响 purge？
  因为旧事务还可能读到历史版本，所以数据库不能提前清理。
- 怎么定位长事务？
  看事务监控、会话持续时间、是否持锁，以及相关系统表和监控平台指标。

## 16. 什么 SQL 最容易引发死锁？

### 30秒结论

- 最容易引发死锁的，通常是多个事务以不同顺序更新同一批数据，或者范围更新、批量更新、缺索引更新把锁范围放大。

### 3分钟回答

- 最经典的场景是事务 A 先锁记录 1 再锁记录 2，事务 B 反过来。
- 现实里批量更新、范围更新、带索引范围的当前读更常见。
- 没走好索引时，锁范围更大，更容易和其他事务交叉等待。

### 追问答案

- 单条 SQL 也会死锁吗？
  会，尤其是底层需要锁多条记录或多个索引区间时。
- 批量更新为什么危险？
  因为持有更多锁、更长事务时间，且不同批次命中顺序未必一致。
- 死锁和锁等待超时有什么区别？
  死锁是循环等待，数据库会主动检测并回滚其中一个；锁等待超时不一定有环。

## 17. 怎么定位线上死锁？

### 30秒结论

- 定位死锁的关键不是只看到报错，而是把当时的 SQL、事务顺序、锁对象和索引路径串起来。

### 3分钟回答

- 先抓到具体报错和 SQL。
- 再看 `show engine innodb status` 等死锁信息，确认双方持有什么锁、在等什么锁。
- 再结合执行计划和业务代码，判断是不是索引、事务边界或加锁顺序问题。

### 追问答案

- `show engine innodb status` 主要看什么？
  看最近一次死锁里双方执行的 SQL、持有锁、等待锁和对应索引。
- 死锁一定要彻底消灭吗？
  高频死锁必须处理；低频死锁未必完全消除，但要控制在可接受范围。
- 死锁发生后应用层怎么处理？
  常见做法是有限次数重试，前提是请求具备幂等性。

## 18. 什么是意向锁？为什么需要它？

### 30秒结论

- 意向锁是表级锁标记，用来告诉数据库这张表里某些行准备加行锁或已经加了行锁，它让表锁和行锁的冲突判断更高效。

### 3分钟回答

- 如果每次加表锁都逐行判断有没有行锁，成本会很高。
- 所以 InnoDB 在对行加锁前，会先在表上加对应的意向锁。
- 这样别的事务要加表锁时，只需看表上的意向锁状态即可。

### 追问答案

- 意向锁是表锁还是行锁？
  它是表级标记，不是直接控制单行并发的业务锁。
- 意向锁会阻塞普通行级操作吗？
  一般不会，它主要和表级锁的兼容性相关。
- 为什么说它是协调机制？
  因为它的核心价值是让整表加锁和部分行已加锁之间的冲突判断更高效。

---

## 19. 唯一索引等值查询和普通索引范围查询，加锁行为有什么区别？

### 30秒结论

- 唯一索引等值查询通常锁更精确，更多是记录锁；普通索引范围查询更容易锁区间，出现间隙锁或临键锁。

### 3分钟回答

- 用唯一索引精确命中一条记录时，通常只需要锁住该记录本身。
- 普通索引上的范围查询，为了防止范围内再插入满足条件的新记录，往往还要锁相关区间。
- 所以索引类型和查询形态不同，锁范围差别会很大。

### 追问答案

- 唯一索引没命中时会怎样？
  没命中时也可能涉及相关区间保护，不能简单理解成没查到就不加锁。
- 为什么范围查询更容易影响插入？
  因为除了锁已有记录，还要防止范围内出现新记录。
- 主键等值更新为什么通常冲突更小？
  因为通常能精确命中单条记录，锁更集中。

---

## 20. 什么情况下锁范围会扩大？

### 30秒结论

- 锁范围扩大通常发生在 SQL 没精确命中索引、范围条件过大、扫描行数很多或隔离级别要求更强时。

### 3分钟回答

- 没走索引时，InnoDB 只能扫描并锁更多记录。
- 范围查询、批量更新和排序扫描本身就会命中更多记录或区间。
- 在可重复读下，普通索引范围当前读还可能带上间隙锁和临键锁。

### 追问答案

- 没走索引为什么会放大锁范围？
  因为行锁依赖索引定位，不能精确定位时只能扫描更多记录并加锁。
- 范围查询为什么锁更多？
  因为天然命中多条记录甚至整个区间。
- 怎么尽量缩小锁范围？
  让 SQL 精确命中索引、减少不必要的范围扫描、缩小事务边界。

---

## 21. 联合索引字段顺序到底怎么定？

### 30秒结论

- 联合索引顺序不是只看区分度，而是综合看高频查询条件、等值匹配优先级、排序需求、覆盖需求和最左前缀复用价值。

### 3分钟回答

- 先看最常用的过滤条件和谁能明显缩小数据范围。
- 再看等值和范围条件，通常高频等值列更适合靠前。
- 还要考虑 `order by`、`group by` 和覆盖索引收益。

### 追问答案

- 区分度最高一定放最前吗？
  不一定，要看真实 SQL 模式和整体复用价值。
- 等值和范围字段谁更靠前？
  通常高频等值列更适合靠前。
- 为什么要考虑排序和覆盖？
  因为好索引不只过滤，还能顺带排序和减少回表。

---

## 22. 区分度不高的字段该不该建索引？

### 30秒结论

- 单独给区分度很低的字段建索引，很多时候收益有限，但如果它经常和其他字段组合过滤，仍然可能有价值。

### 3分钟回答

- 区分度低意味着重复值很多，单靠它很难显著缩小扫描范围。
- 这种单列索引选择性差，优化器可能更倾向全表扫描。
- 但如果它作为联合索引的一部分，和其他列组合后仍可能有收益。

### 追问答案

- 什么叫区分度低？
  不同值很少、重复值很多。
- 低区分度字段什么时候值得建？
  当它经常作为联合条件的一部分，且组合后能显著缩小范围时。
- 怎么验证值不值得建？
  看真实 SQL、执行计划、扫描行数和线上命中率。

## 23. 覆盖索引为什么快？

### 30秒结论

- 覆盖索引快的核心原因是查询需要的列都在索引里，不用再回主表取整行数据。

### 3分钟回答

- 二级索引叶子节点通常只存索引列和主键值，不存整行。
- 如果查询列都包含在索引里，数据库拿到索引记录后就能直接返回结果。
- 相比之下，不覆盖时就得回表，多一次查找路径。

### 追问答案

- 覆盖索引是不是越多越好？
  不是，索引越多，占空间越大，写入维护成本越高。
- 为什么覆盖索引对大扫描更有价值？
  因为扫描记录越多，减少回表次数带来的累计收益越明显。
- 覆盖索引和联合索引有什么关系？
  很多覆盖索引本质上就是设计合理的联合索引。

## 24. 回表为什么会慢？

### 30秒结论

- 回表慢的原因是先走二级索引找到主键，再回聚簇索引取整行，等于多了一次查找路径。

### 3分钟回答

- 二级索引叶子节点一般不存整行，只存索引列和主键值。
- 查询列不在索引里时，就要根据主键再次到聚簇索引拿数据。
- 单条回表影响可能不明显，但扫描几千、几万行时，额外成本会迅速放大。

### 追问答案

- 主键查询为什么没有回表？
  因为聚簇索引叶子节点本来就存整行数据。
- 回表和 `select *` 有什么关系？
  `select *` 更容易超出索引覆盖范围，导致必须回表。
- 怎么减少回表？
  设计合适的覆盖索引、减少不必要查询列、先筛主键再少量回表。

## 25. 什么样的查询适合建联合索引？

### 30秒结论

- 高频、多条件过滤、同时带排序或需要覆盖查询的 SQL，最适合建联合索引。

### 3分钟回答

- 如果长期是固定模式，比如 `where a=? and b=? order by c`，联合索引通常比多个单列索引更有效。
- 联合索引既能利用字段组合顺序做过滤，也可能顺带完成排序和覆盖。
- 查询模式发散、字段组合变化大时，联合索引建太多反而会造成冗余。

### 追问答案

- 多个单列索引能替代联合索引吗？
  不完全能，某些场景优化器会做索引合并，但通常不如合理的联合索引稳定。
- 联合索引字段越多越好吗？
  不是，字段越多索引越胖，维护成本越高。
- 什么查询不适合建联合索引？
  命中频率低、字段顺序不稳定、组合变化大的查询。

## 26. `count(*)` 为什么有时也会慢？

### 30秒结论

- `count(*)` 慢通常不是函数本身慢，而是数据库仍然要扫描满足条件的数据。

### 3分钟回答

- 无条件统计大表时，InnoDB 往往也要扫描，因为没有简单可直接返回的精确总数。
- 带条件时，如果没命中高效索引，也会退化成大范围扫描。
- 即使走索引，如果过滤复杂、需要回表或 join，成本同样不低。

### 追问答案

- `count(1)` 和 `count(*)` 有区别吗？
  在 MySQL 里通常没有本质性能差异。
- 为什么 InnoDB 不能直接返回总行数？
  因为 MVCC 下不同事务看到的可见行数可能不同。
- 什么场景适合做预聚合？
  高频统计、报表看板、首页计数这类允许轻微延迟的场景。

## 27. `order by` 为什么会慢？怎么优化？

### 30秒结论

- `order by` 慢的核心是排序要消耗 CPU 和内存，不能利用索引顺序时就容易出现 filesort。

### 3分钟回答

- 如果排序字段和查询条件能利用索引顺序，数据库可以边扫描边返回。
- 否则就可能需要额外排序，数据量一大时还可能借助临时文件。
- 深分页场景里，`order by + limit offset` 会更慢。

### 追问答案

- filesort 一定落盘吗？
  不一定，小数据量时可能只在内存里完成。
- 为什么排序字段建了索引还是可能慢？
  因为过滤条件、排序方向、联合索引顺序不匹配时，仍可能无法利用索引完成排序。
- `order by` 和 `limit` 怎么配合优化？
  尽量让数据库按索引顺序直接取前 N 条，而不是先扫很多数据再排序。

## 28. `group by` 为什么会慢？

### 30秒结论

- `group by` 慢通常是因为它要先扫描数据、再分组聚合，数据量大时还可能用到临时表和排序。

### 3分钟回答

- `group by` 的本质是把满足条件的数据按某些维度聚合，所以要先拿到参与分组的记录。
- 没有合适索引时容易大范围扫描。
- 多表 join 后再分组会更重，因为 join 成本也叠加进来了。

### 追问答案

- `group by` 和 `order by` 为什么经常一起慢？
  因为都可能引入额外排序和中间结果处理。
- 哪些场景适合预聚合？
  报表、排行榜、仪表盘这类高频统计展示场景。
- 临时表为什么值得警惕？
  因为它意味着数据库要保存中间结果，执行成本通常会上升。

## 29. 子查询和 join 怎么选？

### 30秒结论

- 没有绝对谁更快，核心看执行计划和数据规模。但工程上多表关联通常更倾向先把 join 写清楚，再看执行计划优化。

### 3分钟回答

- 子查询和 join 最终都会被优化器转换和执行，不能机械地说谁一定更快。
- join 的优势是关系更明确，更容易看驱动表、关联条件和索引利用情况。
- 子查询如果能先把结果集缩得很小，也可能是合理写法。

### 追问答案

- 什么情况下子查询更合适？
  当子查询能先快速筛出很小的数据集，或者业务语义更清晰时。
- join 最怕什么问题？
  关联字段没索引、驱动表选择不合理、join 前数据集太大。
- 为什么一定要看执行计划？
  因为表面写法和实际执行路径可能差很多。

## 30. `union` 和 `union all` 的区别及性能影响是什么？

### 30秒结论

- `union all` 直接拼接结果，不去重，通常更快；`union` 还要做去重，额外带来排序或哈希等成本。

### 3分钟回答

- `union` 会合并并去重，`union all` 只是直接拼接。
- 去重本身就有成本，数据量大时会引入额外处理中间结果。
- 所以能明确不需要去重时，优先考虑 `union all`。

### 追问答案

- 为什么 `union` 更慢？
  因为它多了去重步骤。
- 什么时候必须用 `union`？
  业务语义要求最终结果唯一时。
- `union all` 就一定没问题吗？
  不是，如果业务不允许重复而你用了它，结果就会错。

## 31. InnoDB 为什么用 B+ 树，不用红黑树？

### 30秒结论

- 因为数据库索引不只要优化单点查找，还要优化磁盘 IO、范围查询和页局部性；B+ 树多叉、树高低、叶子有序，整体比红黑树更适合。

### 3分钟回答

- 红黑树是二叉结构，数据量一大，树高会明显增加。
- B+ 树一个节点能容纳更多键值，访问页次数更少。
- 叶子节点有序并常通过链表相连，非常适合范围查询。

### 追问答案

- 为什么 B+ 树更适合范围查询？
  叶子节点有序且通常链起来，可以连续遍历。

- B 树和 B+ 树差别在哪？
  B+ 树把数据主要放在叶子节点，非叶子节点更像纯导航。

- 如果全在内存里，红黑树是不是就更好？
  可以更准确地说：纯内存、以单点查找和频繁增删为主时，红黑树常常更有优势；但数据库索引即使数据大量命中内存，通常还是更偏向 B+ 树。
  
  红黑树可能更优，主要是这几点：
  
  1. 纯内存下，磁盘 IO 这个核心矛盾没了，B+ 树“降低树高、减少页访问次数”的优势会明显变小。
  2. 红黑树是二叉树，单次查找就是沿着左右指针往下走，逻辑直接，实现简单。
  3. 数据量不算特别夸张时，红黑树层数虽然更高，但每层判断代价不大，内存里访问也很快。
  4. 红黑树做插入、删除时，只需要局部旋转和变色，不像 B+ 树那样可能涉及节点分裂、合并、借位，维护成本更低。
  
  但为什么数据库即使很多页都在 Buffer Pool 里，还是更喜欢 B+ 树？
  
  1. 数据库不是只做单点查询，还大量做范围查询、排序、`between`、`order by`、前缀匹配，这些场景 B+ 树天然更合适。
  2. B+ 树一个节点里能放很多 key，虽然在内存里，但这种“一个节点管理一批 key”的局部性通常比红黑树“一个节点一个 key、到处指针跳转”更好，更容易利用 CPU cache。
  3. 数据库索引的基本管理单位本来就是“页”，B+ 树和页结构天然匹配；红黑树如果拿来做数据库索引，和页式存储并不契合。
  
  所以面试里可以这样答：如果脱离数据库索引语境，只讨论纯内存有序集合，红黑树确实可能比 B+ 树更合适；但只要回到数据库索引这个问题，哪怕大量数据在内存中，B+ 树整体仍然更优，因为它不仅是在优化磁盘 IO，也在优化范围扫描、页组织和缓存局部性。

## 32. 为什么建议用自增主键？

### 30秒结论

- 因为 InnoDB 聚簇索引按主键顺序组织数据，自增主键更接近顺序写，页分裂少，而且主键越短，所有二级索引也越轻。

### 3分钟回答

- 自增主键通常是在索引末尾追加，写入更稳定。
- 随机主键会更容易插到索引中间，触发页分裂和碎片。
- 二级索引叶子节点里会带主键值，所以主键越长，所有二级索引都会变胖。

### 追问答案

- UUID 做主键为什么不友好？
  因为它通常又长又随机，会导致页分裂更多、二级索引更大。

- 主键短有什么额外好处？
  二级索引占用空间和维护成本更低。

- 一定只能用自增主键吗？
  不是，但要意识到其他主键方案对索引结构的影响。

- 分库分表后，还适合用数据库自增主键吗？
  通常就没那么合适了。单库里自增主键很友好，因为它能让 InnoDB 更接近顺序写；但分库分表后，数据库自增往往只能保证“单库单表内递增且唯一”，很难天然保证全局唯一。
  
  主要问题有这几个：
  
  1. 每个分片各自自增，只能保证本分片不冲突，不能保证全局不重复。
  2. 如果想靠“不同步长 + 不同起始值”解决，比如 2 台机器一个发奇数一个发偶数，那在扩容、缩容时规则会很麻烦。
  3. 数据迁移、分片重组、历史数据回灌时，也容易碰到主键规则不一致的问题。
  4. 从系统设计角度看，分布式场景通常更需要的是“全局唯一、趋势递增、独立发号”，而不是强依赖某个库表自己的自增能力。
  
  所以面试里可以这样答：单库单表下，自增主键通常是 InnoDB 的优选；但分库分表后，更常见的是用分布式 ID，比如雪花算法、号段模式或业务统一发号。这样做的核心，不只是为了解决唯一性，也是为了后续扩容和路由更稳定。

## 33. 页分裂是什么？会带来什么问题？

### 30秒结论

- 页分裂就是数据页放不下新记录时，把一个页拆成两个页并重新分配数据。它会增加写入成本、带来碎片，并影响缓存和索引效率。

### 3分钟回答

- InnoDB 以页为单位存储索引和数据。
- 如果新数据插入到已满页的中间位置，就可能触发页分裂。
- 页分裂会带来额外数据移动、页管理开销和存储碎片。

### 追问答案

- 什么场景最容易页分裂？
  随机主键插入、索引中间位置频繁插入、页接近满载时。
- 页分裂只影响写吗？
  不只影响写，碎片化后也会影响读性能。
- 怎么减少页分裂？
  选择更顺序的主键、减少随机插入、合理设计索引。

## 34. redo log、undo log、binlog 分别解决什么问题？

### 30秒结论

- redo log 解决持久化和崩溃恢复，undo log 解决回滚和 MVCC 旧版本读取，binlog 解决归档、复制和数据同步。

### 3分钟回答

- redo log 是 InnoDB 引擎层日志，核心是重做已提交但未刷盘的数据页改动。
- undo log 记录旧版本，用于事务回滚，也支撑 MVCC。
- binlog 是 Server 层逻辑日志，主要用于主从复制和归档恢复。

### 追问答案

- redo log 为什么能提高写性能？
  因为它把随机数据页更新先转成顺序日志写入。
- undo log 为什么和 MVCC 有关？
  因为快照读要沿旧版本链读取历史可见版本。
- binlog 为什么主从复制要用它？
  因为它记录的是逻辑层面的最终变更事件。

## 35. 两阶段提交为什么需要？

### 30秒结论

- 两阶段提交是为了让 redo log 和 binlog 在崩溃场景下保持一致，减少主库本地提交和复制链路不一致的问题。

### 3分钟回答

- 一次事务提交既涉及 redo log，也涉及 binlog。
- 如果没有协调机制，就可能出现一种日志成功、另一种失败的情况。
- 两阶段提交通过准备状态和最终提交状态，把两种日志串起来。

### 追问答案

- 不做两阶段提交会怎样？
  可能出现本地数据和 binlog 状态不一致。
- 两阶段提交能保证绝对零丢失吗？
  不能，它解决的是两类日志之间的协调问题。
- 为什么它和主从复制强相关？
  因为从库依赖 binlog 同步，binlog 异常会直接影响复制一致性。

## 36. 主从延迟怎么排查？

### 30秒结论

- 要先确认延迟发生在主库产生日志、网络传输还是从库重放，不要只盯一个延迟数字。

### 3分钟回答

- 先确认延迟是瞬时抖动还是长期积压。
- 再看主库是否有大事务、批量更新、DDL 或突发流量。
- 最后看网络和从库回放能力，比如 CPU、IO、慢 SQL、锁冲突等。

### 追问答案

- 主从延迟最常见根因是什么？
  大事务、从库回放能力不足、慢 SQL、磁盘压力、DDL 和突发写流量。
- 大事务为什么会放大主从延迟？
  因为对应的 binlog 更大、重放时间更长。
- 发现延迟后怎么止血？
  先识别延迟敏感请求，必要时临时切主读并限制大事务。

---

## 37. 主从复制为什么会延迟？

### 30秒结论

- 因为主库提交成功不代表从库已经同步完成，主库产生日志、网络传输、从库重放每一段都可能成为瓶颈。

### 3分钟回答

- 主库写入后会把变更写入 binlog，再通过复制线程发送给从库。
- 从库接收后还要重放，所以任一环节慢了都会形成延迟。
- 大事务、DDL、从库资源紧张和查询压力都可能影响重放速度。

### 追问答案

- 主从复制默认是同步的吗？
  不是，默认是异步。
- 从库查询压力大会影响复制吗？
  会，占用 CPU、IO 或锁资源都会拖慢重放。
- 延迟能完全消灭吗？
  很难，工程上更现实的是把它控制在业务可接受范围内。

---

## 38. 半同步复制解决了什么问题？

### 30秒结论

- 半同步复制解决的是主库刚提交就挂了、从库一个都没收到日志的风险。

### 3分钟回答

- 异步复制下，主库提交后直接返回，不等从库确认。
- 半同步复制要求至少有一个从库确认收到 binlog，主库才返回成功。
- 这样能降低主库故障时的数据丢失窗口，但不等于强一致。

### 追问答案

- 半同步是不是就没有延迟了？
  不是，只是至少有从库收到日志，后续应用仍可能延迟。
- 它的代价是什么？
  主库提交路径多一次等待确认，写入延迟可能上升。
- 为什么说它不是强一致？
  因为收到日志不等于所有从库都已经应用完成。

## 39. 主从切换时要注意什么？

### 30秒结论

- 主从切换不是把新主提起来就结束了，重点是复制位点、一致性窗口、流量路由、连接切换和脑裂防护。

### 3分钟回答

- 要确认候选从库的数据足够新，不能切到明显落后的节点。
- 要处理应用连接和读写路由切换。
- 要防止旧主误恢复后继续写入，形成脑裂。

### 追问答案

- 什么是脑裂？
  多个节点都认为自己是主库并对外提供写服务。
- 为什么复制位点很重要？
  因为它决定候选主库的数据新旧程度。
- 切换后还要做什么？
  校验业务连通性、观察延迟和错误率、重建复制关系。

## 40. 分库分表后事务和主键怎么处理？

### 30秒结论

- 分库分表后，本地事务和单库自增主键都不再天然成立。事务要考虑分布式一致性方案，主键要考虑全局唯一和有序性。

### 3分钟回答

- 跨分片后，本地事务只能保证单库内原子性。
- 强一致就要考虑 TCC、消息最终一致性等方案，复杂度会上升。
- 主键方面通常要改成全局 ID，比如雪花算法或号段模式。

### 追问答案

- 为什么分库后本地事务不够用了？
  因为一次业务操作可能涉及多个库，本地事务跨库失效。
- 全局 ID 要注意什么？
  唯一性、生成性能、是否大致有序、是否方便路由。
- 分库分表最容易踩什么坑？
  跨库 join 难、分页排序复杂、事务变难、热点分片和迁移成本高。

## 41. 雪花算法有什么问题？

### 30秒结论

- 雪花算法的优点是本地生成、性能高、基本趋势递增，适合分布式场景；但它并不完美，主要问题集中在时钟回拨、机器号管理、不保证严格全局递增，以及单机每毫秒发号上限这些方面。

### 3分钟回答

- 雪花算法一般会把 ID 拆成时间戳、机器号、序列号几部分，所以它很依赖系统时间和机器标识的正确性。
- 最大风险通常是时钟回拨。如果机器时间被 NTP 校正或人为调小，当前时间戳比上一次发号时还小，就可能导致重复 ID，或者系统为了避免重复而暂停发号。
- 它依赖机器号唯一。如果两台机器错误地配置成同一个机器号，那么在相同时间片内就有可能生成重复 ID。
- 雪花算法通常只是“趋势递增”，不是数据库自增主键那种严格连续、严格单调。多台机器并发发号时，生成结果会交错。
- 单机同一毫秒内能生成的 ID 数量也有限，超过序列号容量后，就得等待下一毫秒。
- 从数据库角度看，它比 UUID 更适合做主键，因为通常更短、也更接近递增；但和单库自增主键相比，对 InnoDB 的写入友好度通常还是差一点。

### 追问答案

- 雪花算法最经典的问题是什么？
  时钟回拨。因为它强依赖时间戳，一旦时间回退，就可能出现重复 ID 或发号失败。
- 为什么雪花算法不是严格有序？
  因为多机并发时，不同机器生成的 ID 会交错，所以一般只能保证大体递增，不能保证全局严格单调。
- 雪花算法适合做数据库主键吗？
  在分库分表场景很常见，通常比 UUID 更适合；但如果是单库单表，数据库自增主键对 InnoDB 往往还是更友好。
- 雪花算法一定比号段模式好吗？
  不一定。雪花算法优点是本地生成、性能高；号段模式优点是更容易控制、对时钟不敏感，具体要看系统对可用性、顺序性和运维复杂度的要求。

## 42. 号段模式和雪花算法怎么选？

### 30秒结论

- 如果更看重本地发号、高性能、弱依赖中心服务，通常选雪花算法；如果更看重稳定、可控、对时钟不敏感，通常选号段模式。两者都常用于分布式 ID，没有绝对谁更好，关键看系统诉求。

### 3分钟回答

- 雪花算法的特点是本地生成，机器拿到机器号后就能自己发号，性能很高，也不需要每次都请求中心服务。
- 但雪花算法强依赖时间戳和机器号管理，时钟回拨、机器号冲突，都是它的典型风险。
- 号段模式的思路是先从中心服务或数据库批量申请一段连续 ID，再在本地内存里慢慢消费，所以它对时钟不敏感，也不用关心机器时间是否回退。
- 号段模式通常更容易做运维和排障，因为 ID 分配规则更集中、可控，出了问题也更容易定位。
- 但它也不是没有代价。它依赖中心发号服务或号段表，如果设计不好，中心服务可能成为瓶颈或单点。
- 从数据库主键角度看，两者通常都比 UUID 更适合。雪花算法一般是“趋势递增”，号段模式往往更接近连续递增；如果非常在意顺序写友好性，号段模式通常会更占优势。

### 追问答案

- 号段模式为什么对时钟不敏感？
  因为它的核心不是依赖本机时间戳发号，而是先申请一段 ID 再顺序使用，所以机器时间回拨一般不会直接影响唯一性。
- 号段模式的风险是什么？
  中心发号服务或号段表如果设计不好，可能成为性能瓶颈、可用性风险点，另外号段申请也要考虑双 buffer、预加载和容灾。
- 什么场景更适合雪花算法？
  机器很多、并发高、希望节点尽量独立发号、不想强依赖中心服务时，雪花算法通常更合适。
- 什么场景更适合号段模式？
  更关注稳定性、可控性、排障便利性，且可以接受引入中心发号能力时，号段模式通常更合适。
- 两者谁更适合做数据库主键？
  一般都可以，但如果更强调接近顺序写、减少随机插入影响，号段模式通常会更友好一些；如果更强调分布式自治和高吞吐，雪花算法更常见。

## 43. 为什么不建议 `select *`？

### 30秒结论

- 因为它容易多查无用列、增加网络和内存开销，还会破坏覆盖索引，让原本能快的查询变慢。

### 3分钟回答

- 很多场景根本不需要所有列，`select *` 会把无用数据也查出来。
- 它更容易导致查询列超出索引覆盖范围，从而回表。
- 表结构变更后，返回结果也会变，可能影响兼容性。

### 追问答案

- `select *` 什么时候危害更大？
  宽表、大字段、高频查询、原本可以走覆盖索引的场景。
- 为什么它会影响覆盖索引？
  因为一旦查询列超出索引包含范围，就不得不回表。
- 小表场景也不能用吗？
  不是绝对不能，但工程习惯上仍更建议按需取列。

## 44. 为什么不建议在索引列上做函数操作？

### 30秒结论

- 因为对索引列做函数、计算或类型转换，往往会破坏原有索引的有序性，导致索引失效。

### 3分钟回答

- 索引是基于原始列值建立的有序结构。
- 在列上套函数后，优化器通常没法直接利用原索引定位。
- 隐式类型转换也有类似问题，本质上也是对列值做处理。

### 追问答案

- 隐式类型转换为什么也危险？
  因为本质上和在列上套函数类似，也可能破坏索引利用。
- 时间字段查询怎么改写更好？
  改成时间范围，比如 `>=` 和 `<` 的组合。
- 一定完全不能做函数吗？
  不是语法不能，而是高频核心 SQL 不建议这么写。

## 45. 批量插入、批量更新怎么做更稳妥？

### 30秒结论

- 原则是分批、限速、可重试、避开高峰，不要一把提交超大事务。

### 3分钟回答

- 一次性改太多数据会把事务做大，带来锁等待、日志膨胀和主从延迟。
- 常见做法是按固定批次拆分，比如每批几百到几千行。
- 还要保证幂等和断点续跑能力。

### 追问答案

- 为什么批量任务要分批提交？
  因为能控制单事务大小，降低锁持有时间和回滚成本。
- 批量更新最怕什么问题？
  锁等待、主从延迟、高峰期拖慢在线业务，以及失败后难恢复。
- 怎么保证失败后可恢复？
  做好幂等、记录处理进度、支持断点续跑。

## 46. 删除大量数据为什么不能一次删完？

### 30秒结论

- 因为一次删太多本质上就是大事务，会带来长时间锁占用、日志膨胀、主从延迟和回滚风险。

### 3分钟回答

- 大量删除不仅会锁住大量数据，还会生成大量 undo、redo 和 binlog。
- 如果中途失败或要回滚，成本会非常高。
- 工程上通常按主键或时间范围分批删，必要时用归档或冷热分离方案。

### 追问答案

- 为什么删除比想象中更重？
  因为删除同样要记录日志、维护索引、支持回滚和复制。
- 分批删除怎么切分更稳妥？
  常见按主键递增区间、时间窗口或固定条数分批。
- 什么场景适合归档而不是直接删？
  历史数据还可能查询、审计要求保留、或直接删除风险太高的场景。

## 47. 线上加索引要注意什么？

### 30秒结论

- 线上加索引要关注锁影响、执行时长、磁盘空间、主从延迟和回滚预案。它不是一条 DDL 命令的问题，而是一项变更操作。

### 3分钟回答

- 先确认索引确实有必要，再评估 DDL 对业务的影响和资源消耗。
- 要检查磁盘空间和主从链路，因为建索引会消耗大量 CPU、IO 和存储。
- 选低峰时段执行，做好监控、演练和回滚预案。

### 追问答案

- 为什么线上加索引可能影响业务？
  因为建索引要扫描数据、构建新结构并消耗大量资源。
- 加索引前要看哪些指标？
  表大小、磁盘空间、CPU/IO、主从延迟、业务高峰时段和目标 SQL 执行计划。
- 怎么降低加索引风险？
  选低峰、优先在线 DDL、先验证收益、做好监控和回滚预案。
