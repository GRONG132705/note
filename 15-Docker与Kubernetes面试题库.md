## 一页学习路线

- 第一轮先学 `1`：先把 Docker 基础概念分清，不然后面 Kubernetes 很容易变成背名词。
- 第二轮学 `2`：理解 K8s 核心对象分别承担什么职责，不要把它们混成一层。
- 第三轮学 `3`：把 Java 服务真正放进容器和 K8s 环境里思考，关注探针、配置、扩缩容和资源限制。
- 第四轮学 `4`：从发布、排障、重启、自愈和资源治理角度建立工程化理解。
- 第五轮学 `5`：再补底层隔离、控制平面、模板化交付、安全边界和有状态场景，建立更完整的平台视角。

## 高频易混点

- 镜像不是容器，镜像更像静态交付物，容器更像镜像运行起来后的实例。
- Pod 不是单纯一台机器，它是 Kubernetes 调度和运行的基本单位。
- Service 不是网关，Ingress 也不是 Deployment，它们作用层完全不同。
- 探针不是“有就行”，配错探针反而可能把服务打得更不稳定。
- 容器化不等于自动高可用，Kubernetes 也不是只要上了就一定稳定。
- CPU 利用率不高不代表服务就没资源问题，CPU Throttling 一样可能把 RT 打抖。
- Secret 不等于普通配置，RBAC 也不等于应用里的角色权限模型。
- 无状态应用和有状态应用在 K8s 里的治理方式差异很大，不能一套 Deployment 思路全覆盖。

## 学习主线

- 先理解打包和交付：
  代码怎么变成镜像，镜像怎么变成容器。
- 再理解编排和治理：
  Pod 怎么运行，Deployment 怎么管理副本，Service 怎么暴露服务。
- 再理解 Java 服务落地：
  配置怎么注入、资源怎么限制、探针怎么配、扩缩容怎么做。
- 最后理解运行态：
  发布、自愈、排障、资源治理和故障止血。
- 再补平台底层和生产边界：
  隔离原理、控制平面、模板化交付、安全边界和有状态场景。

## 设计取舍

- 为什么先容器化，再谈 Kubernetes：
  如果镜像构建、配置解耦、日志和资源意识都没建立好，直接上 K8s 只会把复杂度放大。
- 为什么探针不能只求“尽快发现异常”：
  探针太激进会把启动慢、预热长的 Java 服务误判成不健康，反而制造重启风暴。
- 为什么扩容不等于问题解决：
  扩容只放大应用层承载，如果数据库、缓存回源或下游依赖才是真短板，扩 Pod 只会更快把压力打过去。
- 为什么资源限制不能拍脑袋：
  Java 服务同时受 JVM 内存模型和容器资源限制影响，配置过紧和过松都会带来代价。
- 为什么 Kubernetes 不等于天然高可用：
  平台提供的是能力，能不能稳定运行还取决于发布、探针、配置、监控和团队治理能力。
- 为什么 Helm / GitOps 往往比手写 YAML 更适合生产：
  规模一大以后，模板化、版本化和变更可追踪能力比“能不能写出 YAML”更重要。
- 为什么安全边界不能只看应用代码：
  容器权限、ServiceAccount、NetworkPolicy、Secret 管理都会影响运行时风险面。

  这个说法为什么不完整，扩容前最该先确认什么。
- 自测重点：
  你能不能区分应用层扩容和数据库、缓存、下游依赖瓶颈。

### 自测题 3

- 某 Pod 频繁 OOMKilled，但节点整体内存并没有完全打满。
- 你先回答：
  这更应该先看 JVM 参数、容器资源限制，还是业务流量变化。
- 自测重点：
  你能不能把 JVM 内存和容器限制放在一起分析。

### 自测题 4

- 面试官问“为什么 Kubernetes 不只是帮你起容器”。
- 你先回答：
  除了启动容器，它还把哪些工程动作平台化了。
- 自测重点：
  你能不能顺着发布、自愈、扩缩容、服务发现、配置治理、运行观测讲出来。

### 自测题 5

- 某 Java 服务 CPU 平均利用率不高，但线上 RT 偶发毛刺明显，平台监控里能看到 CPU Throttling。

  为什么“CPU 看起来不高”并不等于没有 CPU 资源问题。
- 自测重点：
  你能不能想到 CFS quota、容器限制、线程竞争和日志开销这几层。

### 自测题 6

- 面试官问“为什么 Helm、GitOps、RBAC 这类东西也属于高级开发要懂的范围”。
- 你先回答：
  它们和真正的服务交付、权限边界、线上治理是什么关系。
- 自测重点：
  你能不能从工程化和平台协作角度回答，而不是只把它们当平台团队的事。

## 实战案例

### 案例 1：Pod 频繁重启，为什么根因最后是探针配置太激进？

#### 背景

- 某 Java 服务刚迁到 K8s，镜像构建和部署都正常，但上线后部分实例频繁重启。

#### 暴露出来的问题

- 应用本身最终能正常启动，但启动时间较长，依赖初始化也偏慢。
- 探针配置过于激进，平台在服务真正 ready 前就反复判定失败。

#### 处理思路

- 先看 Pod 事件、重启原因和探针失败日志。
- 再结合 Java 服务启动时长、预热阶段和依赖初始化时间调整探针。
- 最后把健康检查分成就绪和存活两层，避免职责混用。

#### 学习重点

- 很多容器平台问题不是“程序坏了”，而是治理配置不适配服务特性。

### 案例 2：扩了很多 Pod，为什么系统还是扛不住？

#### 背景

- 某服务高峰期通过平台快速扩容，应用副本数明显增加，但用户侧 RT 并没有明显改善。

#### 暴露出来的问题

- 应用层扩容后，请求更快打到数据库和下游依赖，结果真正短板被进一步放大。
- 数据库连接、缓存回源和依赖 RT 成为新的瓶颈。

#### 处理思路

- 先确认系统真正的最短板是不是应用层。
- 如果瓶颈在数据库、缓存回源或下游服务，先做缓存、限流、削峰或依赖治理，而不是只继续加 Pod。
- 把扩容放回容量评估和瓶颈定位框架里看。

#### 学习重点

- Kubernetes 的扩容能力非常有用，但它解决不了所有层的瓶颈。

### 案例 3：CPU 不高，为什么接口 RT 还是会出现明显毛刺？

#### 背景

- 某 Java 服务部署在 K8s 中，监控看 CPU 平均利用率不算高，但接口 RT 经常出现短时抖动。

#### 暴露出来的问题

- 容器层存在 CPU Throttling，说明进程在 CFS quota 下被限制执行时间。
- 应用线程、日志输出、GC 或短时间并发峰值会把有限 CPU 时间片迅速打满。

#### 处理思路

- 先确认是否真的发生 Throttling，而不是只盯平均 CPU 利用率。
- 再结合容器 CPU limit、线程池规模、热点锁竞争和日志开销判断是真正的算力不足，还是配额过紧。
- 最后再决定是调资源限制、调并发模型，还是先降无意义消耗。

#### 学习重点

- 容器里的“CPU 看起来不高”不等于没有资源限制问题，平台限制和应用行为要放在一起看。

- `1.4` 为什么不能把“能跑在 Docker 里”简单理解成“已经适合生产”？

## 知识地图

- `镜像构建 -> 容器运行 -> 网络与存储 -> 交付稳定性 -> 工程边界`

## 学习目标

- 分清镜像、容器、仓库、网络、卷的边界。
- 理解 Docker 对交付一致性的价值，而不是只停在“能启动”。
- 建立容器化只是第一步的意识。

## 先修知识

- Linux 基础
- 基础服务部署概念

## 本模块总结

- Docker 的核心价值，不只是打包方便，而是运行环境一致性更强。
- 镜像是交付物，容器是运行态实例，这两层一定要分清。
- 真正上生产时，容器化只是起点，运行治理才是难点。

## 典型业务场景

- 开发环境能跑、测试环境不能跑，环境差异导致交付不稳定。
- 服务已经容器化，但上线后依然出现配置、资源、日志和重启治理问题。

## 最小实践建议

- 自己做一次最小 Java 服务镜像，体会镜像和容器的区别。
- 复习时按“镜像、容器、网络、存储、仓库”五个词去串 Docker 基础。

## 1.1 Docker 镜像和容器有什么区别？

### 核心讲解

- Docker 镜像可以理解成一个静态的运行模板，里面包含应用代码、依赖和运行环境。
- 容器则是镜像运行起来后的实例。
- 一个镜像可以启动多个容器。

### 深入理解

- 这题最重要的是分清“交付物”和“运行实例”两层。很多人学 Docker 时，把镜像和容器混成一个概念，后面一到发布和排障就会乱。
- 镜像更偏构建和交付，容器更偏运行和治理。高级岗位更想看到你知道这两层职责不同。

### 追问答案

- 为什么一个镜像可以起多个容器？
  因为镜像是模板，容器是模板运行出来的实例。

### 学习提示

- 记一句话：镜像偏静态，容器偏运行态。

---

## 1.2 Docker 为什么能帮助服务交付更稳定？

### 核心讲解

- 因为它把应用和依赖打成统一镜像，减少了“环境不一致”带来的部署问题。
- 它让开发、测试、预发、生产更容易保持运行环境一致。
- 对 Java 服务来说，这会显著降低“本地能跑、线上不行”的概率。

### 深入理解

- Docker 的价值不是神奇地让服务更快，而是让交付过程更稳定、更可复制。
- 对高级 Java 而言，这题的重点在于“环境一致性”而不是“虚拟化原理”。
- 但也要补一句边界：交付一致不等于运行无忧，探针、资源、配置、日志和治理仍然要单独处理。

### 追问答案

- Docker 是不是解决了所有部署问题？
  不是，它主要解决交付一致性，不自动解决运行治理。

### 学习提示

- 回答时最好补一句“交付稳定”和“运行稳定”是两层问题。

---

## 1.3 Docker 网络、数据卷、镜像仓库分别解决什么问题？

### 核心讲解

- Docker 网络解决容器之间以及容器对外的通信问题。
- 数据卷解决容器生命周期和数据生命周期分离的问题。
- 镜像仓库解决镜像分发、版本管理和统一交付问题。

### 深入理解

- 这题是 Docker 基础里很重要的“三件套”。网络决定怎么连，卷决定数据怎么放，仓库决定镜像怎么发。
- 很多面试会从这里看你是不是只会写 `Dockerfile`，还是理解容器运行起来后的实际问题。

### 追问答案

- 为什么数据卷重要？
  因为容器本身不是天然适合持久化数据的。

### 学习提示

- 建议按“通信、持久化、分发”三个关键词去记。

---

## 1.4 为什么不能把“能跑在 Docker 里”简单理解成“已经适合生产”？

### 核心讲解

- 因为生产环境还要考虑配置注入、日志、探针、资源限制、重启、自愈、发布和监控。
- 能启动只是最小门槛。
- 真正上生产，关键是运行治理能力。

### 深入理解

- 这题最能体现工程意识。很多团队容器化做完后，以为工作结束了，但生产问题往往才刚开始出现。
- Java 服务尤其如此，因为 JVM 参数、内存限制、启动探针、GC、线程池都可能和容器环境产生新问题。

### 追问答案

- 生产环境额外要关注什么？
  配置、资源限制、探针、日志、监控、发布和回滚。

### 学习提示

- 记一句话：容器化解决交付，生产稳定靠治理。

## 典型业务场景

- 服务有多个副本，需要统一管理发布和重启。

- 自己画一张图，把 Pod、Deployment、Service、Ingress、ConfigMap、Secret 串成一条调用链。
- 复习时尽量按“跑、管、连、进、配”去记。

## 2.1 Kubernetes 为什么会成为容器编排的核心方案？

### 核心讲解

- 因为它不只负责启动容器，还能统一管理调度、副本、自愈、服务发现、发布、扩缩容和配置。
- 当服务数量多、实例多、发布频繁时，单靠脚本或单机编排很难扛住复杂度。
- Kubernetes 更像一个运行时治理平台。

### 深入理解

- 这题关键不是“它很流行”，而是“它解决了什么复杂度”。一旦服务多起来，光把容器跑起来并不难，难的是谁来管副本、谁来做发布、谁来保证失败后自动恢复。
- 所以 Kubernetes 对高级 Java 来说，更重要的是它承接了微服务运行治理的很多工程问题。

### 追问答案

- K8s 的核心价值是什么？
  是把容器运行治理平台化、标准化。

### 学习提示

- 记一句话：K8s 重点不是起容器，而是管容器。

---

## 2.2 Pod、Deployment、Service 分别怎么理解？

### 核心讲解

- Pod 是 Kubernetes 里最小的运行单元。
- Deployment 负责管理 Pod 副本、发布和滚动更新。
- Service 负责为一组 Pod 提供稳定访问入口和服务发现。
- 这三者分别解决“跑、管、连”。

### 深入理解

- 这题是 K8s 的基本盘。很多人学 Kubernetes 卡住，就是因为没把 Pod、Deployment、Service 分层。
- 高级岗位里，面试官往往不要求你背 YAML，但会希望你能讲清楚每个对象的职责和边界。

### 追问答案

- 为什么 Service 重要？
  因为 Pod 可能变化和重建，访问入口需要稳定。

### 学习提示

- 记住：Pod 跑实例，Deployment 管副本，Service 管访问。

---

## 2.3 Ingress、ConfigMap、Secret 分别适合解决什么问题？

### 核心讲解

- Ingress 主要解决外部流量入口、域名和路由规则管理。
- ConfigMap 用来管理普通配置。
- Secret 用来管理敏感配置，比如密码、密钥、令牌。
- 它们分别解决入口和配置治理问题。

### 深入理解

- 这题要学的是“入口治理”和“配置治理”怎么从镜像里解耦出来。
- 高级 Java 岗位经常会问配置怎么管、敏感信息怎么管、域名路由怎么收敛，这题正好就是这些问题的最小入口。

### 追问答案

- 为什么 Secret 不能简单等同普通配置？
  因为敏感信息需要更严格的访问和管理边界。

### 学习提示

- 记忆时按“进、配、密”三个词去背。

---

## 2.4 为什么 Kubernetes 不只是“帮你起容器”？

### 核心讲解

- 因为它还负责调度、发布、自愈、探针、扩缩容、服务发现、配置管理和故障恢复。
- 如果只把它看成起容器工具，就会低估它在运行治理里的价值。
- 也会低估它的学习成本和边界。

### 深入理解

- 这题非常适合高级岗位。因为很多团队并不是不会写镜像，而是不会用平台视角理解服务运行。
- 真正成熟的理解是：Kubernetes 把很多原来靠脚本、手工和经验完成的动作平台化了。

### 追问答案

- K8s 为什么会让团队治理方式变化？
  因为发布、扩容、自愈、配置和服务暴露都被平台接管了一部分。

### 学习提示

- 回答时可以补一句：平台能力增强，也意味着配置和治理复杂度上升。

---

# 3. Java 服务部署与治理

## 本模块题目

- `3.1` Java 服务部署到 Docker / K8s 后，要重点关注哪些问题？
- `3.2` 探针为什么重要？存活探针、就绪探针一般怎么理解？
- `3.3` Java 服务在 Kubernetes 里怎么做配置管理？
- `3.4` Java 服务在 Kubernetes 里怎么做扩缩容？
- `3.5` 为什么 Java 服务在容器里要特别关注内存和资源限制？

## 知识地图

- `镜像交付 -> 探针 -> 配置 -> 扩缩容 -> JVM资源治理`

## 学习目标

- 理解 Java 服务进容器后会多出哪些治理点。
- 掌握探针、配置、扩缩容、资源限制这些高频问题。
- 建立“Java 服务在容器里不是裸跑 JVM”的意识。

## 先修知识

- JVM 基础
- Docker 与 K8s 核心对象

## 本模块总结

- Java 服务一旦进入容器和 K8s，重点就不只是代码，还包括 JVM 参数、探针、资源限制和配置治理。
- 很多线上问题不是业务代码写错，而是探针配错、资源限制失衡或扩缩容策略不合适。

## 典型业务场景

- Java 服务启动慢，探针配太激进导致频繁重启。
- 配置写死在镜像里，上线时难以区分环境。
- 容器资源限制太紧，导致 OOMKilled 或频繁抖动。

## 最小实践建议

- 复习时把“探针、配置、扩缩容、资源限制”四个点和 Java 服务特点对应起来。
- 练习把 JVM、线程池、GC、探针和资源限制放在一个排障视角里讲。

## 3.1 Java 服务部署到 Docker / K8s 后，要重点关注哪些问题？

### 核心讲解

- 重点通常包括：启动时间、探针、配置注入、日志、资源限制、JVM 参数、扩缩容和发布回滚。
- Java 服务相比很多轻量服务，更容易受启动时长、内存和 GC 影响。
- 所以容器化之后，JVM 和平台配置要一起看。

### 深入理解

- 这题真正想考的是：你有没有把 Java 服务放进“运行平台”里思考，而不是只会讲业务代码。
- 高级岗位里，这题很容易和 JVM、并发、高并发设计串起来。

### 追问答案

- 为什么 Java 服务在容器里更要关注启动时间？
  因为探针和发布节奏都可能受启动时长影响。

### 学习提示

- 回答时建议按“探针、配置、资源、扩缩容、发布”五层去讲。

---

## 3.2 探针为什么重要？存活探针、就绪探针一般怎么理解？

### 核心讲解

- 存活探针更偏“进程是不是还活着、是否需要重启”。
- 就绪探针更偏“实例现在能不能接流量”。
- 探针配得好，平台能更稳地自愈；配不好，平台会把本来正常的服务打坏。

### 深入理解

- 这题特别典型，因为很多线上不稳定不是服务真坏了，而是探针过于激进。
- 对 Java 服务来说，启动时间长、预热阶段长、依赖初始化慢，都可能让探针误判。
- 所以高级岗位更想看你是不是知道探针是运行治理的一部分，而不是照抄一个 `/health`。

### 追问答案

- 为什么就绪探针和存活探针不能混用？
  因为一个管接流量，一个管重启，职责不同。

### 学习提示

- 记一句话：就绪探针决定接不接流量，存活探针决定要不要重启。

---

## 3.3 Java 服务在 Kubernetes 里怎么做配置管理？

### 核心讲解

- 一般会把环境相关配置从镜像中解耦，结合 ConfigMap、Secret、环境变量和配置中心管理。
- 目标是不让同一套镜像因为环境不同而反复重打。
- 同时敏感配置要和普通配置分层管理。

### 深入理解

- 这题本质上是在看你有没有“镜像不可变、配置可变”的意识。
- 高级 Java 团队里，配置管理不是小事，因为它直接影响发布稳定性和安全边界。

### 追问答案

- 为什么不建议把配置直接写死在镜像里？
  因为环境变化会迫使你频繁重打镜像，交付和治理都会变差。

### 学习提示

- 回答时顺手补一句：普通配置和敏感配置最好分开管。

---

## 3.4 Java 服务在 Kubernetes 里怎么做扩缩容？

### 核心讲解

- 一般会结合副本数、资源使用情况、流量指标和平台自动扩缩容能力来做。
- 扩容不是只加 Pod 数量，还要看下游依赖、数据库、缓存和线程池是不是跟得上。
- 所以扩缩容一定要和容量评估连在一起。

### 深入理解

- 这题特别适合高级岗位，因为很多人会说“上 K8s 可以自动扩容”，但不会补“扩了以后瓶颈会不会转移”。
- 真正成熟的理解是：扩容只能放大应用层承载，不能自动消除数据库、缓存回源或下游依赖瓶颈。

### 追问答案

- 为什么扩容不一定解决问题？
  因为如果真正瓶颈在数据库或下游依赖，应用层多副本只会更快把压力打过去。

### 学习提示

- 回答这题时最好顺手补一句“扩容前先找短板”。

---

## 3.5 为什么 Java 服务在容器里要特别关注内存和资源限制？

### 核心讲解

- 因为 JVM 本身就有堆、元空间、线程栈、直接内存等多类内存消耗，而容器资源限制又会直接影响 OOMKilled、GC 和稳定性。
- 如果资源限制配得太紧，服务可能频繁重启或抖动。
- 如果配得太松，又容易造成资源浪费和节点挤压。

### 深入理解

- 这题最能体现 Java 服务和很多脚本型服务的差异。Java 服务在容器里，资源问题不只是“给多少内存”，而是 JVM 自己的内存模型和平台限制会叠加。
- 所以这题回答时最好把 JVM 和容器资源一起讲，会显得很成熟。

### 追问答案

- 为什么 Java 服务比很多轻量服务更怕资源限制配错？
  因为 JVM 的内存使用比单纯进程常更复杂，启动和 GC 也更受资源限制影响。

### 学习提示

- 记一句话：容器限制配的是进程空间，JVM 用的是多种内存来源。

---

# 4. 发布、排障与工程实践

## 本模块题目

- `4.1` 滚动发布、蓝绿发布、灰度发布在容器平台里怎么理解？
- `4.2` Kubernetes 里服务不稳定时，一般先看什么？
- `4.3` Pod 频繁重启时，一般怎么排查？
- `4.4` 资源限制配错会带来什么问题？
- `4.5` 容器化和 Kubernetes 最终带来的，不只是部署方便，还包括什么能力？

## 知识地图

- `发布方式 -> 运行异常 -> 排障主线 -> 资源治理 -> 平台能力`

## 学习目标

- 理解容器平台里的发布、排障和治理问题。
- 能回答 Pod 重启、探针异常、资源限制失衡这些高频问题。
- 建立平台能力和工程能力的联系。

## 先修知识

- K8s 核心对象
- Java 服务部署与治理

## 本模块总结

- 容器平台真正值钱的地方，不只是部署统一，而是发布、自愈、扩缩容和运行治理能力。
- 高级岗位更想看你是不是知道服务进平台后，新的问题会出现在哪里。

## 典型业务场景

- 新版本上线后部分实例反复重启。
- 探针、资源限制和扩缩容策略配置不当，导致服务不稳定。
- 平台能力很强，但团队并没有真正用好发布和治理能力。

## 最小实践建议

- 把“发布方式”和“故障止血方式”放在一起练习。
- 复习时把 Pod 重启、探针失败、资源失衡当成三个固定排障入口。

## 4.1 滚动发布、蓝绿发布、灰度发布在容器平台里怎么理解？

### 核心讲解

- 滚动发布通常是逐步替换旧实例。
- 蓝绿发布通常是新旧两套环境切换流量。
- 灰度发布通常是先让一部分流量进入新版本再逐步放量。
- 它们的核心区别在于切流方式和风险控制方式。

### 深入理解

- 这题高级感在于你能不能说出“为什么不同场景下选不同发布方式”。
- 容器平台让发布更标准化，但选择哪种方式仍然要看业务风险、回滚要求和流量控制能力。

### 追问答案

- 为什么高风险业务更喜欢灰度或蓝绿？
  因为回滚和风险隔离更可控。

### 学习提示

- 回答时建议顺手补一句“发布方式要和流量治理能力配套”。

---

## 4.2 Kubernetes 里服务不稳定时，一般先看什么？

### 核心讲解

- 一般先看 Pod 状态、重启次数、事件、探针结果、资源使用和最近发布变更。
- 然后再看应用日志、JVM 指标和下游依赖。
- 不建议一上来就只盯代码。

### 深入理解

- 这题本质上是在看平台视角。服务不稳定时，平台层的事件和状态信息往往比应用日志更早给出方向。
- 对高级 Java 来说，排障不能只停在 JVM，而要先确认平台层有没有明显信号。

### 追问答案

- 为什么先看最近发布变更？
  因为很多不稳定问题都和新版本、配置变更或探针配置调整直接相关。

### 学习提示

- 记一句话：先看平台状态，再看应用细节。

---

## 4.3 Pod 频繁重启时，一般怎么排查？

### 核心讲解

- 先看是探针失败、OOMKilled、启动异常、依赖未就绪，还是配置错误导致的重启。
- 然后结合事件、日志、资源限制和探针配置去缩小范围。
- 频繁重启通常不是一个“普通小问题”，而是治理配置明显不合理。

### 深入理解

- 这题非常高频。真正重要的不是背重启原因，而是知道先分类型。
- 如果不先分清是 OOM、探针、启动失败还是配置错误，排查很容易乱。
- 高级岗位会特别看你是否有“平台事件 + 应用日志 + 资源限制”一起看的意识。

### 追问答案

- 为什么 Pod 重启问题常和探针、资源限制一起出现？
  因为它们直接决定平台会不会认为实例不健康或资源超限。

### 学习提示

- 回答时建议先说“先分重启类型，再找具体原因”。

---

## 4.4 资源限制配错会带来什么问题？

### 核心讲解

- 配得太低可能导致 OOMKilled、频繁 GC、探针失败和服务抖动。
- 配得太高可能导致节点资源被抢占、调度不均、资源利用率低。
- 资源限制不是越保守越好，而是要结合服务特性和压测结果。

### 深入理解

- 这题的关键是建立“资源限制也是设计题”的意识。很多团队把资源值当经验数字填，最后上线后问题很大。
- 对 Java 服务而言，资源限制、JVM 参数、线程池和流量模型是联动的，不能分开看。

### 追问答案

- 为什么资源限制问题和 JVM 问题经常连在一起？
  因为 JVM 内存和 GC 行为会直接受容器资源限制影响。

### 学习提示

- 记住：资源限制要靠压测和运行指标验证，不要拍脑袋。

---

## 4.5 容器化和 Kubernetes 最终带来的，不只是部署方便，还包括什么能力？

### 核心讲解

- 更重要的还包括统一交付、标准化发布、自愈、弹性扩缩容、配置治理、服务治理和运行时观测能力。
- 它让很多工程动作平台化了。
- 但前提是团队真的会用这些平台能力，而不是只把它当“新机器”。

### 深入理解

- 这题最适合做收尾题。因为真正成熟的理解，一定会把 Docker 和 Kubernetes 放回工程体系里看，而不是当成一个技术孤岛。
- 高级岗位面试里，如果你能把“交付、发布、运行、观测、治理”串成一条线，这题会很加分。

### 追问答案

- 为什么有些团队上了 K8s 还是很痛苦？
  往往不是平台没用，而是团队还没建立配套的配置、发布、监控和排障能力。

### 学习提示

- 回答时建议按“交付标准化 + 运行治理平台化”两层去讲。

# 5. 生产级治理与安全边界

## 本模块题目

- `5.1` 容器隔离依赖什么底层机制？Namespace 和 Cgroups 怎么理解？
- `5.2` kube-apiserver、scheduler、controller-manager、etcd 分别做什么？
- `5.3` Helm、Kustomize、GitOps 分别解决什么问题？
- `5.4` RBAC、ServiceAccount、NetworkPolicy 为什么重要？
- `5.5` StatefulSet、HPA、Sidecar 分别适合什么场景？

## 知识地图

- `隔离原理 -> 控制平面 -> 模板化交付 -> 安全边界 -> 有状态与弹性能力`

## 学习目标

- 建立 Docker / K8s 在底层隔离、控制平面和生产治理上的基本认知。
- 理解模板化交付、安全边界和有状态场景为什么是高级开发也要懂的内容。
- 形成“平台能力和应用治理是连在一起的”这层意识。

## 先修知识

- Docker 基础
- Kubernetes 核心对象
- Java 服务部署与治理

## 本模块总结

- 这一组重点不是让你变成平台工程师，而是让你知道平台底层和治理边界会怎样反过来影响 Java 服务。
- 高级开发至少要知道：容器怎么隔离、K8s 怎么协调、配置怎么模板化、安全怎么隔离、有状态和无状态怎么区分。

## 常见误解

- 认为 Docker 只是“轻量虚拟机”，却不了解隔离依赖的是 Namespace 和 Cgroups。
- 认为 K8s 只是一堆对象 YAML，不理解控制平面如何把声明式状态收敛成实际状态。
- 认为 Helm、RBAC、NetworkPolicy 这些都只是平台团队的事，和应用稳定性无关。

## 典型业务场景

- 应用在容器里表现异常，但团队说不清到底是代码问题、JVM 问题还是 Cgroup 限制问题。
- 多环境、多服务部署时，手写 YAML 越来越难维护，变更回滚和差异控制变得混乱。
- 服务迁到 K8s 后，权限、网络和 Secret 边界没补齐，导致运行风险上升。

## 最小实践建议

- 复习时至少把 Namespace、Cgroups、Pod、Deployment、Service、Ingress 这几层关系口头讲顺。
- 看一份 Helm chart 或 Kustomize overlay，理解模板化交付和手写 YAML 的差异。
- 自己想一遍：如果要在 K8s 上部署数据库、MQ、缓存，为什么不能只按 Deployment 思路照搬。

## 5.1 容器隔离依赖什么底层机制？Namespace 和 Cgroups 怎么理解？

### 核心讲解

- 容器并不是独立内核，它本质上还是宿主机上的进程，只是通过 Namespace 和 Cgroups 实现了隔离和限制。
- Namespace 更偏“视图隔离”，比如进程、网络、挂载点、主机名。
- Cgroups 更偏“资源限制”，比如 CPU、内存等资源怎么被约束和统计。

### 深入理解

- 这题对高级开发很重要，因为它能帮你理解为什么容器里的资源视角和宿主机不完全一致，也能解释 CPU Throttling、OOMKilled 这些平台现象。
- 不需要把内核细节讲成底层专题，但至少要知道：容器不是虚拟机，资源和隔离都是通过 Linux 机制拼起来的。

### 追问答案

- 为什么说 Docker 容器不是完整虚拟机？
  因为它没有独立内核，本质上还是宿主机上的受隔离进程。

### 学习提示

- 这题最适合作为“容器运行时异常为什么会反过来影响 Java 服务”的起点题。

## 5.2 kube-apiserver、scheduler、controller-manager、etcd 分别做什么？

### 核心讲解

- kube-apiserver 是控制平面的统一入口，负责接收和校验集群对象变更。
- scheduler 负责把 Pod 调度到合适的节点上。
- controller-manager 负责持续把“期望状态”和“实际状态”收敛到一致。
- etcd 负责存储集群的关键状态数据。

### 深入理解

- 学这题的重点，不是背组件名字，而是理解 K8s 为什么能靠声明式方式工作。你提交的是“想要什么”，控制平面再通过调度和控制器不断把实际状态拉回去。
- 这也是为什么高级开发即便不做平台，也最好知道控制平面大概怎么协作。很多“为什么 Pod 没起来、为什么状态不一致”都离不开这条线。

### 追问答案

- 为什么 etcd 这么关键？
  因为它保存的是集群核心状态，没有它，控制平面很难持续收敛对象状态。

### 学习提示

- 记忆时按“入口、调度、收敛、存储”四层去记最稳。

## 5.3 Helm、Kustomize、GitOps 分别解决什么问题？

### 核心讲解

- Helm 更偏模板化和参数化，适合复用和分发一整套应用部署模板。
- Kustomize 更偏在原始 YAML 基础上做分层覆盖和差异管理。
- GitOps 更偏把部署配置版本化、审计化，并让集群状态向 Git 中声明的目标靠拢。

### 深入理解

- 这题要讲的是“为什么生产上很少全靠手写 YAML”。服务一多、环境一多、变更一频繁，模板化和版本化就会直接影响交付质量。
- 对高级 Java 来说，知道这些工具的价值，不是为了取代平台团队，而是为了理解应用交付为什么会从“人手改配置”走向“声明式收敛”。

### 追问答案

- 为什么 GitOps 在生产里很有价值？
  因为它把部署变更纳入版本控制、审计和回滚链路，更容易追踪谁改了什么。

### 学习提示

- 不用把工具细节背太深，但要能讲清“模板化、差异化、版本化”三层价值。

## 5.4 RBAC、ServiceAccount、NetworkPolicy 为什么重要？

### 核心讲解

- RBAC 解决的是“谁能操作集群里的哪些资源”。
- ServiceAccount 解决的是“Pod 以什么身份访问集群或外部能力”。
- NetworkPolicy 解决的是“哪些 Pod 之间网络可达，哪些该被隔离”。
- 它们共同回答的是：平台环境里的权限和网络边界怎么守。

### 深入理解

- 很多团队上了 K8s 后只看调度和发布，却忽略了运行边界。结果应用虽然跑起来了，但权限过大、网络全通、Secret 暴露面很宽，风险反而更高。
- 这题对高级开发的价值在于建立“平台安全边界”意识，而不是要求你成为云安全专家。

### 追问答案

- 为什么 ServiceAccount 不能随便共用？
  因为它会决定 Pod 在集群里的身份和可访问能力，共用过宽容易扩大权限面。

### 学习提示

- 回答时顺手补一句：应用代码安全和平台权限边界要一起看，不能割裂。

## 5.5 StatefulSet、HPA、Sidecar 分别适合什么场景？

### 核心讲解

- StatefulSet 更适合有稳定标识、有序启动或有状态存储需求的场景，比如数据库、消息队列、缓存集群。
- HPA 更适合根据资源或业务指标自动扩缩容无状态服务。
- Sidecar 更适合把日志采集、代理、安全拦截、流量治理等通用能力从主业务容器旁路出去。

### 深入理解

- 这题是很好的收尾题，因为它把“对象选型”拉到了场景层。高级开发不一定自己写 Operator，但至少要知道：不同负载类型在 K8s 上不该用同一种部署思路。
- 尤其是数据库、MQ 这类有状态组件，如果只按 Deployment 思路理解，就很容易低估有序性、稳定标识和存储绑定的重要性。

### 追问答案

- 为什么数据库不适合简单按无状态服务思路部署？
  因为它对稳定标识、数据持久化和有序启动往往有更强要求。

### 学习提示

- 这题按“有状态、弹性、旁路能力”三类场景去记会最顺。
