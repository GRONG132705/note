# 安全与认证面试题库

## 模块导航

- `1. 认证与会话基础`：先建立认证、授权、会话、令牌这些核心概念。
- `2. 权限模型与网关安全`：重点回答 RBAC、接口鉴权、网关、签名、防刷、防重放。
- `3. 常见安全问题与防护`：集中覆盖 SQL 注入、XSS、CSRF、越权、SSRF、接口滥用等高频问题。
- `4. 微服务与开放平台安全设计`：回答单点登录、OAuth2、JWT、双令牌、服务间认证和开放接口保护。
- `5. Spring Security 与实现落地`：把安全设计落到 Spring 生态实现，包括过滤器链、`SecurityContext` 和 JWT 校验接入。
- `6. 数据安全、Secret 管理与排查治理`：覆盖敏感数据保护、密钥管理、安全故障排查和长期治理。

## 一页学习路线

- 第一轮先学 `1`：先分清认证、授权、会话、令牌，不然后面所有问题都会混。
- 第二轮学 `2`：把权限模型、网关鉴权、接口签名、防刷、防重放串成一套边界控制思路。
- 第三轮学 `3`：把常见 Web 安全问题和边界设计问题放到真实系统里理解。
- 第四轮学 `4`：进入微服务和开放平台视角，建立企业级认证与授权框架。
- 第五轮学 `5`：把安全设计落到 Spring Security 实现层，避免只会讲概念不会讲代码入口。
- 第六轮学 `6`：补齐数据安全、Secret 管理、排查与治理，把安全问题收口到生产环境。

## 高频易混点

- 认证解决“你是谁”，授权解决“你能做什么”，两者不是一回事。
- Session、JWT、OAuth2 不是同一层概念，不能直接并列替换。
- 幂等、防重、防重放都是“防重复”，但作用层完全不同。
- 认证成功不等于安全，越权、签名、防刷、权限边界同样关键。
- 网关不是只做转发，它常常还是统一鉴权、限流、签名校验和灰度入口。
- JWT 无状态不等于没有状态问题，它会把注销、续期、踢人这些问题变得更复杂。
- Spring Security 不只是“权限框架”，它更像认证、授权和安全上下文的统一治理入口。
- 敏感字段加密不等于安全已经解决，脱敏、审计、密钥管理和访问控制同样重要。
- Secret 管理不是“把密码放环境变量”这么简单，轮换、权限隔离和审计同样重要。

## 关键对比表

| 对比项 | 核心区别 | 适合回答的重点 |
| --- | --- | --- |
| 认证 vs 授权 | 你是谁 vs 你能做什么 | 入口身份确认和资源访问控制分层 |
| Session vs JWT | 服务端状态 vs 客户端自包含令牌 | 状态控制、吊销能力、扩展性 |
| JWT vs OAuth2 | 令牌格式/载体 vs 授权框架 | 不同层级，不要混答 |
| 幂等 vs 防重放 | 结果只生效一次 vs 防止请求被重复利用 | 业务层 vs 安全层 |
| 网关鉴权 vs 服务内鉴权 | 第一层入口控制 vs 服务自身边界控制 | 第一层防线和最终边界都要守 |
| Access Token vs Refresh Token | 短期访问凭证 vs 刷新凭证 | 续期、失效控制和泄漏风险 |
| 脱敏 vs 加密 | 降低展示暴露 vs 防止明文泄漏 | 面向使用者 vs 面向存储安全 |

## 设计主线

- 先定义主体：
  是用户登录、后台管理、服务间调用，还是开放平台接入。
- 再定义边界：
  认证边界、授权边界、接口边界、网关边界、开放平台边界、数据边界。
- 再选会话方式：
  Session、JWT、双令牌、OAuth2、网关令牌、临时凭证。
- 最后补安全控制：
  权限模型、签名、防重放、防刷、审计日志、Secret 管理、监控告警和复盘治理。

## 最小实验建议

- 用一个最小 Spring Boot 服务分别尝试 Session 登录、JWT 登录和 Spring Security 过滤器鉴权，感受状态管理差异。
- 自己写一个最小签名校验 Demo，模拟时间戳、nonce、签名摘要和重放保护。
- 练习把手机号字段分别做“明文、脱敏展示、加密存储”三种处理，对比它们分别防什么风险。
- 用 Postman 或 Apifox 手动走一遍“Access Token 过期后用 Refresh Token 刷新”的完整流程。

# 1. 认证与会话基础

## 本模块题目

- `1.1` 认证和授权有什么区别？
- `1.2` Session、Cookie、Token 分别怎么理解？
- `1.3` JWT 怎么理解？适合什么场景？
- `1.4` 为什么不能把“登录成功”简单理解成安全问题已经解决？

## 知识地图

- `身份确认 -> 会话建立 -> 令牌承载 -> 权限边界 -> 常见误区`

## 学习目标

- 分清认证、授权、会话、令牌的边界。
- 理解 Session、Token、JWT 分别解决什么问题。
- 建立“安全不是只有登录”这层基本意识。

## 先修知识

- HTTP 基础
- Cookie 基础
- 基础 Web 开发流程

## 本模块总结

- 这一组最重要的，不是背定义，而是分清不同概念所在的层级。
- 很多安全设计错误，都来自“概念混在一起”。
- 如果这组学透，后面的权限、网关、开放平台安全会顺很多。

## 常见误解

- 认为认证和授权只是两种叫法，本质一样。
- 认为 Session、JWT、OAuth2 是同一层面的三选一问题。
- 认为用户只要已经登录，后面就只剩业务功能问题了。

## 典型业务场景

- 用户登录成功后，接口仍然可能被越权访问。
- 前后端分离系统中，团队搞不清 Session、Token、JWT 到底该怎么选。

## 最小实践建议

- 自己画一张图，把认证、授权、Session、Token、JWT 的关系画清楚。
- 练习用一句话分别解释它们解决什么问题。

## 1.1 认证和授权有什么区别？

### 核心讲解

- 认证回答的是“你是谁”，授权回答的是“你能做什么”。
- 认证通常发生在登录或身份校验阶段，授权通常发生在访问某个资源或执行某个操作时。
- 两者经常一起出现，但绝不是同一个东西。

### 深入理解

- 这题虽然基础，但非常关键。很多系统安全问题不是出在“身份没确认”，而是出在“身份确认以后权限边界没管住”。
- 所以学习时一定要形成层次感：认证是入口，授权是访问控制。只有把这一层区分清楚，后面的 RBAC、网关鉴权、越权问题才不会混。

### 追问答案

- 为什么很多系统认证做得还行，越权问题却很多？
  因为身份确认以后，没有把资源访问边界管好。

### 学习提示

- 记一句话：认证看身份，授权看权限。

---

## 1.2 Session、Cookie、Token 分别怎么理解？

### 核心讲解

- Cookie 是浏览器端保存数据的一种机制。
- Session 通常是服务端保存会话状态。
- Token 更像客户端持有的一段凭证，用来证明身份或访问资格。
- 三者经常一起用，但不是同一个层级。

### 深入理解

- 很多初学者会把 Cookie、Session、Token 混成一团，根因就是没分清“存在哪”“状态放哪”“请求怎么带”。
- Session 更偏服务端状态管理，Cookie 更偏浏览器携带和存储，Token 更偏身份凭证。
- 前后端分离和微服务场景里，团队更容易转向 Token/JWT，但并不意味着 Session 过时，只是适用边界不同。

### 追问答案

- Token 一定不需要 Cookie 吗？
  不一定，Token 也可以放在 Cookie 里，只是语义和传统 Session 不同。

### 学习提示

- 学这题时先按“存储位置、状态位置、请求携带方式”三层去记。

---

## 1.3 JWT 怎么理解？适合什么场景？

### 核心讲解

- JWT 可以理解成一种自包含的令牌格式，令牌里通常带有用户标识、过期时间、签发信息等声明。
- 它适合前后端分离、跨服务传递用户身份、对无状态扩展要求更高的场景。
- 但 JWT 不是万能方案，它也有续期、失效控制、体积和泄漏风险问题。

### 深入理解

- 学这题不能只停在“JWT 无状态”这句话上。更关键的是理解：它把部分身份信息带到客户端侧和请求链路里，换来了服务端会话状态减轻，但也带来了吊销困难和泄漏风险。
- 所以 JWT 更适合身份传递和无状态扩展要求高的场景，不代表所有系统都该上。
- 高级岗位面试里，这题常常考的是“你知不知道 JWT 的好处，也知不知道它的代价”。

### 追问答案

- JWT 为什么失效控制更难？
  因为令牌一旦发出去，服务端不像 Session 那样天然持有完整状态。

### 学习提示

- 回答 JWT 时，一定同时说优势和代价。

---

## 1.4 为什么不能把“登录成功”简单理解成安全问题已经解决？

### 核心讲解

- 因为登录成功只说明“身份初步确认过了”，不代表权限边界、接口可信、数据边界和访问链路都安全。
- 很多系统安全问题发生在登录之后，比如越权访问、接口重放、签名缺失、权限配置错误。
- 所以登录成功只是安全链路的起点，不是终点。

### 深入理解

- 这题真正想让你建立的是“安全不是只有登录页”这层意识。高级岗位更关注的是从登录到资源访问这一整条链路，有多少边界已经守住、哪些地方还裸露着。
- 如果把安全简单等同于登录成功，后面会天然忽略权限模型、签名、防刷、防重放、审计日志这些能力。

### 追问答案

- 为什么很多系统“能登录”但还是不安全？
  因为身份确认只解决入口问题，不解决资源边界和接口可信问题。

### 学习提示

- 回答这题时，最好顺手补一句：登录成功以后，安全问题才真正开始分层出现。

# 2. 权限模型与网关安全

## 本模块题目

- `2.1` RBAC 怎么理解？适合什么场景？
- `2.2` 为什么很多系统里权限模型比认证更难设计？
- `2.3` 网关在安全体系里通常承担什么职责？
- `2.4` 接口签名、防重放、防刷分别解决什么问题？
- `2.5` 幂等和防重放有什么区别？

## 知识地图

- `角色与资源 -> 权限边界 -> 网关第一道防线 -> 请求可信 -> 接口滥用防护`

## 学习目标

- 理解 RBAC、网关安全、签名和防重放之间的关系。
- 建立“认证之后还要做权限和接口保护”的完整视角。
- 能分清不同安全手段分别解决什么问题。

## 先修知识

- 认证基础
- HTTP 请求与接口调用
- 微服务入口流量基础

## 本模块总结

- 认证解决身份，权限模型解决资源边界，网关和签名解决入口可信与流量控制。
- 很多安全问题不是单点 bug，而是边界没设计清楚。

## 常见误解

- 认为做了登录就天然有权限控制。
- 认为网关做了鉴权，服务内部就可以默认完全可信。
- 认为幂等和防重放差不多，都是“防重复”而已。

## 典型业务场景

- 后台管理系统里，角色多、菜单多、数据范围多，权限模型越来越难维护。
- 网关只做转发，没有把鉴权、限流和签名校验放到统一入口。

## 最小实践建议

- 试着画一张“用户 -> 网关 -> 服务 -> 资源”的安全链路图。
- 练习区分：这个问题更像权限问题、接口可信问题，还是流量滥用问题。

## 2.1 RBAC 怎么理解？适合什么场景？

### 核心讲解

- RBAC 可以理解成“用户不直接绑定权限，而是先绑定角色，再由角色绑定权限”。
- 它适合后台管理、组织权限、菜单和操作权限比较稳定的场景。
- 它的核心价值是降低权限配置复杂度。

### 深入理解

- RBAC 之所以流行，是因为“用户直接绑权限”在规模变大后几乎不可维护。把权限收在角色上，可以让权限分配更接近组织分工。
- 但 RBAC 也有边界。如果资源粒度很细、动态属性很多、租户边界复杂，单纯 RBAC 往往不够，还会引出数据权限和属性权限问题。

### 追问答案

- RBAC 一定适合所有系统吗？
  不一定，复杂租户、多维数据边界场景下，往往还要补 ABAC 或数据范围控制。

### 学习提示

- 回答时先说“为什么需要角色层”，再说“什么场景它不够用”。

---

## 2.2 为什么很多系统里权限模型比认证更难设计？

### 核心讲解

- 因为认证通常只是确认身份，权限模型却要回答“这个身份能看什么、能操作什么、在什么范围内操作”。
- 资源粒度、数据范围、租户隔离、岗位变化、组织调整，都会让权限问题变复杂。
- 所以权限设计更像业务边界建模，而不是单纯技术接线。

### 深入理解

- 高级岗位看这题，不是想听你说“权限很复杂”，而是想看你能不能意识到权限模型本质上贴着业务组织结构和资源边界走。
- 很多系统权限错配，不是权限框架写得不够多，而是设计时没有先把资源边界、数据边界和角色边界理清楚。

### 追问答案

- 为什么认证做完了，权限问题还会长期折腾？
  因为身份是相对稳定的，资源和边界变化却常常跟着业务持续演进。

### 学习提示

- 这题答得高级一点，要顺手提“数据权限”和“租户边界”。

---

## 2.3 网关在安全体系里通常承担什么职责？

### 核心讲解

- 网关通常承担统一鉴权、令牌校验、限流、防刷、签名校验、灰度入口和审计日志等职责。
- 它是很多系统的安全入口。
- 但网关不是唯一安全层，服务内部仍然要做必要的权限和边界校验。

### 深入理解

- 很多团队把网关理解成“转发器”，这是不够的。对于高级 Java 而言，更成熟的理解是：网关是统一入口控制层。
- 但同时也要有边界意识：不能因为网关做了鉴权，内部服务就彻底不校验。
- 也就是说，网关更像第一道统一防线，不是唯一防线。

### 追问答案

- 为什么网关做了鉴权，服务内部还要校验？
  因为安全不能只押在单点入口上，内部服务仍要守住自己的边界。

### 学习提示

- 回答网关安全时，一定补一句“第一道防线不等于唯一防线”。

---

## 2.4 接口签名、防重放、防刷分别解决什么问题？

### 核心讲解

- 接口签名解决“请求是不是被篡改了、是不是可信调用方发出来的”。
- 防重放解决“同一个合法请求被截获后反复利用”的问题。
- 防刷解决“恶意或异常高频请求持续消耗系统资源”的问题。
- 这三者都在保护接口，但保护层次不同。

### 深入理解

- 高级岗位很看重你能不能把安全问题分层。很多人把签名、防重放、防刷混成一套“接口保护”，但实际上它们在防不同风险。
- 开放平台、支付回调、短信接口、验证码接口这些场景，通常会同时要求三种能力配合，而不是单独只做其中一种。

### 追问答案

- 为什么只做签名还不够？
  因为签名正确的请求被截获后，仍然可能被重复利用。

### 学习提示

- 这题最适合答成“三种风险、三层保护”，不要混。

---

## 2.5 幂等和防重放有什么区别？

### 核心讲解

- 幂等更偏业务语义，回答的是“同一个请求重复执行多次，结果只生效一次”。
- 防重放更偏安全语义，回答的是“请求不能被恶意重复利用”。
- 一个偏结果控制，一个偏请求可信。

### 深入理解

- 很多系统会同时做幂等和防重放，因为它们并不互斥。比如支付回调接口，既要防合法回调被重放，也要保证重复回调不会造成多次扣款。
- 这题特别适合区分高级和普通回答：普通回答会说“都是防重复”，高级回答会主动补“一个偏业务层，一个偏安全层”。

### 追问答案

- 只做幂等能替代防重放吗？
  不能。幂等主要防业务重复生效，不解决请求被非法重复利用的可信性问题。

### 学习提示

- 记一句话：幂等保结果，防重放保请求。

---

# 3. 常见安全问题与防护

## 本模块题目

- `3.1` SQL 注入、XSS、CSRF 分别是什么？怎么防？
- `3.2` 越权问题一般怎么理解？为什么特别危险？
- `3.3` 为什么很多安全问题不是代码 bug，而是边界设计问题？
- `3.4` 文件上传、导出、回调接口这些场景要重点防什么？
- `3.5` 什么是 SSRF？为什么后端服务也要重点防？

## 知识地图

- `经典 Web 漏洞 -> 资源边界 -> 危险场景 -> SSRF 与后端攻击面`

## 学习目标

- 建立常见安全漏洞的基本识别能力。
- 能把“漏洞”和“边界设计”联系起来，而不是只背名词。
- 理解后端高风险场景为什么往往藏在文件、回调、内网访问能力里。

## 先修知识

- HTTP 基础
- 接口开发基础
- 基本的资源访问控制概念

## 本模块总结

- 这一组不是为了做渗透测试，而是为了让你知道哪些点最容易在业务系统里出问题。
- 越权、SSRF、回调接口、防刷，这些对后端高级岗尤其有区分度。

## 常见误解

- 把 SQL 注入、XSS、CSRF 当成“老系统才有”的问题。
- 认为越权只是菜单展示错了，忽略接口和数据边界。
- 认为文件上传、回调、URL 拉取只是普通业务功能，不是安全入口。

## 典型业务场景

- 文件上传接口允许危险文件类型，导致执行风险。
- 服务端根据用户传入 URL 去拉取图片、回调第三方或导出文件，结果打开了内网访问通道。

## 最小实践建议

- 复习时尽量带真实业务场景，比如导出、回调、附件上传、URL 拉取。
- 把“是什么、为什么危险、怎么防”这三层都讲出来。

## 3.1 SQL 注入、XSS、CSRF 分别是什么？怎么防？

### 核心讲解

- SQL 注入是把恶意输入拼进 SQL 里，导致查询被篡改。
- XSS 是恶意脚本被注入页面并在浏览器执行。
- CSRF 是利用用户已登录身份，诱导浏览器发起非预期请求。
- 它们分别作用在数据库层、浏览器渲染层和用户身份链路上。

### 深入理解

- 这题不是要你做安全研究，而是要你能快速分清攻击面在哪里。SQL 注入偏服务端数据访问，XSS 偏前端输出链路，CSRF 偏浏览器身份携带机制。
- 如果你能顺手补出“预编译、输出转义、CSRF Token 或 SameSite Cookie”这些防护思路，这题就比较完整。

### 追问答案

- 为什么它们容易被放在一起问？
  因为都是高频 Web 安全问题，但攻击面和防护方式完全不同。

### 学习提示

- 回答时别只背定义，最好补一句“风险落在哪一层”。

---

## 3.2 越权问题一般怎么理解？为什么特别危险？

### 核心讲解

- 越权通常指用户访问了本不该访问的资源或执行了不该执行的操作。
- 它特别危险，因为攻击者不一定需要绕过登录，只要拿到合法身份就可能越过资源边界。
- 所以越权问题本质上是在考验系统是否真的守住了资源访问边界。

### 深入理解

- 越权最可怕的地方在于，它往往发生在“功能都正常”的系统里。用户是合法登录用户，接口也正常返回，只是返回了不该看到的数据。
- 这意味着它不是传统意义上的“系统挂了”，而是边界悄悄失守了，所以高级岗位会特别看重你有没有这个敏感度。

### 追问答案

- 为什么越权问题不能只靠前端修？
  因为真正的资源边界必须由后端守住。

### 学习提示

- 回答时尽量把“接口权限”和“数据权限”分开说。

---

## 3.3 为什么很多安全问题不是代码 bug，而是边界设计问题？

### 核心讲解

- 因为很多安全问题不是某一行代码写错了，而是系统根本没有把“谁能访问什么、请求是否可信、数据能流到哪里”这些边界设计清楚。
- 所以安全常常是架构问题、边界问题，而不只是 bug 修复问题。

### 深入理解

- 这一题是整份文档里很重要的总观点。很多安全问题不是“多写几个 if”能解决，而是认证边界、授权边界、网关边界、数据边界从一开始就没收清楚。
- 回答这题时，如果能把安全和系统设计联系起来，就会很像高级开发视角。

### 追问答案

- 为什么很多团队修了一个安全漏洞，下次还会再出类似问题？
  因为只修了点，没有补边界和机制。

### 学习提示

- 这题最好和微服务、网关、高并发题库联动着看。

---

## 3.4 文件上传、导出、回调接口这些场景要重点防什么？

### 核心讲解

- 文件上传要重点防危险文件类型、超大文件、恶意脚本和存储滥用。
- 导出接口要重点防越权批量拉取、数据泄漏和大查询打挂系统。
- 回调接口要重点防伪造请求、重放攻击和重复处理。
- 这类场景共同特点是：接口功能看起来正常，但攻击面很大。

### 深入理解

- 这题特别适合体现后端安全意识，因为这些接口都很像“正常业务能力”，但如果边界没收住，后果往往很重。
- 文件上传会打到文件系统和执行边界，导出会打到数据边界和资源边界，回调会打到接口可信和重放边界。

### 追问答案

- 为什么这些场景常被单独问？
  因为它们既贴业务，又特别容易藏高风险入口。

### 学习提示

- 最好按“文件系统、数据泄漏、请求可信”三条线去记。

---

## 3.5 什么是 SSRF？为什么后端服务也要重点防？

### 核心讲解

- SSRF 可以理解成：攻击者诱导服务端替自己去访问某个地址，从而把服务端变成访问跳板。
- 它危险的地方在于：服务端通常能访问内网、元数据地址、内部管理接口，这些位置本来外部用户直接访问不到。
- 所以 SSRF 不是前端漏洞，而是后端服务能力被滥用。

### 深入理解

- 很多后端场景天然容易踩 SSRF，比如图片代理、URL 预览、文件下载、第三方回调校验、通过 URL 拉取数据。
- 这题很适合高级面，因为它既考你对攻击面的理解，也考你有没有后端系统边界意识。真正危险的不是“能发请求”，而是“服务端带着更高权限去发请求”。
- 防 SSRF 时，要优先限制目标地址，而不是只相信调用方传来的 URL。

### 追问答案

- SSRF 为什么在云环境里更危险？
  因为很多内网服务、元数据地址和控制接口只对内可见，服务端一旦被利用就成了跳板。
- 常见防护方式有哪些？
  白名单、协议限制、DNS/IP 校验、禁止访问内网保留地址、出网隔离。

### 学习提示

- 这题最适合和文件下载、回调、图片代理场景一起记。

---

# 4. 微服务与开放平台安全设计

## 本模块题目

- `4.1` 单点登录怎么理解？常见实现思路有哪些？
- `4.2` OAuth2 怎么理解？它解决什么问题？
- `4.3` 为什么 JWT 常配合 Refresh Token？双令牌机制一般怎么设计？
- `4.4` 微服务之间怎么做身份认证和权限传递？
- `4.5` 开放平台接口为什么特别强调签名、时间戳和重放保护？
- `4.6` JWT、Session、OAuth2 在企业系统里一般怎么选？
- `4.7` 服务间认证为什么不能只靠网关？mTLS 解决什么问题？

## 知识地图

- `统一登录 -> 授权委托 -> 双令牌 -> 服务间认证 -> 开放平台接口可信 -> 企业级选型`

## 学习目标

- 能把 SSO、OAuth2、JWT、双令牌、服务间认证放到一个框架下比较。
- 建立开放平台接口保护意识。
- 理解微服务里“网关鉴权”和“服务间认证”不是同一层。

## 先修知识

- 认证与授权基础
- 会话和令牌基础
- 网关与权限模型

## 本模块总结

- SSO 解决统一登录，OAuth2 解决授权委托，JWT 常见于令牌承载，双令牌解决续期和会话控制问题，服务间认证解决内部调用可信。
- 真正复杂的系统，往往是这些方案一起出现，而不是单选题。

## 常见误解

- 把 OAuth2 当成“更高级的登录方式”。
- 认为只要用了 JWT，就天然适合所有前后端分离和微服务场景。
- 认为用户到网关完成鉴权以后，内部服务就不需要再建立可信边界。

## 典型业务场景

- 公司内部多个系统要统一登录。
- 一个开放平台要让第三方在受控范围内访问用户资源。
- 一个前后端分离系统既想无状态扩展，又想控制续期和踢人。

## 最小实践建议

- 自己画一张图，把用户登录、网关、服务间调用、开放平台接入串起来。
- 练习区分“内部统一认证”和“第三方授权接入”这两个场景。

## 4.1 单点登录怎么理解？常见实现思路有哪些？

### 核心讲解

- 单点登录可以理解成：用户在一个认证中心登录后，可以在多个受信任系统之间共享登录状态。
- 它解决的是“多个系统重复登录”的问题。
- 常见实现思路通常会结合统一认证中心、票据/令牌和各业务系统回跳。

### 深入理解

- SSO 的本质不是“方便登录”这么简单，而是统一身份入口和统一会话管理。
- 这题如果答得好，最好能补一句：单点登录也会带来单点风险和统一退出控制难题。

### 追问答案

- 单点登录的最大价值是什么？
  是统一身份入口和减少多系统重复登录。

### 学习提示

- 回答时别只说“一个地方登录处处可用”，要补“统一认证中心”。

---

## 4.2 OAuth2 怎么理解？它解决什么问题？

### 核心讲解

- OAuth2 可以理解成一种授权框架，核心是“资源拥有者授权第三方在受控范围内访问资源”。
- 它解决的是第三方访问资源时，不能直接把用户名密码交给第三方的问题。
- 所以它更偏授权委托，不是简单登录协议。

### 深入理解

- 很多人把 OAuth2 当成“登录方案”，这不够准确。它的核心是授权，不是身份认证本身。
- 在高级 Java 面试里，这题常看你能不能说清“为什么需要授权委托”和“它与 Session、JWT 的层级差异”。

### 追问答案

- OAuth2 和 SSO 是一回事吗？
  不是，SSO 更偏统一登录，OAuth2 更偏授权委托。

### 学习提示

- 记一句话：OAuth2 重点是授权第三方，不是替代所有登录方案。

---

## 4.3 为什么 JWT 常配合 Refresh Token？双令牌机制一般怎么设计？

### 核心讲解

- 因为单纯 JWT 很适合无状态认证，但在续期、注销、踢人和长会话控制上会比较麻烦。
- 所以很多系统会把令牌拆成两层：
  - Access Token：短期使用，负责日常访问
  - Refresh Token：更长期，但只用于换新令牌
- 双令牌的核心价值是：兼顾无状态访问体验和可控的续期能力。

### 深入理解

- 这题是高级面里 JWT 的高频追问。因为真实系统很少只停在“发一个 JWT”这么简单，后面一定会遇到注销、续期、踢人、多端登录这些问题。
- 双令牌方案的重点不是“多一个 token”，而是职责拆分。Access Token 尽量短，泄漏后影响更小；Refresh Token 更谨慎，通常要配更严格的存储、轮换和吊销策略。
- 如果能补一句“Refresh Token 也要防盗用和防滥刷”，这题会更完整。

### 追问答案

- 为什么 Access Token 不直接设置很长有效期？
  因为一旦泄漏，风险窗口会被拉大。
- Refresh Token 被盗了怎么办？
  要结合吊销机制、设备管理、轮换策略和异常检测来收口。

### 学习提示

- 这题最适合按“为什么单 JWT 不够 -> 双令牌怎么分工 -> 又带来什么新风险”这条线去答。

---

## 4.4 微服务之间怎么做身份认证和权限传递？

### 核心讲解

- 常见做法会结合网关统一鉴权、内部令牌、服务间认证、上下文透传和最小权限控制。
- 不能因为是内部服务就默认绝对可信。
- 对高级岗位来说，服务间认证这题的关键是“内部调用链也要有身份和边界”。

### 深入理解

- 很多系统安全设计在用户到网关这一层做得不错，但一进入内部服务就开始默认信任，这其实很危险。
- 微服务内部要回答两个问题：调用方是谁、调用方能拿着什么身份去访问下游资源。
- 回答时尽量带上“网关、服务间认证、上下文透传”三层。

### 追问答案

- 为什么内部服务也要认证？
  因为内部网络不等于绝对可信，服务间同样可能被误用、绕过或伪造。

### 学习提示

- 这题和网关安全一起看，会更容易形成完整边界感。

---

## 4.5 开放平台接口为什么特别强调签名、时间戳和重放保护？

### 核心讲解

- 因为开放平台接口暴露给外部调用方，请求是否可信、是否被篡改、是否被重复利用都会成为高风险问题。
- 签名解决请求可信和防篡改。
- 时间戳和 nonce 常用于缩短请求有效期、防重放。
- 所以这些机制通常要组合使用。

### 深入理解

- 这题本质上是“外部接口比内部接口更怕什么”。开放平台没有内部网络和统一登录那层天然信任，所以必须强化入口可信。
- 回答时最好直接把“签名 + 时效 + 重放保护 + 限频”连起来说，会显得很完整。

### 追问答案

- 为什么只做签名还不够？
  因为签名正确的请求如果被截获后重复使用，仍然可能造成重放风险。

### 学习提示

- 建议把这题和支付回调、回调接口场景对照着记。

---

## 4.6 JWT、Session、OAuth2 在企业系统里一般怎么选？

### 核心讲解

- 更稳的思路是先按场景选：传统单系统和服务端渲染场景，Session 仍然很常见；前后端分离和跨服务身份传递场景，JWT 更常见；涉及第三方授权接入时，OAuth2 更合适。
- 它们不是严格替代关系，而是解决的问题层次不同。
- 所以选型关键是系统边界，不是“哪个更新潮”。

### 深入理解

- 这题高级感很强，因为它考的不是定义，而是选型边界。
- 真正好的回答，通常不是简单说“JWT 更先进”，而是能讲出每种方案在状态管理、扩展性、吊销控制、第三方授权上的取舍。

### 追问答案

- 为什么不能简单说 JWT 比 Session 好？
  因为场景不同，代价和收益也不同。

### 学习提示

- 回答时按“单系统 / 前后端分离 / 第三方接入”三个场景去选最稳。

---

## 4.7 服务间认证为什么不能只靠网关？mTLS 解决什么问题？

### 核心讲解

- 因为网关只守住了外部入口，不能替代服务之间的身份确认和链路可信。
- mTLS 解决的是“通信双方都证明自己身份，并让链路本身加密可信”的问题。
- 所以它更像服务间可信基础设施，不是业务权限模型本身。

### 深入理解

- 这题不需要你展开成 Service Mesh 专题，但高级开发至少要知道：外部用户到网关是一层安全，网关到服务、服务到服务又是另一层安全。
- mTLS 的价值不在于“更复杂”，而在于它让内部服务调用不再只是默认信任，而是具备双向身份校验和链路加密能力。

### 追问答案

- 为什么服务间认证不能只靠网关透传用户身份？
  因为网关之后的内部调用仍可能被伪造、绕过或误用，服务自身仍要识别调用来源。

### 学习提示

- 这题答到“服务间也要分层认证”就够，不用过深展开。

# 5. Spring Security 与实现落地

## 本模块题目

- `5.1` Spring Security 解决什么问题？和认证授权是什么关系？
- `5.2` Spring Security 过滤器链怎么理解？
- `5.3` Spring Security 里 JWT 校验一般怎么接入？
- `5.4` SecurityContext 是什么？为什么重要？

## 知识地图

- `安全框架入口 -> 过滤器链 -> 认证结果写入上下文 -> 权限校验落地`

## 学习目标

- 把安全设计落到 Spring 生态实现层。
- 能讲清 Spring Security 不只是“权限框架”，而是安全链路入口。
- 理解过滤器链、JWT 校验和 `SecurityContext` 的关系。

## 先修知识

- 认证授权基础
- Spring Web 请求处理
- 过滤器链基础

## 本模块总结

- 这一组的重点不是背 API，而是理解 Spring Security 在请求链路里的位置。
- 高级 Java 面试里，这一组非常能区分“只会说安全概念”和“知道代码落在哪里”。

## 常见误解

- 把 Spring Security 理解成“权限注解框架”。
- 认为 JWT 校验写在 Controller 或工具类里手动判断就够了。
- 不清楚 `SecurityContext` 的作用，只知道框架里“能拿到当前用户”。

## 典型业务场景

- 项目里已经用了 JWT，但团队说不清令牌在哪一层被校验、校验后用户信息放在哪。
- 系统里做了角色权限控制，但开发者对 Spring Security 的工作链路理解不清。

## 最小实践建议

- 自己画一张“请求进来 -> 过滤器链 -> 鉴权 -> 写入 SecurityContext -> Controller/Service 权限判断”的流程图。
- 练习用一句话解释“JWT 校验完以后，系统怎么知道当前用户是谁”。

## 5.1 Spring Security 解决什么问题？和认证授权是什么关系？

### 核心讲解

- Spring Security 解决的核心问题是：应用里的身份识别、访问控制和安全拦截如何统一管理。
- 认证回答“你是谁”，授权回答“你能做什么”，Spring Security 则是把这两类能力放到统一安全链路里。
- 它不只是权限注解框架，而是安全过滤、上下文和访问控制的统一入口。

### 深入理解

- 很多人一提 Spring Security 就想到“很重、配置很多”，但它真正的价值是把安全问题从业务代码里抽出来。登录态解析、身份校验、权限判断、请求拦截，本来就不适合散落在 Controller 或 Service 里。
- 这题在高级面里常和网关、安全、微服务一起问。因为单体里你可以只谈一个系统内认证授权，微服务里则还要考虑登录态传递、统一鉴权和边界层校验。

### 追问答案

- Spring Security 只是权限框架吗？
  不是。它更像统一安全治理框架，权限控制只是其中一部分。

### 学习提示

- 这题最适合作为“安全概念”进入“Spring 实现”的桥梁题。

## 5.2 Spring Security 过滤器链怎么理解？

### 核心讲解

- Spring Security 过滤器链可以理解成：请求进入应用后，会先经过一串安全相关过滤器，完成认证、上下文建立、异常处理、权限控制等动作。
- 它的核心价值是把安全逻辑统一挂在请求入口，而不是散落在业务代码里。
- 你不需要背所有过滤器名，但要知道：请求进来后，安全问题先在过滤器链里被处理。

### 深入理解

- 这题的高级感在于：很多安全功能表面看是注解或配置，真正执行入口却在过滤器链。比如 JWT 校验、登录请求处理、异常拦截，最终都要落在过滤器链某个位置。
- 所以学这题要先建立一个心智模型：Controller 执行之前，安全链路已经先跑过一遍。

### 追问答案

- 为什么很多安全逻辑放在过滤器链里而不是 Controller？
  因为安全属于请求入口的统一横切逻辑，放在过滤器层更适合统一治理。

### 学习提示

- 回答时可以先说“过滤器链是安全入口”，这句很稳。

---

## 5.3 Spring Security 里 JWT 校验一般怎么接入？

### 核心讲解

- 常见做法是：在过滤器链中增加一个自定义 JWT 校验过滤器，从请求头拿到 Token，完成解析和校验后，把认证结果写入 `SecurityContext`。
- 这样后面的权限判断、接口访问控制就都能基于当前认证上下文来做。

### 深入理解

- 这题很适合体现“安全设计怎么落代码”。真实系统里，JWT 不是 Controller 手工每个接口自己解析，而是由过滤器链统一处理。
- 关键步骤一般是：
  - 取出 Token
  - 验签与过期判断
  - 解析用户身份和权限信息
  - 写入 `SecurityContext`
  - 放行后续链路
- 这就是为什么说 Spring Security 不是抽象概念，而是有明确运行入口和上下文承载的。

### 追问答案

- JWT 校验完之后，系统怎么知道当前用户是谁？
  因为认证结果已经被写进 `SecurityContext`，后续链路都能拿到当前用户身份。

### 学习提示

- 这题不用一上来就讲复杂源码，先把“过滤器里校验 -> 写入上下文”讲清楚最重要。

---

## 5.4 SecurityContext 是什么？为什么重要？

### 核心讲解

- `SecurityContext` 可以理解成当前请求安全上下文，里面通常放着当前认证主体和相关权限信息。
- 它的重要性在于：后续代码不用重复解析令牌，也不用每层手工传身份，直接从上下文就能拿到当前用户。

### 深入理解

- 这题是 Spring Security 实现层很关键的收口点。过滤器链负责把认证做好，`SecurityContext` 负责把认证结果承接住。
- 如果没有这层上下文，后面的权限注解、资源访问控制、审计记录都会变得更碎片化。

### 追问答案

- 为什么 SecurityContext 很重要？
  因为它把当前认证结果统一挂在请求链路里，让后续权限判断有统一入口。

### 学习提示

- 这题和 `5.2`、`5.3` 一起复习最顺：过滤器链做校验，SecurityContext 承接结果。

- 认为敏感字段只要加密存一下就够了。
- 认为 Secret 管理就是“别把密码写代码里”这么简单。
- 认为安全复盘只是记漏洞原因，不需要沉淀机制和流程。

## 典型业务场景

- 手机号、身份证、银行卡等敏感字段需要安全存储又要满足业务检索。

- 安全事故查到了，但团队不知道后续要补哪些机制。

## 6.1 敏感字段为什么不能只做明文存储？一般怎么做加密和脱敏？

### 核心讲解

- 因为明文存储一旦数据库泄漏、权限错配或内部滥用，敏感数据会被直接读取。
- 常见做法会结合：
  - 存储加密
  - 展示脱敏
  - 严格访问控制
  - 审计记录
- 也就是说，加密和脱敏不是替代关系，而是不同层的保护。

### 深入理解

- 这题高级感很强，因为它已经从“接口安全”走到“数据安全和合规”。
- 真正成熟的回答不是只说“加密一下”，而是知道：
  - 什么时候要可逆加密
  - 什么时候可以脱敏展示
  - 查询和索引怎么处理
  - 内部人员能不能直接通过 SQL 看到原始数据

### 追问答案

- 为什么脱敏不能替代加密？
  因为脱敏主要面向展示层，加密主要面向存储层和泄漏风险控制。

### 学习提示

- 回答时记得补一句：敏感数据安全不仅防外部攻击，也防内部滥用。

---

## 6.2 审计日志为什么重要？一般记录什么？

### 核心讲解

- 审计日志解决的是“出了安全问题后，能不能追到是谁、在什么时间、对什么资源做了什么操作”。
- 它不仅用于排查，也用于合规和内部责任追踪。
- 常见记录内容包括：操作人、时间、资源对象、动作类型、结果状态、来源 IP、请求链路标识等。

### 深入理解

- 审计日志和普通应用日志不一样。普通日志偏排错，审计日志更偏行为追踪和合规留痕。
- 很多高级岗位面试会特别看你有没有“谁改了权限、谁导出了数据、谁批量查了敏感信息”这种意识。

### 追问答案

- 为什么安全问题特别强调审计？
  因为很多风险不是系统挂掉，而是有人做了不该做的事，需要事后追踪。

### 学习提示

- 这题最好和权限错配、敏感数据导出场景一起记。

---

## 6.3 为什么敏感配置不能直接写在 yml 里？生产环境的 Secret 一般怎么管理？

### 核心讲解

- 因为数据库密码、第三方密钥、JWT 签名密钥这类敏感配置，一旦跟代码或配置文件一起泄漏，风险非常大。
- 生产环境里更稳的做法通常会结合：
  - 环境变量
  - 配置中心
  - K8s Secret
  - Vault 类 Secret 管理方案
  - 访问权限隔离与密钥轮换
- 重点不是背方案名，而是理解“Secret 不该和普通配置同一层管理”。

### 深入理解

- 这题体现的是生产环境安全治理意识。很多团队以为把密码挪出代码就够了，但真正成熟的 Secret 管理还包括：
  - 谁能读
  - 谁能改
  - 何时轮换
  - 泄漏后怎么收口
- 所以 Secret 管理本质上不只是“存哪”，而是“权限、生命周期、审计和轮换怎么设计”。

### 追问答案

- 为什么把密码写环境变量也不一定就足够安全？
  因为还要看访问权限、暴露面、审计能力和轮换机制。

### 学习提示

- 回答时别停在“不要写 yml”，要补“生产上一般怎么管 Secret 的全生命周期”。

## 6.4 用户频繁掉登录态，一般先排什么？

### 核心讲解

- 一般先看会话过期策略、令牌有效期、刷新机制、时钟漂移、网关校验和客户端存储是否正常。
- 频繁掉登录态不一定是认证中心故障，也可能是令牌续期、Cookie 域、负载均衡或客户端存储问题。
- 先分清是所有用户一起掉，还是局部场景掉。

### 深入理解

- 这题核心在于“先缩范围”。如果是全部用户一起掉，优先看统一认证、网关和配置；如果只是特定终端掉，更像客户端存储、Cookie 域或前端刷新逻辑。
- 高级面试更看你有没有分层排查意识，而不是一上来只说“看 token 过期时间”。

### 追问答案

- 为什么要先区分是全量还是局部用户？
  因为根因层级完全不同，全量更像服务侧，局部更像终端或环境侧。

### 学习提示

- 记一句话：先分范围，再查会话、令牌、网关和客户端。

## 6.5 接口被恶意刷流量时，一般怎么止血？

### 核心讲解

- 更稳的止血顺序通常是：先限流、封禁异常来源、收紧网关策略、必要时下沉验证码或签名校验，再分析长期防护。
- 第一目标是保核心资源和接口。
- 不要一边被刷一边继续裸奔观察。

### 深入理解

- 这题最重要的是止血优先级。很多安全问题和高并发问题很像，先目标不是找完美根因，而是先保住系统。
- 所以高级岗位回答时，最好能说出短期动作和长期动作的区别：短期限流封禁，长期补签名、防重放、行为识别和告警。

### 追问答案

- 为什么安全场景也要先止血？
  因为恶意请求同样会实时消耗资源和扩大影响面。

### 学习提示

- 建议把这题和 [09-高并发系统设计面试题库.md](E:\学习记录\09-高并发系统设计面试题库.md) 的止血思路一起看。

## 6.6 权限错配导致越权访问时，怎么排查和治理？

### 核心讲解

- 一般先确认越权发生在哪一层：菜单、接口、数据范围还是租户边界。
- 再看是配置错误、权限模型设计问题，还是后端根本没做资源校验。
- 治理时通常要同时补权限模型、接口校验、审计日志和配置流程。

### 深入理解

- 越权问题之所以危险，是因为它常常暴露的是系统边界没守住，而不是某个点坏了。
- 所以这题最好按“发生在哪一层 -> 为什么会发生 -> 怎么堵住 -> 怎么避免再错配”这条线去讲。

### 追问答案

- 为什么越权问题不能只靠前端修？
  因为真正的资源边界必须在后端守住。

### 学习提示

- 回答时尽量把“接口权限”和“数据权限”分开说。

---

## 6.7 安全问题复盘，一般怎么沉淀治理项？

### 核心讲解

- 一般要落到：认证与权限边界补齐、签名或防重放补强、限流和告警规则补齐、审计日志完善、配置流程和测试流程优化。
- 复盘不是只写漏洞原因，而是要把问题沉淀成机制。
- 真正有效的治理，通常会同时补技术和流程。

### 深入理解

- 安全问题最怕“临时修完就算了”。高级岗位更想看到你能把一次问题变成长期能力。
- 所以治理项通常会分为三类：技术边界补强、监控告警补齐、流程和测试前置。

### 追问答案

- 为什么安全问题特别需要流程治理？
  因为很多风险不是代码单点错误，而是配置、流程、审批和测试没兜住。

### 学习提示

- 记一句话：补边界、补告警、补流程。
