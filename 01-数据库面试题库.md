# 索引

- 这一组主要解决“查询为什么快、为什么慢，以及怎么判断慢在哪里”的问题。
- 学的时候建议先理解索引结构，再看索引设计，最后把这些概念和真实查询路径串起来。

## 学习目标

- 理解索引为什么能加速查询，以及为什么有索引也不一定快。
- 能把聚簇索引、二级索引、回表、覆盖索引、联合索引这些概念串起来。
- 能把索引结构、回表、覆盖索引和真实查询路径联系起来理解。

## 先修知识

- B+ 树
- 聚簇索引 / 二级索引
- 回表
- 覆盖索引
- 最左前缀

## 本模块总结

- 索引的核心价值是缩小扫描范围，但前提是索引设计和查询路径真正匹配。
- 索引不是越多越好，错误的索引设计同样会带来写入和维护成本。
- 真正理解索引，不只是记住结构，还要知道它为什么会影响扫描范围、回表和查询成本。

## 最小实践建议

- 用同一类查询分别体验“无索引 / 单列索引 / 联合索引”的差别。
- 重点体会覆盖索引和回表查询的执行路径差异。
- 试着把“最左前缀、回表、覆盖索引、范围查询”这些概念串成一条线。

## 1. MySQL 索引为什么能加速查询？

### 核心讲解

- 索引本质上是帮助数据库缩小扫描范围的数据结构，InnoDB 默认用 B+ 树。它能把全表扫描变成有序查找，减少磁盘 IO 和回表成本。
- 不加索引时只能大范围扫描，数据量一大 IO 和 CPU 开销都会明显上升。
- InnoDB 的索引底层通常是 B+ 树，适合分页读取和范围查询。
- 聚簇索引叶子节点直接存整行数据，二级索引叶子节点通常存主键值，所以有时还会回表。

### 深入理解

- 这题真正要理解的不是“索引能加速查询”这句结论，而是它到底在帮数据库省什么成本。核心省掉的是大范围扫描和大量随机访问。
- 如果没有索引，数据库只能一行一行看过去；有了索引之后，它可以先用索引缩小候选范围，再决定要不要回表拿整行数据。
- 所以索引收益通常和数据量、过滤能力、回表次数直接相关。数据越大、过滤越准、回表越少，索引收益越明显。
- 这也是为什么后面会继续讲联合索引、覆盖索引、回表和最左前缀，它们本质上都在回答“怎么把索引价值榨出来”。

### 追问答案

- 为什么是 B+ 树，不是红黑树？
  因为数据库更关注磁盘 IO，B+ 树多叉、树高更低，更适合页式存储。
- 什么是回表？
  走二级索引查到主键后，再去聚簇索引取整行数据。
- 什么是覆盖索引？
  查询需要的列都在索引里，不需要回主表。

### 学习提示

- 这题适合作为整个“索引模块”的总入口，建议和“回表”“覆盖索引”一起记。
- 学习时不要停在“B+ 树更快”，要继续追问“快在扫描范围、页访问还是回表次数”。

## 2. 聚簇索引和二级索引有什么区别？

### 核心讲解

- 聚簇索引叶子节点直接存整行数据；二级索引叶子节点通常存主键值。前者更适合主键定位，后者更适合按其他字段查询。
- InnoDB 表一定有聚簇索引，通常就是主键索引。
- 二级索引基于普通字段建立，不直接存整行，所以查不全字段时要回表。
- 主键建议尽量短、稳定、递增，因为所有二级索引叶子节点都带主键值。

### 深入理解

- 这题的关键是把“索引结构”和“查询路径”连起来。很多性能问题不是索引有没有建，而是你到底走的是聚簇索引还是二级索引。
- 聚簇索引直接拿到整行，所以主键查询路径最短；二级索引只拿到主键，再回聚簇索引取整行，所以路径更长。
- 这也是为什么主键设计会影响整张表，因为 InnoDB 的二级索引叶子节点都要带主键值，主键越大，所有二级索引都会一起变胖。
- 真正理解这题后，后面的“回表为什么慢”“为什么建议自增主键”就都会更自然。

### 追问答案

- 主键为什么建议自增？
  更接近顺序写，页分裂少，索引维护成本低。
- 没有主键会怎样？
  InnoDB 会选非空唯一索引或生成隐藏列作为聚簇索引。
- UUID 为什么不适合做主键？
  因为长且随机，会导致页分裂更多、索引更大。

### 学习提示

- 学这题时要强行把“二级索引为什么会回表”讲顺，它是索引模块里非常高频的主线。
- 如果你能把“主键设计影响所有二级索引”说出来，这题会明显更完整。

## 3. 联合索引为什么要遵守最左前缀原则？

### 核心讲解

- 联合索引底层按字段组合顺序排序，只有从最左边开始连续使用索引列，数据库才能有效利用这棵索引树。
- `(a,b,c)` 在物理上会先按 a 排，再按 b 排，再按 c 排。
- 所以查询如果直接从 b 或 c 开始，通常没法高效利用索引有序性。
- 最左前缀不是必须三个字段全用，而是从最左边开始连续命中即可。

### 深入理解

- 最左前缀原则本质上不是背规则，而是理解“这棵索引树到底是按什么顺序排的”。
- `(a,b,c)` 这棵树先按 a 分段，再在每一段里按 b 排，再按 c 排，所以如果跳过 a 或中间断层，数据库就很难直接利用这种有序性。
- 很多人把它理解成“必须从左到右全写上”，这也不准确。真正的关键是从左边开始连续命中，能用到多少列要看等值、范围和断层情况。
- 这题学透之后，联合索引顺序设计、索引失效、排序利用索引这些问题都会串起来。

### 追问答案

- 范围查询会影响后面的索引吗？
  通常会，一旦前面用了范围，后面列继续高效利用索引的能力会下降。
- 联合索引字段顺序怎么定？
  看高频过滤条件、区分度、排序需求和覆盖需求。
- `where a = ? and c = ?` 能用到 `(a,b,c)` 吗？
  通常只能稳定用到 a，因为中间缺了 b。

### 学习提示

- 这题别死记口诀，建议自己画一棵 `(a,b,c)` 的简化有序结构，会比背规则更稳。
- 学习时可以顺手和“联合索引字段顺序怎么定”一起看。

## 4. 什么情况下索引会失效？

### 核心讲解

- 常见原因有函数操作、隐式类型转换、前导模糊匹配、计算表达式、不符合最左前缀，以及优化器认为全表扫描更便宜。
- 对索引列做函数或计算，比如 `date(create_time)`，通常会破坏索引利用。
- 隐式类型转换也可能导致索引失效。
- `like '%xx'` 无法利用有序前缀。
- 最终是否走索引要以执行计划为准。

### 深入理解

- “索引失效”这个说法本身有点口语化，更准确地说是“优化器无法高效利用索引”或者“认为走索引不划算”。
- 所以它既可能是写法问题，比如函数、计算、隐式转换，也可能是成本问题，比如扫描范围太大，优化器干脆选全表扫描。
- 真正的工程意识是不要一看到有索引就默认 SQL 会快，而是要用 `explain` 去看它到底有没有命中 `key`、扫描了多少 `rows`、有没有回表或额外排序。
- 这题和“建了索引就一定快是不是误区”本质上是一组题，一个讲失效原因，一个讲性能判断。

### 追问答案

- `like 'abc%'` 为什么能走索引？
  因为保留了左侧前缀。
- 隐式类型转换为什么危险？
  本质上相当于在列上套函数。
- 怎么判断 SQL 有没有走索引？
  看 `explain` 的 `type`、`key`、`rows`、`extra`。
- 怎么验证是不是“写法导致索引失效”？
  最直接的方法是把 SQL 改写前后都跑一遍 `explain`，重点对比 `key` 有没有命中、`type` 是否退化、`rows` 有没有明显放大。

### 学习提示

- 学这题一定要带着执行计划一起看，否则很容易停留在规则背诵层面。
- 最好把“函数、隐式转换、前导模糊、断最左前缀”这几类高频失效场景单独整理成小清单。

## 5. 联合索引字段顺序到底怎么定？

### 核心讲解

- 联合索引顺序不是只看区分度，而是综合看高频查询条件、等值匹配优先级、排序需求、覆盖需求和最左前缀复用价值。
- 先看最常用的过滤条件和谁能明显缩小数据范围。
- 再看等值和范围条件，通常高频等值列更适合靠前。
- 还要考虑 `order by`、`group by` 和覆盖索引收益。

### 深入理解

- 这题最容易答偏成“谁区分度高谁放前面”，但真实设计远不止这一个维度。
- 因为联合索引不是为了某一条 SQL 服务，而是为了让一组高频查询都尽量复用。
- 所以你要同时看过滤、排序、分组、覆盖和最左前缀复用价值。某个字段区分度再高，如果高频 SQL 根本不是按它起手，放最前也未必划算。
- 这题真正想体现的是索引设计思维，而不是机械规则。

### 追问答案

- 区分度最高一定放最前吗？
  不一定，要看真实 SQL 模式和整体复用价值。
- 等值和范围字段谁更靠前？
  通常高频等值列更适合靠前。
- 为什么要考虑排序和覆盖？
  因为好索引不只过滤，还能顺带排序和减少回表。

### 学习提示

- 这题建议和真实 SQL 一起学，单独背结论很容易记混。
- 如果你能说出“区分度只是因素之一，不是唯一标准”，这一题就已经比大多数回答更成熟了。

## 6. 区分度不高的字段该不该建索引？

### 核心讲解

- 单独给区分度很低的字段建索引，很多时候收益有限，但如果它经常和其他字段组合过滤，仍然可能有价值。
- 区分度低意味着重复值很多，单靠它很难显著缩小扫描范围。
- 这种单列索引选择性差，优化器可能更倾向全表扫描。
- 但如果它作为联合索引的一部分，和其他列组合后仍可能有收益。

### 深入理解

- 这题的关键是“低区分度字段不是一票否决”，而是要看它在什么位置、和谁组合。
- 单独建索引时，它很难缩小扫描范围，所以收益有限；但作为联合索引中的一列，它可能仍然有用，比如配合高区分度字段做状态过滤。
- 真正要避免的不是“低区分度字段”，而是“指望它单独承担过滤职责”。
- 所以答题时要体现判断：单列不一定值，但组合场景可能仍然值。

### 追问答案

- 什么叫区分度低？
  不同值很少、重复值很多。
- 低区分度字段什么时候值得建？
  当它经常作为联合条件的一部分，且组合后能显著缩小范围时。
- 怎么验证值不值得建？
  看真实 SQL、执行计划、扫描行数和线上命中率。

### 学习提示

- 学这题时不要把“低区分度字段不能建索引”当成绝对规则。
- 更好的记忆方式是：单列收益弱，联合场景再判断。

## 7. 覆盖索引为什么快？

### 核心讲解

- 覆盖索引快的核心原因是查询需要的列都在索引里，不用再回主表取整行数据。
- 二级索引叶子节点通常只存索引列和主键值，不存整行。
- 如果查询列都包含在索引里，数据库拿到索引记录后就能直接返回结果。
- 相比之下，不覆盖时就得回表，多一次查找路径。

### 深入理解

- 覆盖索引真正省掉的是“回表”这一步，所以它的价值和回表成本强相关。
- 如果只是查几条记录，回表影响可能不大；但一旦扫描很多行，少一次回表就会带来明显收益。
- 这也是为什么很多性能优化最后会落到“能不能把常用查询设计成覆盖索引”。
- 但覆盖索引不是越多越好，因为它通常意味着更胖的联合索引，写入和维护成本也会增加。

### 追问答案

- 覆盖索引是不是越多越好？
  不是，索引越多，占空间越大，写入维护成本越高。
- 为什么覆盖索引对大扫描更有价值？
  因为扫描记录越多，减少回表次数带来的累计收益越明显。
- 覆盖索引和联合索引有什么关系？
  很多覆盖索引本质上就是设计合理的联合索引。

### 学习提示

- 学这题建议和“回表为什么慢”连着看，它们是一正一反两道题。
- 重点不要只记“覆盖索引快”，而是记“快在少一次回表”。

## 8. 回表为什么会慢？

### 核心讲解

- 回表慢的原因是先走二级索引找到主键，再回聚簇索引取整行，等于多了一次查找路径。
- 二级索引叶子节点一般不存整行，只存索引列和主键值。
- 查询列不在索引里时，就要根据主键再次到聚簇索引拿数据。
- 单条回表影响可能不明显，但扫描几千、几万行时，额外成本会迅速放大。

### 深入理解

- 回表本身不是 bug，而是 InnoDB 二级索引结构决定的正常行为。
- 真正的问题在于“回表次数”。少量回表通常可以接受，但一旦扫描行数放大，查找路径叠加就会让成本迅速上升。
- 所以后面很多索引优化，本质上都在做两件事：要么减少扫描范围，要么减少回表次数。
- 这题学透后，你会更容易理解为什么 `select *` 危险、为什么覆盖索引有价值、为什么联合索引设计会影响性能。

### 追问答案

- 主键查询为什么没有回表？
  因为聚簇索引叶子节点本来就存整行数据。
- 回表和 `select *` 有什么关系？
  `select *` 更容易超出索引覆盖范围，导致必须回表。
- 怎么减少回表？
  设计合适的覆盖索引、减少不必要查询列、先筛主键再少量回表。

### 学习提示

- 回表是索引模块里非常关键的性能概念，建议单独记住“路径变长、次数放大、成本累积”这三个关键词。
- 如果你能把回表和覆盖索引一起讲顺，这组题基本就拿下了。

## 9. 什么样的查询适合建联合索引？

### 核心讲解

- 高频、多条件过滤、同时带排序或需要覆盖查询的 SQL，最适合建联合索引。
- 如果长期是固定模式，比如 `where a=? and b=? order by c`，联合索引通常比多个单列索引更有效。
- 联合索引既能利用字段组合顺序做过滤，也可能顺带完成排序和覆盖。
- 查询模式发散、字段组合变化大时，联合索引建太多反而会造成冗余。

### 深入理解

- 联合索引的最佳使用场景，通常是“高频、稳定、可预测”的查询模式。
- 因为它不是通用钥匙，而是针对一组典型 SQL 的定制优化。
- 如果查询条件长期稳定，联合索引可以同时解决过滤、排序、覆盖多个问题；但如果查询模式特别发散，索引很容易建多、建乱，最后反而增加维护成本。
- 所以联合索引最怕的不是字段少，而是场景不稳定。

### 追问答案

- 多个单列索引能替代联合索引吗？
  不完全能，某些场景优化器会做索引合并，但通常不如合理的联合索引稳定。
- 联合索引字段越多越好吗？
  不是，字段越多索引越胖，维护成本越高。
- 什么查询不适合建联合索引？
  命中频率低、字段顺序不稳定、组合变化大的查询。

### 学习提示

- 这题适合和“联合索引字段顺序怎么定”一起复习，一个讲适合什么查询，一个讲具体怎么摆字段。
- 记忆重点是“高频、稳定、多条件、可复用”。

## 10. InnoDB 为什么用 B+ 树，不用红黑树？

### 核心讲解

- InnoDB 用 B+ 树，不是因为它在所有场景里都比红黑树快，而是因为它更适合数据库索引的目标。数据库索引不仅要做单点查找，还要尽量减少页访问、支持范围查询，并和页式存储结构匹配；这几件事正好都是 B+ 树的强项。
- 核心先记一句：数据库索引优化的不是“单次比较次数最少”，而是“整体访问代价更低”。
- 第一，B+ 树是多叉树，一个节点能放很多 key，所以树更矮；红黑树是二叉树，数据量一大树更高，访问路径更长。
- 第二，B+ 树叶子节点有序，通常还会链起来，特别适合 `between`、范围扫描、排序这类数据库高频场景；红黑树更适合单点查找。
- 第三，InnoDB 本来就是按页存储、按页读取的，B+ 树一个节点对应一个页，和数据库页结构天然匹配；红黑树更像纯内存数据结构。
- 所以 InnoDB 选 B+ 树，不是因为红黑树不好，而是因为 B+ 树更符合数据库索引的使用方式。

### 深入理解

- 这题真正要表达的是：数据库索引设计目标和普通内存数据结构的目标不一样。
- 红黑树很擅长在内存里做单点查找和平衡维护，但数据库更关心页访问次数、范围扫描能力和和磁盘/页结构的匹配。
- B+ 树用多叉把树高压低，再把数据集中在叶子节点，既方便做范围扫描，也更符合按页读写的方式。
- 所以不要把这题答成“B+ 树更高级”，而要答成“B+ 树更贴近数据库索引场景”。

### 追问答案

- 为什么 B+ 树更适合范围查询？
  因为数据集中放在有序叶子节点里，找到起点后可以顺着叶子节点一路往后扫，做区间查询和排序都更自然。

- B 树和 B+ 树差别在哪？
  B 树每个节点都可能存数据；B+ 树则把数据主要放在叶子节点，非叶子节点更像导航层。这样 B+ 树非叶子节点能放下更多 key，更适合做索引。

- 如果全在内存里，红黑树是不是就更好？
  纯内存、以单点查找和频繁增删为主时，红黑树确实可能更合适，因为它实现直接、维护成本低。
  但数据库即使很多数据在内存里，也还是更偏向 B+ 树，因为数据库还要做范围查询，而且索引本来就是按页组织的。
  所以可以简单记成：纯内存集合，红黑树可能更优；数据库索引，B+ 树通常更优。

### 学习提示

- 这题在面试里很高频，建议固定答法：树高、范围查询、页结构匹配。
- 如果你能补一句“数据库优化的是整体访问代价，不是单次比较次数”，会很加分。

## 11. 为什么不建议 `select *`？

### 核心讲解

- 因为它容易多查无用列、增加网络和内存开销，还会破坏覆盖索引，让原本能快的查询变慢。
- 很多场景根本不需要所有列，`select *` 会把无用数据也查出来。
- 它更容易导致查询列超出索引覆盖范围，从而回表。
- 表结构变更后，返回结果也会变，可能影响兼容性。

### 深入理解

- 这题本质上不是代码风格问题，而是查询路径和接口契约问题。
- 一旦写成 `select *`，你就默认接受了“多拿列、难覆盖、可能回表、结果集随表结构变化”这些副作用。
- 在宽表、高频查询、链路长的场景里，这种副作用会被明显放大。
- 所以面试里最好不要把它答成“习惯不好”，而要答成“它会破坏覆盖索引、增加传输和兼容成本”。

### 追问答案

- `select *` 什么时候危害更大？
  宽表、大字段、高频查询、原本可以走覆盖索引的场景。
- 为什么它会影响覆盖索引？
  因为一旦查询列超出索引包含范围，就不得不回表。
- 小表场景也不能用吗？
  不是绝对不能，但工程习惯上仍更建议按需取列。

### 学习提示

- 学这题时，把它和“覆盖索引”“回表”一起理解，会比单独背规范更有用。
- 如果只能记一句，就记“`select *` 最容易把本来能覆盖的查询打回回表查询`”。

## 12. 为什么不建议在索引列上做函数操作？

### 核心讲解

- 因为对索引列做函数、计算或类型转换，往往会破坏原有索引的有序性，导致索引失效。
- 索引是基于原始列值建立的有序结构。
- 在列上套函数后，优化器通常没法直接利用原索引定位。
- 隐式类型转换也有类似问题，本质上也是对列值做处理。

### 深入理解

- 这题的关键不是“函数不能用”，而是“你一旦改变列值，原始索引顺序就不好直接用了”。
- 比如时间列本来是按原值排好的，你再套一层 `date()`，数据库没法直接按原索引结构去定位目标区间。
- 隐式类型转换也是同样的逻辑，本质上是让数据库先处理列值，再比较。
- 所以更稳的写法通常是把条件改写成范围，而不是把函数套在索引列上。

### 追问答案

- 隐式类型转换为什么也危险？
  因为本质上和在列上套函数类似，也可能破坏索引利用。
- 时间字段查询怎么改写更好？
  改成时间范围，比如 `>=` 和 `<` 的组合。
- 一定完全不能做函数吗？
  不是语法不能，而是高频核心 SQL 不建议这么写。

### 学习提示

- 这题建议直接和“索引失效”一起记，属于最经典的高频追问点。
- 学习时重点练“函数写法”和“范围写法”的改写能力。

## 13. 什么情况下宁可不建索引？

### 核心讲解

- 索引不是越多越好。如果字段区分度很低、命中频率不高、写入压力很大，或者索引已经明显冗余时，宁可不建，也不要为了“看起来更全”把写入和维护成本堆上去。
- 索引的收益来自“减少扫描范围”，但它也有代价：占空间、拖慢写入、增加维护成本、让优化器选择更复杂。
- 所以如果某个字段很少参与过滤、排序、关联，或者区分度特别低，单独建索引的收益往往有限。
- 另外，写多读少的表也要特别谨慎，因为每多一个索引，插入、更新、删除时都要额外维护一份结构。
- 还有一种常见情况是索引冗余。比如已经有合适的联合索引，再补一个价值不大的单列索引，可能收益不明显，代价却是真实存在的。
- 所以“该不该建索引”不是看字段重不重要，而是看它能不能稳定带来查询收益，且这个收益值不值得支付额外的写入和维护成本。
- 面试里可以这样答：字段高频命中、能显著缩小扫描范围时，索引才更值得；如果命中少、选择性差、写压力大或已有索引能覆盖，就可以考虑宁可不建。

### 深入理解

- 这题真正想建立的是“索引是有成本的”这个意识。
- 很多初学者只记得索引能加速读，却忽略了它会占空间、拖慢写、放大维护复杂度。
- 所以该不该建，不取决于字段看起来重不重要，而取决于它能不能稳定带来查询收益。
- 真正成熟的判断方式，是把读收益和写成本一起算，而不是默认“多建总比少建好”。

### 追问答案

- 什么字段最容易出现“建了也收益不大”？
  区分度很低、过滤能力弱、很少参与查询条件的字段。
- 为什么写多的表要更谨慎？
  因为每个索引都会放大插入、更新、删除时的维护成本。
- 怎么判断一个索引是不是冗余？
  看它是否被真实 SQL 稳定命中，以及已有联合索引是否已经覆盖了它的大部分价值。
- 不建索引是不是就一定不优化？
  不是。有时更合理的优化路径是改 SQL、改查询路径、做缓存，或者重新设计表结构。

### 学习提示

- 这题很适合和“建了索引就一定快是不是误区”连着复习，两题是一组。
- 如果你能主动补一句“索引收益和写成本要一起看”，这题就会很完整。

## 14. 建了索引就一定会快，这是不是个误区？

### 核心讲解

- 这是典型误区。索引能不能带来收益，要看查询模式、选择性、回表成本、排序分组需求和写入代价。建了索引不等于一定更快，建错了反而会更慢。
- 很多人把“加索引”当成慢 SQL 的万能答案，但数据库性能从来不是只看有没有索引。
- 如果索引选择性差、扫描行数依然很大，或者 SQL 最终要大量回表、排序、分组，那即使走了索引，收益也可能很有限。
- 另外，写多读少的表上索引越多，插入、更新、删除维护成本就越高，整体吞吐反而可能下降。
- 还有一种常见误区是“字段上建了索引，所有相关 SQL 都会自动变快”。实际上联合索引顺序不匹配、函数操作、隐式类型转换，都会让索引价值大打折扣。
- 面试里可以这样答：索引不是装上就生效的外挂，真正决定性能的是索引是否和查询路径匹配，以及它带来的读收益值不值得写成本。

### 深入理解

- 这题是整个索引模块的收口题，本质上在提醒你：索引是成本和收益的平衡，不是万能外挂。
- 只要查询路径不匹配、选择性不够、回表太多、排序分组没利用上索引，哪怕 `key` 命中了，也未必真的快。
- 相反，如果读收益不大、写压力很高，索引甚至可能让系统整体更慢。
- 所以判断索引有没有价值，永远不能只看“有没有建”，而要看“有没有被正确利用、扫描行数有没有变小、整体吞吐有没有变好”。

### 追问答案

- 为什么有索引还是可能慢？
  因为可能选择性差、回表多、排序分组没利用上索引，或者 SQL 写法让索引失效。
- 索引会带来什么副作用？
  占空间、拖慢写入、增加维护成本，还可能让优化器选择更复杂。
- 什么情况下建索引收益往往不大？
  低区分度、低命中率、写多读少、或者已有索引已基本覆盖时。

### 学习提示

- 这题适合作为索引模块的最后一题来记，它能把前面的“最左前缀、回表、覆盖、失效、顺序设计”全串起来。
- 如果面试里时间不多，这题可以作为你对索引部分的收口表达。
- 索引优化最容易忽略什么？
  只盯“有没有索引”，却不看执行计划、扫描行数和真实查询路径。

# SQL 优化

- 这一组主要讨论高频 SQL 场景为什么会慢，以及该怎么排查和优化。
- 不要把 SQL 优化简单理解成“加索引”，执行计划、分页、排序、分组、join 和写法本身都可能成为瓶颈。

## 学习目标

- 理解慢 SQL 不只是“有没有索引”的问题，而是扫描、回表、排序、聚合、join 和锁等待共同作用的结果。
- 能分清哪些问题优先靠改 SQL 解决，哪些问题已经上升到表结构或数据模型层面。
- 能建立一条比较稳定的排查路径，而不是一慢就盲目补索引，并能用执行计划辅助判断问题。

## 先修知识

- `explain`
- 扫描行数
- 回表
- `Using filesort` / `Using temporary`
- 深分页
- join 基础

## 本模块总结

- SQL 优化的核心不是“写得花”，而是让数据库少扫、少排、少回表、少做无意义中间结果。
- 慢 SQL 的根因很多时候不是单点问题，而是索引、写法、分页方式和数据模型一起叠加出来的。
- 真正稳定的优化通常建立在“先定位，再看执行计划，再改路径”这条链路上。
- 更稳的学习顺序通常是：先定位，再看执行计划，再拆成扫描 / 回表 / 排序 / 聚合 / join，最后决定改 SQL、改索引还是改模型。

## 最小实践建议

- 先用 `explain` 学会看一条 SQL 大致是怎么执行的。
- 再选几条典型 SQL，对比改写前后的执行计划差异。
- 尝试区分“SQL 本身慢”和“SQL 在等锁”这两类完全不同的问题。

## 1. 慢 SQL 应该怎么排查和优化？

### 核心讲解

- 排查顺序一般是 `先定位 SQL -> 区分 SQL 本身慢还是锁等待/资源问题 -> 如果是 SQL 本身慢，再看执行计划 -> 最后决定改 SQL、补索引还是调数据模型`。

- 先通过慢查询日志、APM 或业务日志定位具体 SQL。

- 第二步先判断慢在哪里，是 SQL 本身扫描慢、排序慢，还是其实耗在锁等待、资源抖动、下游依赖。

- 如果判断问题在 SQL 本身，再去看 `explain`，重点关注扫描行数多不多、有没有回表、排序、临时表和全表扫描。

- 最后再决定下一步是改 SQL、补索引，还是已经上升到表结构和数据模型问题。

- 如果已经确认问题在 SQL 本身，下一步最核心的工具通常就是 `EXPLAIN`。

### 深入理解

- 这题真正要理解的，不只是结论本身，而是“1. 慢 SQL 应该怎么排查和优化？”背后的原理、适用边界和工程取舍。
- 学习时建议把它和本模块前后题串起来看：它解决什么问题、在什么场景下最有价值、又会带来什么代价或限制。
- 如果这题偏原理，就重点理解数据流转、执行路径或状态变化；如果这题偏设计，就重点理解业务约束、方案选择和失败场景。
- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- 排查时第一步最重要的是什么？
  先把“到底是哪条 SQL 慢、慢在什么时间段、影响哪个接口”定位清楚，不要一上来就猜索引问题。
- 慢 SQL 和锁等待怎么区分？
  SQL 本身慢通常表现为扫描行数大、执行计划差；锁等待则是语句本身没问题，但时间耗在等待锁。
- 什么时候要怀疑不是 SQL 本身的问题？
  如果执行计划并不差，但线程还是长时间卡住，就要优先怀疑锁等待、资源抖动、下游依赖或数据库当时的整体负载。
- 排查时怎么快速判断“问题到底是不是这条 SQL 本身”？
  可以把慢日志、`explain`、执行耗时和会话状态一起看。如果执行计划并不差，但线程长时间卡住，更要怀疑锁等待、资源抖动或下游依赖，而不是只盯 SQL 写法。
- 什么时候该优先改 SQL，什么时候该考虑改模型？
  如果只是少量 SQL 写法有问题，优先改 SQL；如果很多高频查询都在围绕同一个结构性问题打补丁，就该考虑表结构或数据模型。
- 慢 SQL 优化最容易犯什么错误？
  不先定位耗时来源，就直接补索引或改写 SQL，最后容易把真正的问题方向带偏。

### 学习提示

- 先把这题的核心结论讲顺，再尝试用一个真实业务场景去解释它。
- 最好和相邻题一起复习，形成一条完整主线，而不是孤立背诵单个知识点。

## 2. EXPLAIN / 执行计划怎么看？

### 核心讲解

- `EXPLAIN` 这一题重点不是讲慢 SQL 的排查流程，而是讲“拿到执行计划后具体怎么看”。通常按 `id -> possible_keys -> key -> type -> rows -> filtered -> extra` 这个顺序看，会更容易抓到 SQL 的主要代价在哪里。

- 如果第 1 题解决的是“先排查什么”，那这一题解决的就是“执行计划到手后怎么读”。一般可以按这个顺序看：

- 第一步看 `id`，它主要用来判断多表查询、子查询的执行层次和大致顺序。

- 第二步看 `possible_keys`，确认优化器认 为哪些索引理论上可以选。

- 第三步看 `key`，确认最终到底用了哪个索引；`possible_keys` 里有，不代表最终一定会用。

- 第四步看 `type`，判断扫描方式好不好。通常从好到差大致可以理解成 `const`、`ref`、`range`、`index`、`all`，越靠后扫描范围往往越大。

- 第五步看 `rows`，它表示优化器预估要扫描多少行。这个值不是绝对精确，但很适合判断 SQL 是不是在做大范围扫描。

- 第六步看 `filtered`，它表示扫描出来的数据里，大概有多少比例能继续通过条件过滤。`rows` 和 `filtered` 一起看，更容易判断过滤效率。

- 第七步看 `extra`，这里经常藏着很多关键信号，比如 `Using where`、`Using index`、`Using temporary`、`Using filesort`。

- 其中几个高频点要重点理解：

- `Using index` 通常说明走了覆盖索引，不需要回表。

- `Using filesort` 不一定真的是“用文件排序”，但它说明排序没完全利用索引，通常要关注排序代价。

- `Using temporary` 往往表示中间结果需要临时表，`group by`、`order by`、复杂 join 里比较常见。

- 如果 `key` 看起来用了索引，但 `rows` 依然很大，也不代表这个 SQL 就一定快，因为可能索引选择性差、回表很多，或者后续排序聚合代价高。

- 所以面试里可以这样答：`EXPLAIN` 不是只看有没有索引，而是要结合 `id + possible_keys + key + type + rows + filtered + extra` 一起判断，重点看执行层次、候选索引、扫描范围、过滤效果、是否回表、是否临时表、是否额外排序。

### 深入理解

- 这题真正要理解的，不只是结论本身，而是“2. EXPLAIN / 执行计划怎么看？”背后的原理、适用边界和工程取舍。
- 学习时建议把它和本模块前后题串起来看：它解决什么问题、在什么场景下最有价值、又会带来什么代价或限制。
- 如果这题偏原理，就重点理解数据流转、执行路径或状态变化；如果这题偏设计，就重点理解业务约束、方案选择和失败场景。
- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- `id` 主要看什么？
  主要看多表查询、子查询的执行层次和顺序，通常能帮助判断谁先执行、谁后执行。
- `possible_keys` 和 `key` 有什么区别？
  `possible_keys` 是优化器认为理论上可选的索引，`key` 是最终实际选用的索引。
- `type = all` 一定很差吗？
  通常说明全表扫描，要重点关注；但如果表本来就很小，或者优化器判断全表扫更便宜，也不一定就是问题。
- `rows` 很大就一定慢吗？
  不绝对，但它通常意味着扫描范围大，是很重要的风险信号。
- `extra` 里最值得优先关注哪些信号？
  常见最值得先看的是 `Using index`、`Using filesort`、`Using temporary`。它们通常分别对应覆盖索引、排序代价和临时表代价。
- `Using filesort` 一定很糟吗？
  不一定，但它意味着排序没完全走索引。数据量小时问题不大，数据量大时就要重点优化。
- `Using index` 为什么通常是好事？

## 3. 深分页为什么慢？怎么优化？

### 核心讲解

- 深分页慢的核心原因是 `limit offset, size` 在 offset 很大时，数据库通常要先扫描并丢弃前面大量记录。

- 比如 `limit 100000, 20`，很多人会误以为数据库能直接“跳到第 100001 行开始取”。但实际执行时，数据库通常还是要先找到并跳过前面那 100000 条记录，再拿后面的 20 条，所以前面的扫描和丢弃成本非常真实。

- 如果这条 SQL 还带 `order by`，问题会更重。因为数据库可能不仅要扫描很多记录，还要先把它们排好序，再把前面大量结果丢掉。特别是排序又没法利用索引时，排序成本和深分页成本会叠加。

- 所以深分页的优化通常分两类：

- 第一类是改分页方式。比如用游标翻页、基于上次位置翻页，这类方法的核心是不再依赖大 offset 去“跳页”。

- 第二类是减少扫描和回表成本。比如先走覆盖索引查主键，再根据主键少量回表，这样至少可以让前面那一大段“跳过的数据”扫描得更轻。

## 5. `order by` 为什么会慢？怎么优化？

### 核心讲解

- `order by` 慢的核心不是语法本身慢，而是数据库在不能利用索引顺序时，需要额外做排序。数据量一大，CPU、内存、临时文件这些成本就都会上来。

- 先抓一个核心：数据库最喜欢的情况，是按索引顺序一路扫出来，边扫边返回结果。这样就不需要额外排序。

- 真正让 `order by` 变慢的，是排序没法直接利用索引顺序。这时数据库就可能要把满足条件的数据先拿出来，再额外做排序。数据量一大，排序的 CPU、内存，甚至临时文件成本都会明显上升。

## 9. `union` 和 `union all` 的区别及性能影响是什么？

### 核心讲解

- `union all` 本质上是直接拼接结果，通常更快；`union` 在拼接后还要去重，所以会多一层中间结果处理。数据量一大，真正慢的通常不是关键字本身，而是去重代价。

- 先抓一个核心：`union all` 只是把多个结果集直接拼起来返回，而 `union` 不只要拼接，还要保证最终结果不重复。

- 也就是说，`union` 比 `union all` 多做了一步去重。数据量小时差异可能不明显，但数据量一大，去重带来的中间结果处理、排序或哈希代价就会逐渐放大。

- 所以这题不要只记“`union` 更慢”，而是要理解它为什么更慢：慢的不是语法名字，而是“拼接之后还要再处理一遍结果”。

- 工程上更稳的选择原则通常是：如果业务明确允许重复，优先考虑 `union all`；如果业务语义要求最终结果唯一，那就必须接受 `union` 的额外成本。

- 这题可以总结成一句话：`union all` 解决的是结果拼接，`union` 解决的是拼接后还要去重；什么时候该用哪个，先看业务语义，再看性能代价。

## 10. 什么时候该改 SQL，什么时候该改表结构？

### 核心讲解

- 先判断问题是在“查询路径”还是在“数据模型”。如果主要是写法、索引利用和扫描范围不合理，优先改 SQL；如果很多查询都在被同一个结构性问题反复拖慢，就该考虑改表结构。

- 改 SQL 通常是第一选择，因为成本更低、验证更快、回滚也更容易。像条件改写、减少无效列、避免深分页、减少回表、让排序分组更容易利用索引，这些都属于先看 SQL 能不能解决的问题。

- 这一类问题的共同特点是：数据模型本身没有明显错，只是当前 SQL 没把现有结构用好。比如本来有合适索引，但写法导致没法命中；本来可以先过滤再 join，却把结果集做大了；本来只是分页方式不对，却误以为必须改表。

- 但如果你发现很多 SQL 都在重复绕同一个坑，比如为了一个高频查询不得不频繁 join、某些字段总要额外冗余到查询里、冷热数据长期混在一起、多个统计场景都只能硬扫大表，那问题往往已经不只是写法，而是表结构和模型没有服务主要查询路径。

- 这时候继续微调 SQL，收益往往越来越有限，因为你只是在现有模型上打补丁。更合适的做法通常是改表结构，比如补更合适的索引、调整字段设计、增加冗余字段、拆表、做预聚合，甚至重新梳理主键和查询路径。

- 一个实用判断标准是：如果问题主要集中在少量 SQL，而且改完后路径就顺了，优先改 SQL；如果同类问题在多个高频查询里反复出现，说明该从表结构层面解决系统性问题。

- 面试里可以把它总结成一句话：SQL 优化优先解决局部执行路径问题，表结构优化用来解决反复出现的模型问题；不要一上来就大改模型，也不要长期靠 SQL 打补丁掩盖结构缺陷。

- 为什么通常先改 SQL？
  因为成本更低、风险更小，也更容易通过执行计划和耗时直接验证收益。

## 11. 慢 SQL 都是索引问题，这是不是个误区？

### 核心讲解

- 这是非常常见的误区。索引当然重要，但它只是慢 SQL 的高频原因，不是总开关。慢 SQL 慢的可能是扫描、回表、排序、分组、join、锁等待，甚至是数据模型和访问方式本身。

- 现实里的慢 SQL，确实经常和索引有关，但“慢 SQL = 没索引”这个判断太粗了。索引解决的是缩小扫描范围、帮助有序查找，它不是所有性能问题的总开关。

- 第一类常见问题是扫描和回表问题，比如没命中索引、命中低选择性索引、扫描范围太大、回表过多。这一类问题和索引关系最直接，但也不是只要“有索引”就一定好。

- 第二类常见问题是执行过程问题，比如排序、分组、深分页、join 后结果集膨胀。这里真正重的往往不是“找数据”本身，而是后面的中间结果处理和额外代价。

- 第三类问题甚至不是 SQL 路径本身慢，而是时间主要耗在锁等待、长事务、资源竞争上。应用看到的是 SQL 变慢，但慢点可能根本不在索引。

- 所以更稳的排查顺序应该是：先定位具体 SQL 和耗时来源，再看执行计划、扫描范围、回表、排序、临时表、join 路径，以及是否存在锁等待和资源瓶颈，而不是一上来就补索引。

- 面试里可以总结成一句话：索引问题只是慢 SQL 的高频原因，不是唯一原因；真正的优化前提是先判断慢在扫描、回表、排序、分组、join，还是慢在锁和资源竞争。

### 追问答案

- 什么场景即使有索引也可能慢？
  深分页、复杂排序分组、大量回表、低选择性扫描、复杂 join，这些场景即使走了索引，整体代价也可能还是很高。

## 1. MySQL 事务的四大特性和隔离级别怎么答？

### 核心讲解

- 事务四大特性是 ACID。隔离级别解决的是“并发事务彼此能看到什么”，从弱到强常见有读未提交、读已提交、可重复读、串行化，InnoDB 默认是可重复读。

- 先看 ACID。原子性强调要么都成功要么都失败；一致性强调事务前后数据状态合法；隔离性强调并发事务之间影响受控；持久性强调提交后的结果不会因为宕机丢失。

- 隔离级别解决的是并发事务读取彼此数据时“能看到什么”。读未提交最弱，可能读到别人未提交的数据；读已提交解决脏读，但前后两次读取同一行，结果仍可能不同；可重复读进一步让事务内快照读结果更稳定；串行化最严格，但并发能力也最低。

- InnoDB 默认选可重复读，不是因为它完美，而是因为它在一致性和性能之间比较折中。后面很多题，比如 MVCC、幻读、当前读为什么要加锁，其实都是在补这个隔离级别背后的实现逻辑。

### 深入理解

- 这题真正要理解的，不只是结论本身，而是“1. MySQL 事务的四大特性和隔离级别怎么答？”背后的原理、适用边界和工程取舍。

- 什么是脏读、不可重复读、幻读？
  脏读是读到未提交数据；不可重复读是同一行前后读结果不同；幻读是范围查询前后条数不同。

## 2. MVCC 是什么？解决了什么问题？

### 核心讲解

- MVCC 是多版本并发控制，核心是让快照读读到“对当前事务可见的历史版本”，从而减少很多读写冲突。

- 问题先摆清楚：如果读写并发时所有读都去抢最新版本、都去加锁，系统并发会很差。MVCC 的意义，就是让很多普通查询不用和写操作硬碰硬。

- InnoDB 会为一行数据维护多个版本。旧版本依赖 undo log 串成版本链，查询时再结合 Read View 判断当前事务到底能看到哪个版本。

- 这样在读已提交和可重复读下，很多普通 `select` 都可以直接读历史可见版本，不必为了“读一致”去主动加锁。

- 但它的边界也很重要：MVCC 主要服务快照读，不服务当前读。像 `select ... for update`、`update`、`delete` 这类要读最新数据并控制并发修改的场景，仍然要依赖锁。

- 这题真正要理解的，不只是结论本身，而是“2. MVCC 是什么？解决了什么问题？”背后的原理、适用边界和工程取舍。

- 快照读和当前读有什么区别？
  快照读读历史可见版本；当前读读最新版本并可能加锁。
- undo log 在这里起什么作用？
  保存旧版本数据，支撑回滚和版本链读取。
- MVCC 能解决幻读吗？
  快照读下能大幅弱化幻读影响，但当前读还需要间隙锁、临键锁。
- MVCC 是如何解决 RC 和 RR 的读写冲突？

- 最容易误解成什么？
  误以为 MVCC 能解决所有并发问题。它解决的是很多快照读并发问题，不等于替代当前读、锁控制和事务设计。

## 3. MySQL 里的行锁、表锁、间隙锁、临键锁怎么理解？

### 核心讲解

- 行锁锁具体记录，表锁锁整张表；间隙锁锁的是记录之间的区间；临键锁可以理解成“记录锁 + 间隙锁”的组合，核心目标是控制范围并发插入。

- 先抓一个原则：InnoDB 想尽量把锁做细，所以最常见的是行级锁。但这里说的“行锁”并不是抽象地锁住某一行，而是通常依赖索引去定位具体记录或区间。

- 表锁比较好理解，就是整张表层面的控制；行锁是更细粒度的控制；间隙锁锁的不是已有记录，而是记录之间的区间；临键锁则可以理解成既锁住命中的记录，也顺带把相邻区间保护起来。

- 为什么要有后面这两种锁？因为当前读下，如果你只锁住已存在的记录，不锁区间，别人仍然可能在这个范围里插入新数据，范围结果就会变化。

- 所以这一题不要只背定义，更重要的是抓住一句话：很多锁问题最后都落到“锁的是单点，还是锁的是区间”。

## 4. 为什么会出现幻读？MySQL 是怎么处理的？

## 5. 可重复读下，普通 `select` 和 `select ... for update` 为什么表现不同？

## 6. 读已提交和可重复读怎么选？

### 核心讲解

- 核心看业务在“读取最新结果”和“事务内结果稳定”之间怎么取舍。RC 更接近最新已提交结果，并发更友好；RR 更强调事务内快照读稳定。

- 先看区别。RC 每次快照读都会生成新的 Read View，所以更容易读到别人最新提交的结果；RR 第一次快照读生成 Read View 后，整个事务复用，所以同一事务多次读取更稳定。

- 如果业务更在意“我这次读到的尽量是最新提交结果”，RC 会更自然；如果业务更在意“我这个事务里前后看到的结果别变来变去”，RR 更合适。

- 但选择不是只看概念，还要看代价。RR 一致性更强，排查锁范围和当前读问题时也更复杂；RC 虽然更贴近最新结果，但事务里前后读到不同结果会更多。

- 所以核心交易链路、订单和资金类场景通常更偏 RR；后台管理、报表、对“每次读到最新结果”更敏感的场景，有时会考虑 RC。

## 7. 大事务为什么危险？

### 核心讲解

- 大事务危险的核心不是“执行久一点”，而是它会把锁持有时间、undo/redo 压力、主从延迟和回滚成本一起放大。

- 大事务本质上是一次事务里改了很多数据，或者做了很多操作。问题不只是“它自己慢”，而是它会把影响范围一起放大。

- 第一层风险是锁。事务越大，持锁时间通常越长，别的事务更容易排队，热点表和热点行冲突会很明显。

- 第二层风险是日志和版本压力。大事务会生成更多 undo、redo 和 binlog，占用更多内存、磁盘和复制资源。

- 第三层风险是失败代价。事务越大，回滚越痛苦，因为撤销本身也要花时间，甚至会比提交时更难受。

- 所以大事务的问题，本质上是“单次操作太大，把锁、日志、复制、回滚的代价一起放大了”。

## 8. 长事务会带来哪些问题？

## 9. 什么 SQL 最容易引发死锁？

- 最容易引发死锁的，通常不是“语法特殊”的 SQL，而是多个事务以不同顺序拿同一批锁，或者范围更新、批量更新、缺索引更新把锁范围放大。

## 11. 什么是意向锁？为什么需要它？

### 核心讲解

- 意向锁可以理解成一种表级“标记”。它不是业务侧直接感知到的行锁，而是用来告诉数据库：这张表里某些行已经准备加锁或正在被行锁控制，这样表锁和行锁之间的冲突判断会更高效。

- 这题最容易抽象化，其实可以把它想简单一点：如果每次有人想加表锁，都要逐行检查这张表里有没有行锁，成本会非常高。

- 所以 InnoDB 设计了一层协调标记。事务在对某些行加锁前，会先在表层放一个对应的意向锁，告诉数据库“这张表里已经有行级加锁意图了”。

- 这样别的事务如果想直接对整张表加表锁，不用逐行检查，只要先看表上的意向锁状态，就能快速判断是否冲突。

- 所以意向锁的价值不在于它直接控制业务并发，而在于它让“整表锁”和“部分行已经加锁”之间的协调更高效。

## 12. 唯一索引等值查询和普通索引范围查询，加锁行为有什么区别？

### 核心讲解

- 唯一索引等值查询通常锁得更精确，更多是记录锁；普通索引范围查询更容易把锁扩到区间，出现间隙锁或临键锁。

- 用唯一索引做等值查询时，数据库通常能精确命中一条记录，所以很多场景下只需要锁住该记录本身，锁范围相对更小。

## 13. 什么情况下锁范围会扩大？

## 14. 线上锁等待怎么排查？

### 核心讲解

- 线上锁等待排查，核心不是只看谁慢，而是先找出“谁在等”“在等谁”“为什么一直不释放”。常见根因包括长事务、热点行更新、索引不命中、批量操作和大事务。

- 先区分是死锁还是普通锁等待。死锁通常会被数据库主动检测并回滚其中一方；普通锁等待则可能一直卡到超时。

- 排查时第一步通常是看当前会话状态，比如 `show processlist;`，先找出哪些线程处于等待状态，SQL 在执行什么。

- 第二步要找阻塞源头。很多时候真正的问题不是“被卡住的 SQL”，而是前面有一个事务已经拿了锁，但迟迟没有提交。

- 第三步看事务情况，重点关注长事务、未提交事务、批量更新、批量删除，以及是否存在同一行或同一范围上的热点竞争。

- 第四步看索引是否合理。如果更新或删除条件没有命中合适索引，数据库可能扫描和锁住比预期更大的范围，锁冲突就会明显放大。

- 第五步结合业务场景判断，是不是高峰期批处理、定时任务、DDL、库存扣减、账户更新这类典型热点操作引起的。

- 常用抓手通常包括：

- `show processlist;` 看当前等待线程和正在执行的 SQL。

- `information_schema.innodb_trx` 看当前活跃事务、开始时间和是否有长事务。

- 锁等待相关视图，看谁持有锁、谁在等待锁。

- `show engine innodb status`，适合快速看最近的死锁、锁等待和 InnoDB 内部状态。

## 15. 乐观锁和悲观锁怎么选？

### 核心讲解

- 冲突少、吞吐优先、能接受失败重试时，更适合乐观锁；冲突高、必须强控制并发修改顺序时，更适合悲观锁。核心不是哪种更高级，而是业务冲突概率和失败代价。

- 乐观锁的思路是“先假设冲突少”，更新时再校验版本号或条件，发现别人先改过就失败重试。

- 悲观锁的思路是“先假设冲突会发生”，在读取或修改前就先把记录锁住，减少并发修改碰撞。

- 所以如果业务本来冲突概率不高，而且失败后容易重试，比如资料更新、后台配置修改，乐观锁通常更轻量，对吞吐更友好。

- 但如果是库存扣减、账户余额、抢单这类高冲突场景，单纯靠乐观锁可能会带来大量失败重试，这时悲观锁通常更稳，因为它用更早的加锁换更确定的修改顺序。

- 本质上，乐观锁是在用失败重试换并发，悲观锁是在用提前加锁换确定性。选哪种，不是看谁高级，而是看冲突概率和失败代价谁更敏感。

## 1. redo log、undo log、binlog 分别解决什么问题？

### 核心讲解

- redo log 主要解决“事务提交了，但数据页还没刷盘时，宕机后怎么恢复”；undo log 主要解决“事务执行到一半怎么回滚，以及快照读怎么看历史版本”；binlog 主要解决“数据库到底做过哪些逻辑变更，需要怎么复制、归档和恢复”。

- 可以把它们理解成分别在回答三类完全不同的问题：

- 第一类问题是：机器宕机了，已经提交的事务怎么尽量不丢？这主要由 redo log 解决。

- 第二类问题是：事务执行到一半失败了，或者主动 `rollback` 了，怎么把已经改过的数据撤回去？这主要由 undo log 解决。

- 第三类问题是：数据库到底发生过哪些逻辑变更，这些变更怎么复制给从库、怎么做归档和时间点恢复？这主要由 binlog 解决。

- redo log 是 InnoDB 引擎层的物理日志，更偏“数据页修改已经发生了什么”。它的价值在于：事务提交时不必立刻把真实数据页刷盘，只要先把 redo log 持久化，宕机后就能靠它把已提交事务的修改重做出来，这就是 crash recovery。

- undo log 也是 InnoDB 里的日志，但它更偏“修改前的旧值是什么”。所以它一方面支撑事务回滚，另一方面也支撑 MVCC，因为快照读要顺着版本链往回找到当前事务能看到的旧版本。

- binlog 是 MySQL Server 层的逻辑日志，记录的是“发生了什么变更”。它不负责页级重做，所以不是单机崩溃恢复的核心工具；但它特别适合拿去做主从复制、归档、增量同步和时间点恢复。

- 所以这三者不是一个东西拆成三份，而是事务可靠性、回滚能力、复制归档能力分工协作。真正该记住的是：redo 重点是崩溃恢复，undo 重点是回滚和 MVCC，binlog 重点是复制和归档。

- 这题真正要理解的，不只是结论本身，而是“1. redo log、undo log、binlog 分别解决什么问题？”背后的原理、适用边界和工程取舍。

- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- redo log 为什么能提高写性能？
  因为它把随机更新数据页，先转成顺序写日志，事务提交时先落 redo log 就行，不必每次都同步刷整个数据页。
- undo log 为什么和 MVCC 有关？
  因为一行记录更新后，旧版本并不会立刻消失，快照读要沿着 undo log 形成的版本链去找当前事务能看到的那个历史版本。
- binlog 为什么主从复制要用它？
  因为它记录的是逻辑层面的变更事件，从库拿到这些变更后可以重新执行或回放，所以它适合跨实例同步。
- redo log 能不能替代 binlog？
  不能。redo log 是引擎层的物理日志，更偏向页修改和崩溃恢复；binlog 是 Server 层逻辑日志，才适合复制和归档。
- undo log 会一直保留吗？
  不会。等没有事务再需要读取这些旧版本时，undo log 才能被清理；如果长事务很多，undo log 就会堆积。
- 为什么说这三种日志不是一个东西拆三份？
  因为它们解决的问题不同：redo 负责崩溃后把已提交修改重做出来，undo 负责回滚和历史版本读取，binlog 负责逻辑变更历史、复制和归档。

## 2. 两阶段提交为什么需要？

### 核心讲解

- 这里说的两阶段提交，不是分布式事务里的那个 2PC，而是 MySQL 在一次本地事务提交时，用 `prepare -> 写 binlog -> commit` 这条链路协调 redo log 和 binlog，避免一个成功、另一个失败，导致主库数据和 binlog 历史不一致。

- 先把问题摆清楚：一次事务提交时，MySQL 不只是改内存里的数据，还要同时处理两套关键日志。

- redo log 属于 InnoDB，引擎层日志，主要负责崩溃恢复；binlog 属于 MySQL Server 层逻辑日志，主要负责复制和归档。

- 问题在于，这两套日志不是同一个组件写的。如果事务提交过程中宕机，就可能出现“本地已经按 redo 认为事务成功了，但 binlog 还没记下来”，或者“binlog 记下来了，但 redo 还没真正确认”的情况。

- 所以 MySQL 才需要这套两阶段提交链路。可以简单记成三步：

- 第一步：先写 redo log，并标记为 `prepare`。

- 第二步：再写 binlog，确保逻辑变更历史也被记下来。

- 第三步：最后把 redo log 标记成 `commit`，事务正式完成。

- `prepare` 的意思可以理解成：这个事务的物理修改我已经准备好了，但还不能立刻算最终提交，要等 binlog 也确认成功，才能真正完成。

- 如果没有这套协调机制，就容易出现两类典型问题：本地数据恢复出来了，但 binlog 没记下来；或者 binlog 记下来了，但主库本地恢复后却没有这次事务。前者会让从库同步不到，后者会让主库和复制历史对不上。

- 所以两阶段提交解决的，本质上是 redo log 和 binlog 的一致性问题。它不是在解决所有宕机风险，而是在保证“主库本地最终保留下来的事务”和“对外可复制的事务历史”尽量一致。

- 这题真正要理解的，不只是结论本身，而是“2. 两阶段提交为什么需要？”背后的原理、适用边界和工程取舍。

### 追问答案

- 不做两阶段提交会怎样？
  可能出现 redo log 和 binlog 只有一个成功，最终导致主库本地数据和 binlog 内容不一致，进一步引发主从不一致。
- `prepare` 状态是什么意思？
  表示 redo log 已经写好了，但事务还不能直接算最终提交，还要等 binlog 也成功后，才能进入真正的 `commit`。

  不能，它解决的是 redo log 和 binlog 的协调一致性问题，不等于解决所有宕机、硬件故障、复制延迟带来的风险。

  因为从库是靠 binlog 同步的。如果主库本地已经提交，但 binlog 没记下来，从库就拿不到这次事务，复制链路就会出现偏差。

  它解决的是 redo log 和 binlog 之间的协调一致性问题，不解决备份缺失、硬件损坏、复制延迟和所有数据丢失风险。

## 3. 什么时候该依赖 binlog 恢复，什么时候不能只靠它？

### 核心讲解

- binlog 更适合做归档恢复、时间点恢复和主从同步；但单机崩溃恢复不能只靠它，因为它解决的不是页级持久化和 crash recovery 问题。

- 这题本质上不是继续讲 binlog 定义，而是在判断：你遇到的是哪一种恢复场景。

- 如果你要做的是主从复制、归档恢复、时间点恢复，binlog 就很关键。因为它记录的是逻辑变更历史，可以把某个时间段内发生过的变更重新回放出来。

- 最典型的用法就是全量备份加 binlog。比如先恢复一份全量备份，再把后续某段时间的 binlog 回放进去，就能把数据推到误操作前的某个时间点，这就是 PITR。

- 所以 binlog 更像“变更历史和回放素材”，redo log 更像“单机崩溃后的本地恢复保障”。很多恢复场景并不是二选一，而是全量备份、binlog、redo log 各自负责不同层面的工作。

## 4. 有了 binlog、redo log、undo log 就万无一失了，这是不是个误区？

- redo log、undo log、binlog 各有职责，这让 MySQL 在事务回滚、崩溃恢复、复制归档上都更可靠。

## 1. MySQL 主从复制和读写分离有哪些问题？

### 核心讲解

- 主从复制和读写分离能分担读压力，但天然会带来主从延迟、一致性窗口和路由复杂度问题。真正难的不是“能不能分流”，而是关键读请求怎么兜底。

- 主库负责写，从库负责读，主库把 binlog 传给从库，从库重放日志完成同步。这样最大的收益是把读流量从主库分出去，也为高可用打基础。

- 但问题也很直接：主库提交成功，不等于从库立刻同步完成。于是“写后立刻读”就可能读到旧数据，这就是最典型的一致性窗口问题。

- 所以读写分离真正难的地方，不是把读流量打到从库，而是要判断哪些读请求能接受延迟，哪些读请求必须兜底回主库。

- 工程上通常会结合主读兜底、短时间会话主读、读己之写、关键链路强制主读等策略处理一致性问题。

- 所以这一题可以总结成一句话：主从复制和读写分离解决的是读扩展，但代价是天然引入一致性窗口和路由复杂度。

## 2. 主从复制为什么会延迟？

### 核心讲解

- 因为主库提交成功不代表从库已经同步完成。复制链路至少要经过“主库产生日志 -> 网络传输 -> 从库回放”三段，任何一段跟不上，延迟就会出现。

- 主从复制本质上不是“主库一提交，从库立刻就完成”，中间至少要经历三个阶段：主库产生日志、网络传输日志、从库回放日志。

- 只要这三段里任何一段跟不上，延迟就会出现。

- 主库侧如果有大事务、批量更新、DDL 或突发写流量，binlog 生成和发送压力就会上升。

- 网络侧如果链路抖动、带宽不足、跨机房延迟高，binlog 传输就会变慢。

- 从库侧通常是最容易积压的地方，因为日志拉下来之后还要真正执行。执行过程中如果遇到大事务、慢 SQL、索引缺失、磁盘压力、CPU 紧张，重放速度就会下降。

- 所以面试里更好的表达是：主从延迟不是某一个点的问题，而是复制链路上“生成、传输、回放”三个阶段里，任何一段成为瓶颈都会造成延迟。

## 3. 主从延迟怎么排查？

### 核心讲解

- 排查主从延迟，核心不是只盯一个延迟数字，而是先确认它卡在主库产生日志、网络传输，还是从库回放。只有先定位链路位置，后面排查才不会跑偏。

- 排查主从延迟，第一步不是直接猜原因，而是先定位延迟卡在哪一段：是主库写 binlog 慢、主从传输慢，还是从库 SQL 回放慢。

- 先看复制状态。常用命令是 `show replica status\G`，老版本也可能用 `show slave status\G`。

- 重点看几个字段：

- `Seconds_Behind_Master`，先判断延迟量级，但它只能作为参考，不能单靠这一个字段下结论。

- `Replica_IO_Running` 和 `Replica_SQL_Running`，确认 IO 线程和 SQL 线程是不是正常。

- `Relay_Log_Space`、`Read_Master_Log_Pos`、`Exec_Master_Log_Pos`，看是“日志没拉下来”，还是“日志已经拉下来了但执行不过来”。

- 如果是多线程复制，还可以看 worker 线程有没有堆积。

- 如果发现 IO 线程异常或者主库位点增长慢，要看主库侧：

- 有没有大事务、批量更新、长事务、DDL。

- 可以结合慢日志、业务发布记录、监控平台上的 TPS/QPS、binlog 生成速度来判断。

- 如果现场能查数据库，常见抓手包括 `show processlist;`、慢查询日志、`information_schema.innodb_trx`，看看有没有特别大的事务一直没结束。

- 如果日志传输有问题，要看网络层：

- 主从之间网络抖动、带宽不足、跨机房链路波动，都会让 binlog 传输变慢。

- 这时要结合机器监控、网络监控、复制报错信息一起看，而不是只看数据库内部状态。

- 如果日志已经到从库，但执行慢，那重点就在从库回放能力：

- 看从库 CPU、IO、磁盘延迟、buffer pool 命中率是不是有压力。

- 看 SQL 线程是不是被大事务、索引缺失、锁冲突、DDL 卡住。

- 可以用 `show processlist;`、慢日志、`performance_schema`、系统监控工具如 `top`、`iostat`、`vmstat` 一起确认。

- 所以更完整的面试回答可以是：先用 `show replica status\G` 判断复制卡在“拉日志”还是“执行日志”，再结合主库事务情况、网络状态、从库资源和慢 SQL，定位到底是哪一段导致延迟。

## 4. 半同步复制解决了什么问题？

### 核心讲解

- 半同步复制解决的是异步复制里“主库刚提交就挂了、从库一个都没收到日志”的风险，它本质上是在缩小数据丢失窗口，不是在消灭延迟。

- 异步复制下，主库提交后直接返回，不等从库确认，所以主库刚提交就故障时，最近那部分 binlog 可能还没到任何从库。

- 半同步复制做的事很简单：要求至少有一个从库确认收到 binlog，主库才向客户端返回成功。

- 这样做的价值是缩小主库故障时的数据丢失窗口，因为至少有一个从库已经拿到这次事务对应的日志。

- 但它不等于强一致。收到日志不等于已经执行完成，更不等于所有从库都同步完成，所以后续仍然可能有回放延迟和可见性延迟。

- 所以半同步复制本质上是在“提交延迟”和“丢失窗口”之间做权衡，用更高的写入路径成本换更小的数据丢失风险。

## 5. 读写分离下怎么避免读到旧数据？

### 核心讲解

- 读写分离最大的问题不是“能不能分流”，而是主库刚写完、从库还没同步时，读请求可能读到旧数据。解决思路通常不是消灭延迟，而是按一致性要求分级，对关键读请求做主读兜底。

- 读写分离下，写请求通常先到主库，读请求打到从库。但从库复制有延迟，所以主库刚提交，不代表从库立刻可见。

- 最典型的问题就是“读己之写”失败：用户刚下单、刚改密码、刚提交资料，紧接着读接口却还看不到最新结果。

- 所以工程上通常不会把所有读都盲目打到从库，而是按业务一致性要求做分级。

- 常见做法有几种：

- 第一，关键请求直接回主库。比如写后立即读、账户余额、支付结果、订单状态这类强一致场景。

- 第二，设置一个短时间窗口，写操作完成后，在一段时间内让这个用户或这个请求链路继续读主库。

- 第三，延迟路由或读己之写策略。比如把最近写过的用户打上标记，后续一小段时间内强制主读。

- 第四，对一致性要求没那么高的场景继续读从库，比如资讯、报表、非关键详情页。

- 更本质地说，读写分离不是“所有读都去从库”，而是“把不同一致性要求的读请求路由到不同地方”。

- 面试里可以总结成一句话：读写分离下避免读到旧数据，核心不是消灭延迟，而是识别强一致读场景，并通过主读、延迟路由、读己之写等策略兜住关键请求。

## 6. 什么情况下适合做读写分离，什么情况下不适合？

- 所以它更适合读多写少、查询量大、从库能稳定承担读流量的系统，比如内容平台、订单查询、后台报表、商品详情等。

## 7. 主从切换时要注意什么？

### 核心讲解

- 主从切换不是把一个从库提成主库就结束了，真正要管的是复制位点、一致性窗口、流量路由、连接切换和脑裂防护。

- 第一步是确认候选从库的数据足够新，不能把明显落后的节点直接提成主库，否则切换成功了，数据也可能已经回退。

- 第二步是处理流量。新主起来后，应用连接、读写路由、连接池、代理层都要跟着切，不然业务流量可能还在打向旧主或错误节点。

- 第三步是防脑裂。旧主如果误恢复后继续对外写入，就会出现多个节点都认为自己是主库的情况。

- 所以主从切换从来不是单一步骤，而是一整套链路：选最新节点、切写流量、挡旧主、校验业务连通性、重建复制关系。

- 这也是为什么“数据库支持主从复制”不等于“系统已经具备稳定高可用”，因为故障检测、切换时机、路由切换和脑裂防护都还要额外设计。

- 什么是脑裂？
  多个节点都认为自己是主库并对外提供写服务。
- 为什么复制位点很重要？
  因为它决定候选主库的数据新旧程度。
- 切换后还要做什么？
  校验业务连通性、观察延迟和错误率、重建复制关系。
- 主流关系型数据库是否支持主从切换，是否支持自动切换？

- MySQL 是怎么解决主从切换和自动切换这类问题的？

  误以为“选一个最新从库提主”就完成了切换，忽略了旧主封禁、路由切换和业务连通性校验这些真正容易出事故的环节。

## 8. 有了主从复制就等于高可用了，这是不是个误区？

### 核心讲解

- 是误区。主从复制只是高可用体系的一部分，不等于自动切换、流量切换、脑裂防护、读写一致性这些问题都已经解决。

- 主从复制解决的是数据从主库同步到从库，为读扩展和故障切换提供基础。

- 但真正的高可用不只是“有从库”，还包括故障检测、主从切换、流量路由、连接恢复、复制位点判断、脑裂防护、切换后的验证。

- 面试里可以这样答：主从复制是高可用的基础设施，不是高可用的全部；没有切换机制和配套治理，只能算“能同步”，不一定算“真正可用”。

## 9. 什么时候该上半同步，什么时候异步就够了？

### 核心讲解

- 如果业务对主库故障后的数据丢失窗口比较敏感，可以考虑半同步；如果更在意写入延迟和系统简单性，异步往往就够了。核心取舍是“少丢数据”还是“更低延迟”。

- 异步复制的好处是简单、写入路径短、延迟更低，但主库刚提交就挂掉时，可能存在从库还没收到日志的窗口。

- 半同步复制会要求至少有一个从库确认收到日志后，主库才返回成功，所以它能缩小这个数据丢失窗口。

- 但代价也很明确：主库提交路径更长，对网络和从库状态更敏感，写入延迟会上升。

- 所以如果是订单、支付、核心交易这类不能轻易接受主库故障后丢最近几笔数据的业务，半同步通常更值得考虑。

- 如果是对极低写延迟更敏感，或者业务本身能接受少量复制窗口数据风险，异步往往就够了。

- 面试里可以总结成一句话：半同步是在写延迟和数据安全窗口之间做交换，值不值得上，取决于业务是否真的在意那段复制窗口。

  不是，它只是缩小数据丢失窗口，不等于强一致，也不等于零丢失。

## 1. 什么情况下该分库分表，什么情况下不该分？

### 核心讲解

- 分库分表不是性能优化的默认答案，而是单库单表在容量、并发、维护窗口上真的扛不住时的架构手段。如果问题还能通过索引、SQL、缓存、冷热分离、读写分离解决，就不一定该急着分。

- 分库分表真正解决的是单库单表的数据量、写入压力、维护成本、扩容空间这些天花板问题。

- 所以当单表过大、索引维护成本高、批量任务困难、单机资源接近上限，或者单库已经成为明显瓶颈时，分库分表才有现实意义。

- 但它的代价非常大：跨库 join 变难、分页排序复杂、分布式事务麻烦、全局 ID 要重设计、数据迁移和扩容成本也上来。

- 所以如果问题本质只是慢 SQL、索引没建好、冷热数据没分开、缓存没用好、读流量还可以靠从库分担，那先分库分表往往是过早设计。

- 一个更稳的思路通常是：先把单库做到位，再看是不是真的碰到了单库极限，而不是一遇到性能问题就把分库分表当银弹。

- 面试里可以这样答：该不该分库分表，不是看“听起来高级不高级”，而是看单库是否真的达到瓶颈，以及你是否准备好接受它带来的复杂度。

## 2. 分库分表后事务和主键怎么处理？

### 核心讲解

- 分库分表后，本地事务和单库自增主键都不再天然成立。事务要考虑跨库一致性方案，主键要考虑全局唯一、趋势有序和路由友好性。

- 一旦一次业务操作涉及多个分片，本地事务就只能保证单库内原子性，跨库这一层要靠更高一级的一致性方案去兜底。

- 如果业务真的要求强一致，就可能要考虑 TCC、SAGA、消息最终一致性等方案；无论选哪一种，系统复杂度都会明显上升。

- 另一个马上会冒出来的问题是主键。单库里的自增主键只能保证库内唯一，分库后就不够用了，通常要换成全局 ID。

- 这个全局 ID 不能只看唯一性，还要看生成性能、是否大致有序、是否方便路由，以及对数据库写入是否友好。

- 所以分库分表后最先变难的，往往不是“SQL 怎么写”，而是事务怎么兜底、主键怎么统一、路由怎么落下去。

## 3. 雪花算法有什么问题？

### 核心讲解

- 雪花算法的优点是本地生成、性能高、基本趋势递增，适合分布式场景；但它并不完美，主要问题集中在时钟回拨、机器号管理、不保证严格全局递增，以及单机每毫秒发号上限这些方面。

- 雪花算法一般会把 ID 拆成时间戳、机器号、序列号几部分，所以它很依赖系统时间和机器标识的正确性。

- 最大风险通常是时钟回拨。如果机器时间被 NTP 校正或人为调小，当前时间戳比上一次发号时还小，就可能导致重复 ID，或者系统为了避免重复而暂停发号。

- 它依赖机器号唯一。如果两台机器错误地配置成同一个机器号，那么在相同时间片内就有可能生成重复 ID。

- 雪花算法通常只是“趋势递增”，不是数据库自增主键那种严格连续、严格单调。多台机器并发发号时，生成结果会交错。

- 单机同一毫秒内能生成的 ID 数量也有限，超过序列号容量后，就得等待下一毫秒。

- 从数据库角度看，它比 UUID 更适合做主键，因为通常更短、也更接近递增；但和单库自增主键相比，对 InnoDB 的写入友好度通常还是差一点。

- 所以雪花算法不是不能用，而是你要先接受它的前提：依赖时间、依赖机器号、依赖运维治理。

## 4. 号段模式和雪花算法怎么选？

### 核心讲解

- 如果更看重本地发号、高性能、弱依赖中心服务，通常选雪花算法；如果更看重稳定、可控、对时钟不敏感，通常选号段模式。两者都常用于分布式 ID，没有绝对谁更好，关键看系统诉求。

- 雪花算法的特点是本地生成，机器拿到机器号后就能自己发号，性能很高，也不需要每次都请求中心服务。

- 但雪花算法强依赖时间戳和机器号管理，时钟回拨、机器号冲突，都是它的典型风险。

- 号段模式的思路是先从中心服务或数据库批量申请一段连续 ID，再在本地内存里慢慢消费，所以它对时钟不敏感，也不用关心机器时间是否回退。

- 号段模式通常更容易做运维和排障，因为 ID 分配规则更集中、可控，出了问题也更容易定位。

- 但它也不是没有代价。它依赖中心发号服务或号段表，如果设计不好，中心服务可能成为瓶颈或单点。

- 从数据库主键角度看，两者通常都比 UUID 更适合。雪花算法一般是“趋势递增”，号段模式往往更接近连续递增；如果非常在意顺序写友好性，号段模式通常会更占优势。

## 1. 为什么建议用自增主键？

### 核心讲解

- 在单库 InnoDB 这个默认场景下，自增主键通常更友好，因为聚簇索引按主键顺序组织数据，自增更接近顺序写、页分裂少，而且主键越短，所有二级索引也越轻。到了分库分表或业务需要全局 ID 的场景，这个结论就不能直接照搬。

- 自增主键通常是在索引末尾追加，写入更稳定。

- 随机主键会更容易插到索引中间，触发页分裂和碎片。

- 二级索引叶子节点里会带主键值，所以主键越长，所有二级索引都会变胖。

### 深入理解

- 这题真正要理解的，不只是结论本身，而是“1. 为什么建议用自增主键？”背后的原理、适用边界和工程取舍。

- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- UUID 做主键为什么不友好？
  因为它通常又长又随机，会导致页分裂更多、二级索引更大。

- 主键短有什么额外好处？
  二级索引占用空间和维护成本更低。

- 一定只能用自增主键吗？
  不是，但要意识到其他主键方案对索引结构的影响。

- 分库分表后，还适合用数据库自增主键吗？
  通常就没那么合适了。单库里自增主键很友好，因为它能让 InnoDB 更接近顺序写；但分库分表后，数据库自增往往只能保证“单库单表内递增且唯一”，很难天然保证全局唯一。
  
  主要问题有这几个：
  
  1. 每个分片各自自增，只能保证本分片不冲突，不能保证全局不重复。
  2. 如果想靠“不同步长 + 不同起始值”解决，比如 2 台机器一个发奇数一个发偶数，那在扩容、缩容时规则会很麻烦。
  3. 数据迁移、分片重组、历史数据回灌时，也容易碰到主键规则不一致的问题。
  4. 从系统设计角度看，分布式场景通常更需要的是“全局唯一、趋势递增、独立发号”，而不是强依赖某个库表自己的自增能力。
  
  所以面试里可以这样答：单库单表下，自增主键通常是 InnoDB 的优选；但分库分表后，更常见的是用分布式 ID，比如雪花算法、号段模式或业务统一发号。这样做的核心，不只是为了解决唯一性，也是为了后续扩容和路由更稳定。

## 2. 页分裂是什么？会带来什么问题？

- 如果新数据插入到已满页的中间位置，就可能触发页分裂。

## 3. 设计数据库表时要注意什么？

### 核心讲解

- 建表设计不能只看“字段够不够用”，还要同时考虑业务语义、查询模式、数据约束、扩展性和后续维护成本。很多线上性能和变更问题，其实在建表阶段就已经埋下了。

- 第一，要先把主键想清楚。主键不仅决定唯一性，还会影响聚簇索引组织方式、二级索引大小和后续写入性能，所以通常要优先选稳定、短、适合索引的主键方案。

- 第二，字段类型要尽量准确、短小，不要图省事全部用 `varchar`。类型选得太宽，会浪费存储，也会让索引更胖，查询和维护成本都上去。

- 第三，要提前想好哪些字段允许 `null`，哪些应该有默认值，哪些必须加唯一约束。约束不是可有可无的，它本质上是在数据库层帮你兜住数据质量。

- 第四，大字段要特别小心。像长文本、JSON、富文本、详细描述这类字段，不要轻易和高频核心字段混在一起做热点查询表，也不要随便参与索引；很多场景下更适合拆到附表，或者至少避免在高频查询里直接带出来。

- 第五，表结构要结合主要查询路径设计，而不是先拍脑袋建表，后面再被动补索引。比如经常按什么条件查、按什么字段排序、哪些字段会 join，这些都会影响字段设计和索引设计。

- 第六，金额、状态、时间这类高频字段也要设计清楚。金额不要随便用 `float`、`double`；状态字段要语义稳定、枚举收敛；时间字段通常至少要有创建时间和更新时间，必要时还要补业务时间、删除时间。

- 第七，要考虑后续扩展和变更成本。像创建时间、更新时间、状态字段、软删除标记、租户标识这类字段，很多业务表其实一开始就应该考虑，不然以后再改大表，成本会更高。

- 第八，范式和冗余要结合业务权衡。不是越规范越好，也不是越冗余越方便，而是要看读写比例、查询复杂度、数据一致性要求和未来扩展方式。

- 这道题更像一张总览图：它负责先把“主键、字段、约束、查询路径、扩展成本”这些点串起来，后面时间字段、排序规则、JSON、拆表这些题是在分别展开其中的关键细节。

- 面试里可以把它总结成一句话：建表设计既是在定义数据结构，也是在提前决定查询效率、数据质量和后续演进成本。

### 追问答案

- 主键为什么很重要？
  因为它不只是唯一标识，还会影响 InnoDB 的数据组织方式、二级索引大小、插入顺序和后续扩展方案。
- 字段类型为什么不能随便用 `varchar`？
  因为类型越宽、长度越大，存储和索引成本越高，而且会弱化数据语义，后续校验和优化也更麻烦。
- 大字段为什么要特别小心？
  因为它会显著放大行大小、IO 和缓存开销，如果还参与高频查询或索引，性能和维护成本都会明显变差。
- 大字段一般怎么处理？
  核心原则是少参与热点查询、少进索引、必要时拆表。高频主表尽量只保留核心字段，把长文本、JSON、富文本、附件描述这类内容放到附表或专门存储里。
- 金额字段为什么不建议用 `float` / `double`？
  因为浮点数有精度问题，做金额计算容易出现误差。工程上更常见的是用定点数，或者直接用“分”这类最小货币单位的整数。
- 状态字段设计要注意什么？
  要让状态值收敛、语义清楚，不要把一个状态字段设计成什么都往里塞的“万能字段”，否则后面业务判断、索引设计和数据治理都会很乱。
- 哪些字段适合加唯一约束？
  业务上天然唯一、且真的不允许重复的数据，比如用户名、订单号、业务单号、租户内唯一编码等。
- 建表时为什么要提前考虑索引？
  因为索引本质上是为查询路径服务的，如果建表时完全不考虑查询模式，后面往往只能靠补索引甚至改表结构来救火。
- 建表时要不要预留扩展字段？
  不建议为了“也许以后会用”随便预留一堆无意义字段，但像审计字段、状态字段、版本号、租户字段这类高频通用字段，通常值得提前考虑。

## 4. datetime 和 timestamp 怎么选？

### 核心讲解

- 两者都能存时间，但 `datetime` 更像“业务时间”，范围更大、语义更稳定；`timestamp` 更像“系统时间”，占用更小，且和时区、自动更新时间特性关联更强。大多数业务时间字段通常优先考虑 `datetime`。

- `datetime` 和 `timestamp` 最大的区别，不只是存储格式，而是使用语义和适用场景不同。

- `datetime` 更适合表达业务含义明确的时间，比如下单时间、预约时间、活动开始时间，因为它的语义更直接，也不容易因为时区处理让人困惑。

- `timestamp` 更适合记录系统层面的时间，比如创建时间、更新时间，尤其是在需要配合自动更新特性时会比较方便。

- 另外，`timestamp` 的时间范围比 `datetime` 小得多，所以如果业务时间跨度很大，通常更适合用 `datetime`。

- 真正线上容易出问题的，不是“这两个类型谁绝对更高级”，而是时区、序列化、应用层解析和数据库字段语义没统一。

- 所以面试里可以这样答：如果这是业务时间，优先考虑 `datetime`；如果是系统记录时间，且希望利用 `timestamp` 的一些特性，可以用 `timestamp`。核心是字段语义要统一，时区处理要一致。

## 5. 字符集和排序规则为什么会影响唯一索引和查询结果？

### 核心讲解

- 字符集决定“怎么存和怎么解释字符”，排序规则决定“怎么比较和怎么排序字符”。唯一索引、等值比较、模糊查询、大小写敏感与否，都会受它们影响，所以这不是纯展示问题，而是会直接影响数据正确性。

- 很多人建表时只关心字段类型，却忽略了字符集和排序规则，但字符串比较到底算不算相等、能不能插入重复值，很多时候就取决于这两个配置。

- 比如某些排序规则下，`abc` 和 `ABC` 比较时可能被认为相等；如果你在这种规则下建唯一索引，就可能出现“看起来不同，数据库却不让插”的情况。

- 反过来，如果业务明明要求大小写严格区分，但排序规则是大小写不敏感的，那查询和唯一约束结果就可能和业务预期不一致。

- 这不仅影响唯一索引，也影响 `where` 过滤、`order by` 排序、`group by` 聚合，甚至影响索引能否稳定符合预期。

- 所以建表时，像用户名、邮箱、编码、标签、业务编号这类字符串字段，不能只想“用 `varchar` 就行”，还要想清楚比较规则是不是符合业务语义。

- 面试里可以总结成一句话：字符集决定字符串怎么存，排序规则决定字符串怎么比；唯一索引和查询结果本质上都依赖“怎么比”，所以配置不对就会直接影响业务正确性。

  因为唯一索引判断“是否重复”，本质上就是在做字符串比较，而比较规则由排序规则决定。

## 6. JSON 字段适合什么场景？什么时候不适合？

### 核心讲解

- JSON 字段适合存结构不稳定、扩展频繁、查询要求不高的附加信息；如果字段会频繁过滤、排序、关联、做强约束，那通常还是拆成结构化字段更合适。

- JSON 的优势是灵活。业务初期字段变化快、不同记录结构不完全一致时，用 JSON 存附加属性会比频繁改表更省事。

- 所以它比较适合存扩展配置、额外属性、动态表单结果、日志附加信息、第三方回包等“有结构但结构不稳定”的内容。

- 但 JSON 不是万能方案。只要某些字段开始频繁参与查询、排序、join、唯一约束、统计分析，继续把它们埋在 JSON 里，查询和索引设计就会越来越别扭。

- 另外，JSON 字段也容易把“本该有明确语义和约束的数据”变成一团弱约束结构，后面治理、校验和迁移都会更麻烦。

- 所以更稳的思路通常是：核心字段结构化，低频扩展字段放 JSON。不要因为 JSON 灵活，就把主模型设计偷懒成一个大包。

- 面试里可以这样答：JSON 适合解决“结构多变”的问题，不适合掩盖“核心字段没想清楚”的问题。凡是高频查询、高频过滤、强约束字段，通常还是拆出来更好。

## 7. 什么时候该拆表，什么时候不该拆？

### 核心讲解

- 当大字段、冷热数据、低频附加信息已经明显拖累主表性能时，可以考虑拆表；如果拆表只会让查询更复杂、收益又不明显，那就不该为了“看起来更规范”硬拆。

- 拆表最常见的动机是把不同访问特征的数据分开。比如主表是高频查询，但某些字段很大、很少读、变化快，这种时候拆出去往往更合理。

- 冷热分离也是典型场景，核心链路只保留高频字段，把历史字段、扩展字段、长文本字段放到附表，可以减少主表行宽和索引维护成本。

- 但拆表不是越细越好。只要拆完后大部分查询都必须 join 回来，或者应用层维护成本明显上升，而性能收益又不明显，那就属于过度拆分。

- 所以判断标准通常不是“字段多不多”，而是访问模式是否真的不同，以及拆完后核心查询是不是更稳定、更轻。

- 面试里可以总结成一句话：当不同字段的访问频率和性能诉求明显不同，拆表才更有价值；如果拆了只是让业务更复杂，通常就不值得。

- 什么场景不适合拆表？

# 线上 DDL 与数据变更

- 这一组主要讲两类高风险操作：批量改数据，以及在线改表。它们共同的问题不是语法难，而是很容易把事务、锁、日志、主从延迟和兼容性风险一起放大。
- 学这一组时要抓住一条主线：先控制单次改动的体量，再理解结构变更为什么危险，然后搞清楚 DDL 为什么会被锁卡住，最后再看工具和 online DDL 的边界。

## 学习目标

- 理解批量插入、更新、删除为什么不能简单“一把梭”，以及为什么要分批、限速、可重试。
- 理解线上 DDL 为什么危险，以及 online DDL、MDL、pt-osc、gh-ost 分别在解决什么问题。
- 能从“数据变更风险控制”“结构变更风险控制”“锁与工具兜底”三个视角看待线上改表。

## 先修知识

- 大事务
- undo / redo / binlog
- online DDL
- MDL
- 主从延迟

## 本模块总结

- 很多线上变更事故，本质上不是 SQL 语法问题，而是改动体量太大、锁太重、兼容性没处理好。
- 批量数据改动和 DDL 变更虽然形式不同，但都要回答同一件事：这次改动会不会把事务、锁、日志、复制和业务窗口压垮。
- 稳定的做法通常不是“更大胆地改”，而是把大改动拆小、把高风险步骤前移评估、把锁风险和回滚路径提前想清楚。

## 4. 线上表加字段要注意什么？

### 核心讲解

- 线上加字段本质上也是 DDL 变更，但相比加索引，它更要关注是否锁表、是否重建表、默认值带来的代价，以及应用和数据的前后兼容性。

- 线上表加字段，第一步不是直接执行 `alter table`，而是先判断这次变更会不会触发表重建、长时间锁表，是否支持在线 DDL。

- 第二步要看字段定义本身，比如是不是 `not null`、有没有默认值、是不是大字段。不同定义对执行代价影响很大。

- 第三步要考虑应用兼容性。老代码是否能容忍这个字段不存在，新代码是否能容忍这个字段暂时为空，这往往比 DDL 本身更容易出事故。

- 工程上更稳的做法通常是分阶段发布：先加可空字段，再上线代码开始写入，再逐步读取新字段；如果未来要改成非空或加约束，最好再拆成后续步骤。

- 还要关注主从延迟、磁盘空间、DDL 执行时间和发布窗口，避免在高峰期直接改大表。

## 5. metadata lock（MDL）是什么？为什么线上 DDL 会被它卡住？

### 核心讲解

- MDL 是 MySQL 的元数据锁，用来保护表结构一致性。线上 DDL 经常不是“执行很慢”，而是先被已有事务拿着的 MDL 卡住，导致 `alter table` 一直排队，后面的读写也可能被连带堵住。

- MySQL 不只是给数据加锁，也会给表结构相关操作加锁，这就是 metadata lock。

- 普通查询和事务在访问表时，也会持有相应的 MDL，只是很多时候你平时感知不到。

- 当你执行 DDL，比如 `alter table` 时，数据库通常需要更强的 MDL，来保证改表过程中表结构不会被别的会话并发改乱。

- 问题就在于，如果前面已经有长事务、慢 SQL、长时间不提交的会话占着表的 MDL，DDL 就拿不到更高等级的锁，只能一直等。

- 更麻烦的是，DDL 一旦在等待队列里，后面新的读写请求也可能被影响，所以现场常见现象不是“DDL 很慢”，而是“业务突然都卡住了”。

- 所以面试里可以这样答：线上 DDL 被卡，很多时候不是 DDL 本身太重，而是前面有事务或查询长期占着 MDL，导致 DDL 拿不到需要的元数据锁，进而把后续请求也拖住。

### 追问答案

- 普通 `select` 也会和 DDL 互相影响吗？
  会。普通查询虽然不是在改表，但也会持有相应的 MDL，长查询或长事务就可能挡住 DDL。
- 为什么 DDL 等待时后面的请求也可能被堵住？
  因为 DDL 进入锁等待队列后，新的请求和锁兼容判断会受到影响，容易形成“前面一个 DDL，后面一串会话排队”的现象。
- 怎么避免线上 DDL 被 MDL 卡住？
  选低峰期、提前清理长事务、避免大查询长时间占表，并在执行前先观察活跃会话和事务状态。
- 发现 DDL 被 MDL 卡住时先看什么？
  先看当前活跃事务、长时间执行的 SQL、是否有未提交事务占着目标表，再判断要不要先清理阻塞源头。
- 最常见的误判是什么？
  误以为 DDL 慢一定是执行代价太高，实际上很多时候它根本还没开始干活，只是在等锁。

## 6. 为什么线上 DDL 有时要用 pt-osc 或 gh-ost？

### 核心讲解

- 因为很多大表上的 `alter table` 即使语法能执行，也不代表线上风险可接受。`pt-osc` 和 `gh-ost` 这类工具，本质上是在用“旁路变更 + 渐进切换”的方式，降低锁表、长时间阻塞和主从延迟风险。

- 线上 DDL 最大的问题不是“能不能改”，而是改的过程中会不会长时间持锁、消耗大量 IO、拖慢主从，甚至影响线上写流量。

- MySQL 某些版本和某些 DDL 虽然支持 online DDL，但不是所有变更都能做到真正低风险。有的仍然会持有元数据锁，有的会触发表重建，有的执行时间会很长。

- 所以在大表、核心表、业务高峰场景下，很多团队不会直接赌一条 `alter table`，而是会考虑 `pt-osc` 或 `gh-ost` 这类在线变更工具。

- 它们的核心思路可以简单理解成：

- 先建一张新表，并把新结构提前建好。

- 再把旧表数据逐步搬到新表。

- 搬迁过程中持续同步增量变更，尽量让新旧表保持接近一致。

- 最后在合适时机做一次切换，把流量从旧表切到新表。

- 这样做的好处是，大部分重活都在“后台渐进完成”，而不是把压力集中在一次长时间 DDL 上。

- 两者也有差异：

- `pt-osc` 更早、更通用，常见做法是通过触发器同步增量数据。

- `gh-ost` 更偏 MySQL 主从复制场景，通常通过 binlog 捕获增量变更，对主库触发器依赖更少。

- 面试里可以这样答：不是所有线上 DDL 都必须上工具，但当表很大、业务不能接受长时间阻塞、online DDL 风险仍然偏高时，`pt-osc`、`gh-ost` 这类工具会更稳，因为它们把一次高风险结构变更拆成了渐进迁移和最终切换。

- 这题真正要理解的，不只是结论本身，而是“6. 为什么线上 DDL 有时要用 pt-osc 或 gh-ost？”背后的原理、适用边界和工程取舍。

### 追问答案

- online DDL 不是已经能在线了吗，为什么还要这些工具？
  因为 online DDL 不等于完全无锁、无风险，也不是所有 DDL 都支持低代价在线执行。大表和核心表上，很多团队还是会更保守。
- `pt-osc` 和 `gh-ost` 本质解决什么问题？
  解决的是大表 DDL 过程中的锁影响、阻塞风险、主从延迟和一次性切换成本。

  不需要。小表、低峰期、风险可控、且 DDL 本身支持低代价在线执行时，直接做 online DDL 也可以。
- 最常见的误判是什么？
  误以为上了工具就一定没有锁和切换风险，实际上工具只是把风险改成了更可控的渐进迁移。

## 7. online DDL 就等于零风险，这是不是个误区？

### 核心讲解

- 这是典型误区。online DDL 只能说明风险降低了，不代表完全无锁、无阻塞、无主从延迟，也不代表所有 DDL 都能低代价在线执行。

- 很多人看到某个 DDL 支持 online，就会误以为可以随时放心在线改大表。

- 但实际场景里，DDL 仍然可能持有元数据锁、触发表重建、消耗大量 IO、拖慢主从，甚至因为 MDL 排队把业务卡住。

- 另外，不同 MySQL 版本、不同变更类型、不同表大小，对 online DDL 的真实代价差异也很大。

- 所以 online DDL 更像“降低风险的能力”，不是“零风险许可”。

- 面试里可以总结成一句话：online DDL 很重要，但它不是免死金牌；线上改表是否安全，最终还要看锁、资源、版本、表规模和业务窗口。

  MDL 解释了 DDL 为什么会被锁卡住，pt-osc/gh-ost 解释了为什么有时要绕开直接改表，这一题是在最后收口：即使有 online DDL 能力，也仍然要把这些风险一起考虑。

# 数据库与缓存

- 这一组主要帮助区分数据库和缓存各自擅长解决什么问题，以及为什么缓存虽然有效，却不能替代数据库根因优化。
- 学这一组时要建立一条完整主线：缓存最擅长放大读路径收益，但它会带来一致性和回源风险，所以必须先判断问题到底适不适合缓存。

## 学习目标

- 理解缓存适合解决哪些典型问题，以及哪些问题本质上仍然应该先回到数据库侧处理。
- 能建立“先看根因，再决定要不要缓存”的判断习惯，而不是把缓存当成统一补救方案。
- 能把缓存收益、缓存一致性成本、回源风险和数据库优化边界放在一起看。

## 先修知识

- 热点读
- 缓存一致性
- 回源
- 缓存穿透 / 击穿 / 雪崩

## 本模块总结

- 缓存最擅长放大重复读路径的收益，但不擅长掩盖数据库根因问题。
- 如果数据库模型、SQL、索引本身就有问题，缓存只能短期缓解，不能真正根治。
- 这组的关键不是“要不要缓存”，而是“缓存到底该承担哪部分责任，以及它会带来什么额外复杂度”。

## 最小实践建议

- 先拿一个热点详情页场景，判断它为什么适合缓存。
- 再拿一个慢 SQL 或高写入场景，判断为什么缓存并不能直接解决问题。

- 练习把“数据库优化优先”和“缓存放大收益”这两个顺序用自己的话讲清楚。

## 1. 什么场景该用缓存，什么问题不能指望缓存解决？

### 核心讲解

- 默认优先场景是高频读、热点数据、重复计算和可容忍短暂不一致；但缓存不是数据库设计错误的遮羞布，慢 SQL、坏索引、复杂一致性和写压力根因，不能简单靠缓存掩盖。

- 缓存最大的价值，是把大量重复读请求从数据库前面拦下来，降低数据库读压力和响应时间。

- 所以它特别适合热点详情、首页数据、排行榜、配置项、统计结果、短时间内重复读取很多次的数据。

- 如果数据变更频率不高、读取频率很高，缓存收益通常很明显。

- 但缓存解决不了所有问题。如果数据库本身有慢 SQL、索引设计差、锁冲突、大事务、写放大这些根因，单纯加缓存往往只是把问题暂时藏起来。

- 另外，缓存还会引入自己的复杂度，比如缓存一致性、失效策略、穿透、击穿、雪崩、回源放大、热点 key 等问题。

- 所以更稳的原则通常是：先把数据库模型、SQL、索引做对，再用缓存去放大收益，而不是拿缓存替代本该做好的数据库优化。

- 面试里可以总结成一句话：缓存最适合“放大已经正确的读路径”，不适合“掩盖本来就错误的数据库设计”。

### 深入理解

- 这题真正要理解的，不只是结论本身，而是“1. 什么场景该用缓存，什么问题不能指望缓存解决？”背后的原理、适用边界和工程取舍。

### 追问答案

- 什么场景最适合上缓存？
  高频读、热点数据、相同结果被大量重复读取、允许短时间内数据略旧的场景。
- 什么问题不能指望缓存解决？
  慢 SQL 根因、坏索引、锁冲突、大事务、复杂强一致要求、持续高写入压力，这些都不能简单靠缓存解决。

## 2. 什么时候该先优化数据库，什么时候才值得上缓存？

### 核心讲解

- 如果问题本质是慢 SQL、索引设计差、锁冲突或写压力，应该先优化数据库；如果问题本质是高频重复读、热点读、重复计算，缓存才更值得优先上。

- 缓存最适合解决的是“同样的结果被反复读取”，也就是热点读和重复读问题。

- 但如果数据库本身的查询路径就是错的，比如没索引、索引失效、分页策略糟糕、锁冲突严重，那先上缓存往往只是把问题暂时盖住。

## 3. 缓存穿透、击穿、雪崩怎么理解？

### 核心讲解

- 三者本质上都是“缓存没拦住请求，导致数据库回源压力放大”，区别在于原因不同：穿透是查根本不存在的数据，击穿是热点 key 失效瞬间并发回源，雪崩是大量 key 同时失效或缓存层异常，整体把数据库压垮。

- 缓存穿透，通常是请求查的是根本不存在的数据。因为缓存里没有，数据库里也没有，如果这类请求很多，就会直接打到数据库。

- 缓存击穿，通常是某个非常热点的 key 刚好失效，在它重建缓存的瞬间，大量请求同时回源，数据库会在一个点上被猛打。

- 缓存雪崩，通常是大量 key 同时过期，或者缓存层整体异常，结果不是一个热点回源，而是大面积请求一起打回数据库。

- 这三类问题虽然名字不同，但本质上都和“回源压力控制”有关：缓存一旦没拦住请求，数据库是否还能扛住，是核心问题。

- 所以工程上常见的思路是：对穿透做空值缓存或布隆过滤，对击穿做热点保护、互斥重建，对雪崩做过期时间打散、限流降级和缓存层高可用。

- 面试里可以总结成一句话：穿透、击穿、雪崩不是缓存术语背诵题，本质上都是缓存失效后数据库回源压力怎么被放大的问题。

### 追问答案

- 三者最核心的区别是什么？
  穿透是查不存在的数据，击穿是单个热点 key 失效后被并发打穿，雪崩是大量 key 一起失效或缓存层整体异常。

  因为真正危险的不是“缓存没命中”本身，而是缓存没拦住之后，数据库会不会被瞬间打垮。
- 常见缓解思路是什么？
  穿透常见做空值缓存、参数校验、布隆过滤；击穿常见做热点 key 保护、互斥重建；雪崩常见做过期时间打散、限流降级、缓存层高可用。

## 4. 上了缓存数据库压力就一定能解决，这是不是个误区？

### 核心讲解

- 是误区。缓存能显著缓解高频读压力，但它并不能自动消灭慢 SQL、坏索引、写压力、锁冲突和一致性问题。缓存加得不对，反而会把系统变得更复杂。

- 一旦缓存失效、回源放大、热点穿透，原来的数据库问题会更明显地暴露出来。
