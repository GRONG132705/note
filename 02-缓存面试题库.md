## 学习目标

- 理解缓存最适合解决什么问题，以及哪些问题不能简单靠缓存掩盖。
- 能把读多写少架构、热点 Key、本地缓存、Redis 和数据库之间的关系讲清楚。
- 能从架构设计、热点治理、Key 设计、预热、容量治理和命中率几个维度回答缓存系统设计题。

## 先修知识

- 热点读
- 回源
- 本地缓存 / 分布式缓存
- TTL
- 缓存命中率

## 本模块总结

- 缓存的核心价值，是把大量重复读挡在数据库前面，降低回源和尾延迟。
- 真正稳的缓存架构，不只是多放一层 Redis，还要考虑本地缓存、热点治理、失效兜底和监控。
- 缓存设计不是只看“查得快不快”，而是要一起看命中率、可用性、回源风险和维护成本。
- 更完整的缓存设计路径通常是：先判断值不值得缓存，再设计读路径，然后处理热点、预热、Key、命中率和容量问题。

## 最小实践建议

- 先拿一个读多写少场景，画出 `客户端 -> 网关 -> 本地缓存 -> Redis -> MySQL` 的链路。
- 再模拟一个热点 Key 失效场景，想清楚回源是怎么把数据库打穿的。
- 试着把“缓存架构设计、热点治理、预热、Key 设计、命中率排查、容量治理”串成一条完整主线。

## 1.1 什么场景适合上缓存，什么问题不能指望缓存解决？

### 30秒结论

- 默认更适合上缓存的，是高频读、热点数据、重复计算和可容忍短暂不一致的场景；但缓存不是数据库根因问题的遮羞布，慢 SQL、坏索引、锁冲突和高写压力不能简单靠缓存掩盖。

### 3分钟回答

- 缓存最大的价值，是把大量重复读请求从数据库前面拦下来，降低数据库读压力和接口 RT。
- 所以它特别适合热点详情、首页数据、排行榜、配置项、统计结果这类读多写少、重复读取多的内容。
- 如果数据变更频率不高、读取频率很高，缓存收益通常会很明显。
- 但如果数据库本身有慢 SQL、索引设计差、锁冲突、大事务、写放大这些根因，单纯加缓存往往只是把问题暂时藏起来。
- 而且缓存还会带来一致性、失效、穿透、击穿、雪崩、热点 Key 和回源放大这些新复杂度。
- 所以更稳的原则通常是：先把数据库模型、SQL、索引做对，再用缓存去放大收益，而不是拿缓存替代本该做好的数据库优化。

### 追问答案

- 什么场景最适合上缓存？
  高频读、热点数据、重复计算多、允许短时间内数据略旧的场景。
- 什么问题不能指望缓存解决？
  慢 SQL 根因、坏索引、锁冲突、大事务、持续高写入压力和强一致链路。
- 为什么说缓存会带来新复杂度？
  因为你还要额外处理一致性、失效、回源、热点和故障切换，不是“多加一层”就结束了。

## 1.2 如何设计一个读多写少的缓存架构？

### 30秒结论

- 典型设计通常是 `CDN -> Nginx/OpenResty -> 本地缓存 -> Redis -> MySQL`，核心目标是减少回源、抗峰值、控制尾延迟；真正回答这题时，要先判断是不是热点读，再判断能不能接受旧数据，最后才决定本地缓存、Redis、预热、降级和一致性策略。

### 3分钟回答

- 先看流量模型。如果是典型读多写少，我会优先把绝大部分读请求挡在缓存层，减少数据库直接承压。
- 更稳的回答顺序通常是：先判断是不是热点读和重复读，再判断业务能不能接受短暂旧数据，再决定要不要上本地缓存、多级缓存、预热和降级兜底。
- 架构上一般会做入口层、应用层、缓存层和数据库层。热点优先本地缓存，其次 Redis，数据库只处理少量未命中的请求。
- 一致性上会根据业务容忍度选择旁路缓存、异步刷新或者延迟双删，不盲目追求强一致。
- 稳定性上要补热点治理、限流熔断降级和监控，至少盯住 QPS、P99、缓存命中率、数据库连接数和错误率。
- 这题讲完后，下一步通常就会被追问热点 Key、命中率和缓存失效后的回源风险。

### 追问答案

- 为什么要加本地缓存，直接 Redis 不行吗？
  本地缓存能进一步降低网络开销和 Redis 压力，特别适合极端热点数据。
- 多级缓存最大的价值是什么？
  最大价值是分层拦截流量，把绝大部分请求挡在更便宜、更快的层级。
- 缓存层设计时最容易忽略什么？
  最容易忽略的是故障兜底和一致性窗口，很多系统只盯命中率，不盯失效场景。

## 1.3 设计缓存系统时，一般从哪些维度考虑？

### 30秒结论

- 更稳的回答顺序通常是 6 个维度：值不值得缓存、读路径怎么分层、能不能接受旧数据、热点怎么保护、Key 和 TTL 怎么设计、命中率和容量怎么治理。

### 3分钟回答

- 第一，先判断这个数据值不值得缓存，是不是高频读、重复读、回源贵，而且能接受短暂旧数据。
- 第二，再设计读路径，决定是不是只用 Redis，还是要上本地缓存、多级缓存、预热和降级。
- 第三，看一致性要求高不高，决定旁路缓存、异步刷新还是更重的补偿方案。
- 第四，看热点风险大不大，如果有热点，就要提前设计热点识别、预热、请求合并和回源保护。
- 第五，看 Key 和 TTL 设计是否稳定，因为很多命中率问题并不是缓存层慢，而是 Key 设计本身有问题。
- 第六，最后一定要补容量和监控，包括命中率、淘汰策略、内存利用率、回源比例和数据库承压情况。
- 这题适合作为系统设计总题，后面的热点、预热、Key、命中率和淘汰，其实都可以挂在这 6 个维度下面展开。

### 追问答案

- 为什么先别急着画架构图？
  因为如果先不判断业务特征和一致性要求，后面的缓存层设计很容易变成拍脑袋。
- 为什么热点和容量都要提前考虑？
  因为缓存最容易出问题的地方，往往不是平时命中，而是热点失效和内存打满时。
- 这题和后面几题是什么关系？
  这题是总框架，后面的本地缓存、热点、预热、Key、命中率和淘汰，都是它的展开。

## 1.4 本地缓存和 Redis 怎么配合？

### 30秒结论

- 一般是本地缓存挡第一层极端热点，Redis 挡第二层共享缓存，数据库做最终兜底。核心是降低网络开销和分布式缓存压力。

### 3分钟回答

- 本地缓存的优势是访问速度快、没有网络开销，适合极高频热点和短 TTL 数据。
- Redis 的优势是分布式共享、容量更大、更新更集中，适合跨节点共享数据。
- 两者配合时要特别注意一致性和失效广播，因为本地缓存天然会有节点间延迟。
- 所以更稳的实践通常是：极端热点优先本地缓存，跨实例共享结果走 Redis，最终再由数据库兜底。
- 但本地缓存不是默认必选项，如果热点不极端、数据更新频繁、节点一致性要求高，只用 Redis 反而更稳。

### 追问答案

- 本地缓存最大的优点是什么？
  极低延迟和零网络开销。
- 本地缓存最大的缺点是什么？
  节点间数据副本分散，一致性和失效传播更难。
- 什么场景适合只用 Redis，不上本地缓存？
  热点不极端、数据更新频繁、节点间一致性要求更高的场景。

## 1.5 如何处理热点 Key？

### 30秒结论

- 热点 Key 的核心不是查得快，而是不能在失效或抖动时把数据库打穿。常见手段是热点探测、请求合并、本地缓存、预热、互斥保护和限流降级。

### 3分钟回答

- 热点 Key 的本质是访问高度集中，一旦失效，就容易引发瞬时回源风暴。
- 先要识别热点，常见方式有网关日志、Redis 统计、应用埋点和监控平台。
- 识别出来后，治理手段通常包括本地缓存、请求合并、singleflight、互斥锁和限流保护。
- 如果热点非常极端，还可以做 Key 拆分、静态化、异步构建，把最重的查询尽量提前或旁路化。
- 这题重点要讲“识别 + 保护 + 回源治理”；而预热更适合单独放在下一题里讲“提前准备”。

### 追问答案

- 热点探测怎么做？
  通过网关日志、Redis 统计、应用埋点或监控平台识别短时间高集中访问。
- 本地缓存会不会有一致性问题？
  会，所以本地缓存更适合热点读和弱一致场景。
- 热点 Key 最容易忽略什么？
  很多人只盯住命中率，却没设计失效瞬间的回源保护和降级方案。

## 1.6 缓存预热怎么做？

### 30秒结论

- 缓存预热就是在流量真正打进来之前，提前把热点数据加载到缓存里，避免活动开始或系统重启后瞬间回源。

### 3分钟回答

- 预热常用于大促、活动页、排行榜、详情页这类已知热点场景。
- 常见方式有离线任务提前加载、启动时加载、按访问日志分析热点后回填。
- 预热不是一次性动作，还要配合 TTL、逻辑过期和监控，不然只是把问题延后。
- 预热本质上是把热点准备工作做在流量进来之前，所以它和热点治理、回源保护是同一条线。
- 这题的重点是“提前准备”，和上一题的热点保护不是一回事。

### 追问答案

- 什么数据适合预热？
  已知热点、访问集中、生成成本高、活动前可预测的数据。
- 预热过多会有什么问题？
  会占用大量缓存空间，还可能把真正的高价值热点挤掉。
- 为什么预热后还要配合监控？
  因为热点变化很快，预热数据未必等于真实线上热点。

## 1.7 缓存 Key 怎么设计更合理？

### 30秒结论

- Key 设计要兼顾唯一性、可读性、可扩展性和批量治理能力。核心原则是前缀分层、业务语义清晰、维度稳定。

### 3分钟回答

- 一般会按 `系统:业务:对象:标识` 这样的格式设计，既能避免冲突，也方便后续批量删除和排查。
- Key 不要拼太多无意义参数，否则很容易造成缓存碎片化。
- 对于可能批量失效的场景，还要提前考虑版本号、租户维度、区域维度等治理能力。
- 常见反模式是把所有查询参数都原样拼进 Key、没有版本前缀、没有批量治理能力，结果看起来唯一，实际很难命中、也很难维护。
- 这题往往会和命中率一起被问，因为很多命中率问题本质上都出在 Key 设计不稳定。

### 追问答案

- 为什么 Key 设计不能太随意？
  因为后续排查、批量清理、命中率分析和多业务隔离都依赖 Key 规则。
- 为什么不建议把所有参数都拼进 Key？
  因为容易导致同类数据被切成大量碎片，命中率反而变差。
- Key 前缀有什么价值？
  便于按业务分组治理、监控和批量失效。

## 1.8 缓存命中率低怎么排查？

### 30秒结论

- 命中率低不一定是缓存没做好，排查顺序通常是：先按业务拆分，再看 Key 和 TTL，再看更新频率，最后判断业务是不是天然不适合缓存。

### 3分钟回答

- 第一步先区分是全局命中率低，还是某一类业务、某一类 Key 的命中率低，不要只看一个总平均值。
- 第二步看 Key 粒度和 TTL 是否合理，是否因为参数拼接过多导致缓存碎片化，或者 TTL 太短导致刚写入就过期。
- 第三步看数据更新频率，如果写特别频繁，缓存可能天然很难稳定命中。
- 第四步再判断业务本身是不是离散度太高、复用率太低，这种场景即使硬上缓存，收益也可能有限。
- 所以命中率排查不是先怪缓存层，而是先确认是不是缓存策略、数据特征或业务模式本身就不匹配。
- 这题里最容易忽略的一点是：命中率低不一定是坏事，有些业务天然就很离散，本来就不适合追求高命中。

### 追问答案

- 命中率低最常见的原因是什么？
  Key 粒度过细、TTL 不合理、缓存对象太离散、写频繁读也频繁。
- 是不是命中率越高越好？
  也不能绝对化，高命中率如果靠极长 TTL 换来大量旧数据，也未必合理。
- 命中率低时第一步先看什么？
  先按业务维度拆分命中率，不要只看全局平均值。

---

## 1.9 缓存容量不足时，怎么做淘汰与容量治理？

### 30秒结论

- 缓存容量不足时，不能只盯淘汰策略本身，更要一起看容量规划、热点分布、TTL 设计和命中率。淘汰策略本质上是在内存有限时，决定谁该被清掉。

### 3分钟回答

- 在 Redis 这类内存缓存里，空间一定有限，所以必须同时考虑“怎么淘汰”和“容量够不够”。
- 常见策略比如 LRU、LFU、TTL 优先或随机淘汰，不同策略适合不同访问模式。
- 面试里不只要会背策略名字，更要说明热点稳定、访问频率分布和内存压力会影响策略选择。
- 如果只调淘汰策略、不看容量规划、Key 设计和热点分布，很多问题其实解决不了。
- 所以这题本质上是容量治理问题，和命中率、热点保护一起理解会更完整；如果要继续往下追 Redis 级别的淘汰细节，再看第 `3.9` 题会更顺。

### 追问答案

- LRU 和 LFU 怎么选？
  热点变化快时 LRU 常更直观；热点长期稳定、访问频率差异明显时 LFU 更有优势。

- 淘汰策略会影响命中率吗？
  会，策略选错会把真正的热点数据提前淘汰掉。

- 为什么缓存容量规划和淘汰策略要一起看？
  因为只调策略不看容量，很多问题本质上还是内存不够。

- 到这里为止，更多是在回答“缓存怎么设计才稳”；下一组会继续回答“缓存为什么会不一致、为什么会被打穿，以及故障时怎么保住系统”。

## 1.10 什么情况下不适合上本地缓存？

### 30秒结论

- 本地缓存不是默认必选项。如果数据更新频繁、节点间一致性要求高、热点不够集中，或者应用实例很多且失效广播成本高，通常不适合优先上本地缓存。

### 3分钟回答

- 本地缓存最大的价值是极低延迟和零网络开销，但代价是每个节点都有一份副本，一致性和失效传播会更难。
- 如果数据更新特别频繁，本地缓存刚写进去就要失效，收益可能很有限。
- 如果业务要求各节点读取结果尽量一致，本地缓存带来的延迟失效窗口会更敏感。
- 如果热点不够极端，只用 Redis 已经足够快，再叠本地缓存反而会增加复杂度。
- 另外，当实例数量很多、节点频繁伸缩时，本地缓存的失效广播、预热和治理成本也会明显上升。
- 所以回答这题时，重点不是说本地缓存不好，而是说明它更适合“极端热点、弱一致、读远大于写”的场景。

### 追问答案

- 本地缓存最适合什么场景？
  极端热点、读取频率极高、允许短时间弱一致的场景。
- 为什么实例越多越要谨慎？
  因为实例越多，副本越分散，失效传播和预热治理成本越高。
- 本地缓存是不是一定比 Redis 更优？
  不是，它只是更快，但不代表整体系统更稳。

## 1.11 缓存和 CDN / 浏览器缓存怎么分工？

### 30秒结论

- 浏览器缓存、CDN、服务端缓存不是替代关系，而是分层分工：浏览器缓存离用户最近，CDN 负责边缘分发，服务端缓存负责业务数据加速，数据库做最终数据源。

### 3分钟回答

- 浏览器缓存更适合静态资源、本地可复用内容，目标是减少重复下载和提升首屏体验。
- CDN 更适合把静态资源和部分可边缘缓存内容提前分发到离用户更近的节点，降低回源和跨地域延迟。
- 服务端缓存，比如本地缓存和 Redis，更适合缓存业务数据、接口结果、热点查询和计算结果。
- 这三层不是二选一，而是从“离用户越来越近”到“离业务数据越来越近”的分层设计。
- 工程上常见顺序是：静态资源优先浏览器缓存和 CDN，动态业务结果优先服务端缓存，数据库只做最终兜底。
- 所以面试里回答时，不要把缓存只理解成 Redis，要把前端缓存、边缘缓存和业务缓存放在一条链上看。

### 追问答案

- CDN 能缓存动态接口吗？
  某些可缓存的动态内容可以，但前提是缓存策略、时效性和一致性都能接受。
- 浏览器缓存和 Redis 最大区别是什么？
  浏览器缓存更偏客户端资源复用，Redis 更偏服务端业务数据加速。
- 为什么要做多层缓存分工？
  因为不同层距离用户和数据源的位置不同，最擅长解决的问题也不同。

## 1.12 设计一个秒杀场景的缓存方案，应该怎么答？

### 30秒结论

- 秒杀场景的缓存设计重点不是“页面快一点”，而是“热点抗住、库存别超卖、数据库别被打穿”。回答时一般按“静态化 + 多级缓存 + 限流削峰 + 异步下单 + 库存校验”这条线展开。

### 3分钟回答

- 第一层通常是静态化和 CDN，把活动页、商品信息、图片资源尽量提前缓存，减少活动开始时的页面回源。
- 第二层是服务端缓存，把商品详情、秒杀配置、库存展示等热点数据挡在本地缓存和 Redis。
- 第三层要做热点保护和限流削峰，因为秒杀的核心问题不是平均流量，而是瞬时峰值。
- 第四层是库存控制，真正下单时不能只信缓存展示，要有 Redis 原子扣减、异步队列、数据库最终校验等防超卖手段。
- 第五层要做降级和兜底，比如抢购人数太多时直接返回排队中、活动火爆提示，而不是把所有请求都打到底层。
- 所以这题的标准回答不是单点技术，而是一整条链：页面静态化、热点缓存、流量削峰、异步处理、库存一致性和故障兜底。

### 追问答案

- 为什么秒杀场景不能只靠 Redis 缓存？
  因为秒杀不只是读快，还涉及瞬时峰值、库存扣减和防超卖。
- 秒杀为什么经常要上异步队列？
  因为需要削峰，把高并发下单请求变成后端能稳定处理的节奏。
- 秒杀里缓存最大的作用是什么？
  是提前拦热点、减少回源和承接峰值，不是替代库存正确性设计。

## 本模块总结

- 缓存一致性很少有绝对完美方案，真正稳定的做法通常是接受窗口、设计补偿、监控异常。
- 穿透、击穿、雪崩名字不同，但本质上都是“缓存没拦住请求，数据库会不会被打垮”的问题。
- 高可用设计里，最重要的不是缓存坏了还能不能查，而是故障时系统能不能先止血。
- 更完整的理解顺序通常是：先看为什么会不一致，再看怎么收敛窗口，最后看缓存失效或故障时怎么保住数据库。

- 最后模拟 Redis 故障时的应急顺序，练习用“先止血，再恢复”来回答。

## 2.1 缓存一致性与高可用问题，一般从哪些链路看？

### 30秒结论

- 更稳的回答顺序通常是 3 条线：先看写路径为什么会不一致，再看读路径为什么会被打穿，最后看缓存层故障时怎么先止血。

### 3分钟回答

- 第一条线是写路径，也就是缓存和数据库为什么天然不是原子一致，这会引出旁路缓存、双写不一致、延迟双删和异步补偿。
- 第二条线是读路径，也就是缓存失效后数据库会不会被打穿，这会引出逻辑过期、布隆过滤器、穿透、击穿、雪崩。
- 第三条线是缓存层故障本身，比如 Redis 挂了、主从切换抖动、整体命中率骤降，这会引出限流、降级、静态兜底和预热恢复。
- 所以这一组不是一堆零散术语，而是在回答三个问题：为什么会不一致、为什么会回源放大、出了故障怎么保系统。
- 后面的题，基本都可以挂在这三条线下面展开。

### 追问答案

- 为什么要先分写路径和读路径？
  因为一致性问题主要发生在写路径，穿透击穿雪崩主要发生在读路径，两类问题根因不同。
- 为什么 Redis 故障要单独看一条线？
  因为它不是单个 Key 的一致性问题，而是整个缓存层失效后系统怎么止血的问题。
- 这题和后面几题是什么关系？
  这题是总框架，后面的旁路缓存、双写不一致、逻辑过期、穿透击穿雪崩和兜底，都是它的展开。

## 2.2 旁路缓存模式怎么理解？

### 30秒结论

- 旁路缓存的核心流程是 `先查缓存，未命中查数据库并回填；写时更新数据库，再删除缓存`。它是工程上最常见、复杂度最低的缓存模式。

### 3分钟回答

- 读请求先查缓存，命中就直接返回；没命中再查数据库并把结果写回缓存。
- 写请求一般不直接更新缓存，而是先更新数据库，再删除缓存，让后续读请求自然回填。
- 这个模式实现简单、侵入低，适合大多数读多写少业务。
- 这题重点是把标准读写流程和适用场景讲清楚，不必在这里展开所有一致性补偿细节。

### 追问答案

- 为什么写时通常选择删缓存而不是更新缓存？
  因为缓存不是最终数据源，删缓存后让后续读请求自然回填，链路更简单。
- 旁路缓存最大的缺点是什么？
  是并发场景下可能出现短暂不一致，尤其是删缓存和回填之间。
- 什么场景不适合只靠旁路缓存？
  对一致性要求特别高、写入频繁且并发复杂的场景。

## 2.3 缓存和数据库双写不一致怎么理解？

### 30秒结论

- 双写不一致的本质是缓存和数据库不是一个原子操作单元，只要写路径分成两步，就一定存在失败窗口和并发时序问题。

### 3分钟回答

- 不管是先写缓存再写库，还是先写库再删缓存，只要不是同一个原子事务，就会有不一致风险。
- 真正危险的地方通常不在“语句怎么写”，而在失败窗口，比如删缓存失败、消息丢失、补偿没执行到。
- 另外还有并发时序问题，比如删缓存后，其他读请求又把旧值从数据库回填回缓存。
- 所以这题应该重点讲“不一致根因 + 失败窗口 + 并发回填 + 收敛手段”，而不是重复旁路缓存流程本身。

### 追问答案

- 为什么先更新缓存再更新数据库更危险？
  因为数据库才是最终数据源，一旦后者失败，就会留下错误缓存。
- 为什么先更新数据库再删缓存更常见？
  因为至少最终真实数据先落到了数据库，不一致窗口更容易靠后续读回填和补偿收敛。
- 如果必须尽量收敛不一致怎么办？
  配合延迟双删、异步刷新、消息补偿或 binlog 驱动更新。

---

## 2.4 缓存一致性怎么保证？

### 30秒结论

- 缓存一致性没有绝对完美方案，工程上更常见的思路是按一致性要求分层：基础场景用删缓存，要求更高时加延迟双删、异步刷新、MQ 或 binlog 补偿。

### 3分钟回答

- 一致性问题的根因在于缓存和数据库不是同一个原子单元，所以工程上更常见的是收敛不一致窗口，而不是追求口头上的绝对一致。
- 基础方案通常是旁路缓存加删除缓存，因为实现简单、适用面广。
- 如果并发窗口更敏感，可以加延迟双删来补一次回填窗口。
- 如果链路更复杂、要求更高，还会引入 MQ 异步刷新、binlog 订阅更新、失败重试和补偿任务。
- 所以这题的关键不是只背一个方案，而是讲出“删缓存 -> 延迟双删 -> 异步补偿”的分层思路；如果被追问为什么会不一致，上一题的双写不一致就是它的根因题。

### 追问答案

- 为什么不直接更新缓存？
  因为缓存不是最终数据源，直接双写数据库和缓存更容易不一致。
- 延迟双删时间怎么定？
  参考业务 RT、缓存回填耗时和线程调度，通常要压测校准。
- 删除缓存失败怎么办？
  要有重试、告警和异步补偿机制，不能假设删缓存必然成功。

---

## 2.5 延迟双删怎么理解？适合什么场景？

### 30秒结论

- 延迟双删本质上是旁路缓存的补偿增强方案：更新数据库后删一次缓存，等一小段时间再删一次，降低旧值被并发回填的概率。

### 3分钟回答

- 常规旁路缓存下，删缓存后如果刚好有并发读请求读到旧库值并写回缓存，就会出现短暂脏缓存。
- 延迟双删的思路就是在这个回填窗口后，再补一刀删除，把错误回填的旧值再清掉。
- 它不能做到绝对一致，但对一致性要求比普通旁路缓存更高、又不想引入更复杂链路的场景比较实用。
- 所以它更像“补偿增强手段”，而不是独立的一套一致性体系。

### 追问答案

- 延迟双删的时间怎么定？
  没有固定值，要结合业务 RT、缓存回填耗时和调度延迟压测确定。
- 延迟双删能保证绝对一致吗？
  不能，它只是降低并发回填旧值的概率。
- 什么时候不适合延迟双删？
  写特别频繁、对一致性又极其敏感的场景，仅靠它通常不够。

---

## 2.6 逻辑过期怎么设计？

### 30秒结论

- 逻辑过期的核心不是让缓存真的过期，而是在缓存值里带一个业务过期时间。读请求先返回旧值，再由后台异步刷新。

### 3分钟回答

- 它的优势是热点 Key 不会因为物理过期瞬间失效，能显著减少击穿风险。
- 读请求发现逻辑过期后，可以先返回旧值，再由后台线程尝试加锁并刷新。
- 这种方案本质上是用短暂旧数据换系统可用性，所以适合弱一致、高并发读场景。
- 逻辑过期通常会出现在热点治理和击穿防护里，所以它既是一种一致性策略，也是一种高可用手段。

### 追问答案

- 逻辑过期和物理过期的区别是什么？
  物理过期是 Key 真失效；逻辑过期是 Key 还在，只是业务上认为它旧了。
- 为什么逻辑过期更适合热点数据？
  因为它避免了热点 Key 真失效瞬间的大量回源。
- 它最大的代价是什么？
  是可能短时间返回旧数据。

---

## 2.7 布隆过滤器在缓存体系里解决什么问题？

### 30秒结论

- 布隆过滤器主要解决缓存穿透问题，用来提前挡掉大量本就不存在的数据请求，减少数据库无效查询。

### 3分钟回答

- 它本质上是一个空间效率很高的概率型数据结构，用多个哈希函数判断某个元素大概率存在还是一定不存在。
- 在缓存层里，先查布隆过滤器，如果判断一定不存在，就直接返回，不再穿透到数据库。
- 它的代价是存在误判，所以它不是替代缓存和数据库，只是前置过滤层。
- 这题最好放在穿透治理链路里理解，而不是单独当成一个孤立算法题。

### 追问答案

- 布隆过滤器会漏判吗？
  不会把存在的数据判断成不存在，但会把少量不存在的数据误判为存在。
- 为什么它适合挡穿透？
  因为大量恶意或无效请求里，最重要的是尽快判断出一定不存在的部分。
- 布隆过滤器的数据怎么维护？
  通常在数据初始化时导入，后续新增数据同步写入，删除场景要更谨慎。

---

## 2.8 缓存穿透、击穿、雪崩分别是什么？怎么解决？

### 30秒结论

- 穿透是查不存在的数据，击穿是热点 Key 失效瞬间大量并发回源，雪崩是大量缓存同时失效导致整体回源。三者名字不同，但本质上都是“缓存没拦住请求，数据库会不会被打垮”的问题。

### 3分钟回答

- 缓存穿透本质是请求绕过缓存直接打到数据库，比如恶意请求不存在的 ID。常见治理是参数校验、空值缓存和布隆过滤器。
- 缓存击穿一般发生在热点数据上。某个热点 Key 过期后，大量请求同时回源打数据库，常用互斥锁、singleflight、逻辑过期和后台刷新控制。
- 缓存雪崩是大批量 Key 在同一时间失效，或者 Redis 整体异常。应对方法是随机过期打散、分批预热、多级缓存、降级兜底和限流保护。
- 回答这题时重点要讲三者区别和治理策略，不必再把布隆过滤器和逻辑过期的定义从头展开，点到对应手法即可。

### 追问答案

- 布隆过滤器有什么误判？
  可能把不存在的数据误判为存在，但不会把存在的数据误判为不存在。
- 逻辑过期和互斥锁怎么选？
  更看重可用性和低 RT 用逻辑过期，更看重数据新鲜度用互斥锁。
- Redis 整体不可用时怎么处理？
  先止血做限流降级和静态兜底，再排查故障并补缓存预热。

---

## 2.9 Redis 挂了以后系统怎么兜底？

### 30秒结论

- Redis 故障最怕的是流量瞬间打穿数据库，所以第一优先级不是恢复功能，而是先止血：限流、降级、静态兜底和本地缓存保护。

### 3分钟回答

- 一旦 Redis 不可用，大量原本命中缓存的流量会直接落到数据库，数据库往往扛不住。
- 所以第一步是限流和降级，把非核心流量先砍掉。
- 第二步依赖本地缓存、静态数据、只读模式等方式保住核心查询。
- 第三步再排查 Redis 本身是网络问题、主从切换问题，还是集群节点故障。
- 真正稳定的回答顺序通常是“先止血，再恢复，再预热”，而不是一上来只说怎么修 Redis。

### 追问答案

- 为什么 Redis 故障会拖垮数据库？
  因为原本被缓存挡住的大量读流量会瞬间回源。
- 什么叫缓存降级？
  就是在缓存异常时，不再追求完整实时数据，而是优先返回兜底结果保系统可用。
- Redis 恢复后为什么还要做预热？
  因为恢复瞬间如果缓存还是空的，回源流量仍可能再次击穿数据库。

---

## 2.10 先删缓存还是先更新数据库，为什么？

### 30秒结论

- 工程上更常见也更稳的顺序，是先更新数据库，再删除缓存。因为数据库才是真实数据源，先落库至少能保证正确数据先写进去，再通过删缓存和后续回填逐步收敛一致性。

### 3分钟回答

- 如果先删缓存，再更新数据库，中间一旦有并发读，就可能把旧值重新查出来并回填回缓存。
- 如果先更新数据库，再删缓存，虽然仍然有窗口，但至少真实数据已经先落到了数据库。
- 这也是为什么旁路缓存场景里，工程上更常见的是“更新数据库 + 删除缓存”，而不是反过来。
- 当然，这个顺序并不能消灭所有不一致，只是把风险控制在更容易补偿和收敛的方向。
- 所以这题重点不是说某种顺序绝对完美，而是说明为什么“先更新数据库再删缓存”通常更稳。

### 追问答案

- 先删缓存再更新数据库最大风险是什么？
  中间有并发读时，可能把旧值重新回填回缓存。
- 先更新数据库再删缓存就完全没问题吗？
  也不是，删除缓存失败、并发回填这些问题仍然存在。
- 为什么数据库顺序更重要？
  因为数据库是真实数据源，最终一致性必须围绕它来收敛。

---

## 2.11 删除缓存失败怎么办？

### 30秒结论

- 删除缓存失败不能靠“再试一下”糊过去，稳的做法通常是“重试 + 异步补偿 + 告警 + 兜底过期时间”一起上。

### 3分钟回答

- 删缓存失败的风险在于：数据库已经是新值，但缓存还留着旧值，后续读请求会持续读到脏数据。
- 第一层通常会做同步重试，先尽量在当前链路把删除补成功。
- 第二层会做异步补偿，比如 MQ、重试任务、binlog 驱动刷新，避免主链路失败后就彻底丢掉这次删除。
- 第三层要有监控和告警，不然线上可能长期存在旧缓存却没人发现。
- 第四层还会配合 TTL，让旧缓存即使删不掉，也不会无限期存在。
- 所以这题真正想表达的是：删缓存不是“发一条命令就结束”，而是要把失败当成常态来设计补偿链路。

### 追问答案

- 为什么 TTL 不能完全代替删除缓存？
  因为 TTL 只能让旧值最终过期，不能保证变更后尽快收敛。
- 为什么要加告警？
  因为如果补偿也失败，系统可能长期带着脏缓存运行。
- 最容易忽略什么？
  最容易忽略的是“删缓存失败”本身也需要工程化处理，而不是默认它一定成功。

---

## 2.12 热点 Key 失效瞬间怎么避免并发回源？

### 30秒结论

- 热点 Key 失效最怕的是并发回源打穿数据库，常见办法是互斥锁、singleflight、逻辑过期、请求合并、本地缓存和限流降级配合使用。

### 3分钟回答

- 热点 Key 的危险不在平时命中，而在失效瞬间大量请求一起落到数据库。
- 最直接的办法是互斥锁或 singleflight，让同一时刻只有一个请求回源，其他请求等待或复用结果。
- 如果更看重低 RT 和可用性，逻辑过期是更常见的做法：先返回旧值，再异步刷新。
- 如果热点特别高，还可以叠本地缓存、静态兜底、请求合并和限流降级，避免所有流量都压在一个点上。
- 所以这题本质上不是“怎么查缓存”，而是“失效瞬间怎么保护数据库和下游系统”。

### 追问答案

- 互斥锁和逻辑过期怎么选？
  更看重新鲜度和严格回源控制时偏互斥锁，更看重可用性和低 RT 时偏逻辑过期。
- 为什么热点 Key 比普通 Key 更危险？
  因为它一旦失效，并发回源会在瞬间集中爆发。
- 这题和击穿是什么关系？
  这题本质上就是在回答缓存击穿场景里最关键的保护手段。

## 3.1 Redis 作为缓存核心中间件，一般从哪些基础能力理解？

### 30秒结论

- 更稳的回答顺序通常是 4 条线：性能为什么快、数据怎么组织、数据怎么恢复、内存压力下怎么治理。

### 3分钟回答

- 第一条线是性能来源，也就是 Redis 为什么快，这会引出内存访问、单线程执行模型、数据结构优化和网络模型。
- 第二条线是数据组织，也就是 Redis 常见数据结构分别适合什么场景，这会引出 `String`、`Hash`、`Set`、`ZSet`、`Stream` 等。
- 第三条线是数据恢复，也就是 Redis 掉电或重启后怎么保数据，这会引出 `RDB`、`AOF` 和 `AOF` 重写。
- 第四条线是内存治理，也就是过期 Key 怎么删、内存不够时怎么淘汰，这会引出过期删除和淘汰策略。
- 所以这组题不是 Redis 杂项堆砌，而是在回答“Redis 为什么适合做缓存核心中间件，以及它靠什么把这件事做稳”。
- 后面的题，基本都可以挂在这 4 条线下面展开。

### 追问答案

- 为什么先看性能，再看数据结构？
  因为 Redis 先是一个高性能内存系统，然后才是各种具体数据结构的组合。
- 为什么持久化、过期删除、淘汰要分开讲？
  因为它们分别在回答“怎么保数据”“怎么删失效数据”“内存不够时怎么取舍”三个不同问题。
- 这题和后面几题是什么关系？
  这题是总框架，后面的性能、数据结构、持久化、过期删除和淘汰，都是它的展开。

## 3.2 Redis 为什么快？

### 30秒结论

- Redis 快的核心原因是基于内存、单线程模型避免了大量锁竞争、数据结构高效，再加上网络 IO 模型和实现细节优化。

### 3分钟回答

- 第一，Redis 主要操作内存，少了磁盘随机 IO，这是性能基础。
- 第二，它核心命令执行是单线程，不需要像多线程共享内存程序那样频繁加锁。
- 第三，Redis 内部数据结构针对常见场景做了优化，比如哈希表、跳表、压缩结构等。
- 第四，它的网络模型也比较高效，单线程处理命令并不代表只能处理很低并发。
- 所以 Redis 快不是因为它只用单线程，而是内存、数据结构和执行模型一起作用的结果。

### 追问答案

- Redis 单线程为什么还能这么快？
  因为瓶颈主要不在 CPU 算力，而在内存访问和网络处理，单线程反而减少了上下文切换和锁开销。
- Redis 现在完全是单线程吗？
  核心命令执行仍以单线程模型为主，但新版本在网络读写等方面已经有多线程优化。
- Redis 快是不是就一定适合所有场景？
  不是，超大 Value、复杂聚合、重计算和强事务场景未必适合放 Redis。

---

## 3.3 Redis 常见数据结构有哪些？分别适合什么场景？

### 30秒结论

- Redis 常见数据结构有 `String`、`Hash`、`List`、`Set`、`ZSet`、`Bitmap`、`HyperLogLog` 和 `Stream`。面试里关键不是背名字，而是说清楚各自适合什么业务场景。

### 3分钟回答

- `String` 最通用，适合缓存对象、计数器、分布式锁和简单 KV。
- `Hash` 适合存对象字段，比如用户信息、配置项，能按字段读写。
- `List` 常用于消息队列、时间线、任务列表这类有顺序的数据。
- `Set` 适合去重、标签、共同好友、黑名单等无序不重复场景。
- `ZSet` 适合排行榜、延迟队列、按分值排序的数据。
- `Bitmap` 适合签到、在线状态、用户活跃标记；`HyperLogLog` 适合近似去重计数；`Stream` 更适合消息流场景。
- 这题真正重要的不是背全，而是能把“结构特点”和“典型场景”稳定对应起来。

### 追问答案

- 为什么排行榜常用 `ZSet`？
  因为它天然支持按分值排序，还能高效取 TopN。
- `Hash` 和 `String` 缓存对象怎么选？
  如果更关注整体读写简单，常用 `String`；如果经常只更新对象部分字段，`Hash` 更合适。
- `Set` 和 `List` 的核心区别是什么？
  `Set` 无序且去重，`List` 有序且允许重复。

---

## 3.4 Redis 为什么不适合存超大 Value？

### 30秒结论

- 超大 Value 的问题不只是占内存，而是会放大网络传输、序列化、主从复制、持久化和阻塞风险。

### 3分钟回答

- 一个超大 Value 读写时会带来更长的网络传输时间，RT 很容易被拉高。
- 主从复制和持久化时，这类大对象也会放大同步和刷盘成本。
- 如果频繁修改，还可能让单线程事件循环长时间处理一个请求，影响其他请求延迟。
- 所以工程上更推荐把大对象拆分成更细粒度的数据结构，而不是把一大坨 JSON 全塞进一个 Key。
- 这题本质上是在提醒：Redis 虽然快，但并不意味着任何数据形态都适合硬塞进去。

### 追问答案

- 超大 Value 最大的风险是什么？
  是会同时放大 RT、复制、持久化和阻塞风险，而不是单纯多占一些内存。
- 怎么判断 Value 算不算太大？
  没有绝对统一值，但如果已经明显影响 RT、复制或网络，就说明需要拆分了。
- 拆分大对象时要注意什么？
  要兼顾读取成本、字段访问模式和 Key 数量，不是拆得越碎越好。

---

## 3.5 Redis 持久化有哪些方式？怎么选？

### 30秒结论

- Redis 常见持久化方式有 `RDB` 和 `AOF`。`RDB` 更适合快照恢复，`AOF` 更适合降低数据丢失窗口，工程上很多场景会组合使用。

### 3分钟回答

- `RDB` 本质是定时生成某个时间点的数据快照，恢复速度快，文件也更紧凑，但两次快照之间的数据可能丢失。
- `AOF` 是把写命令按追加日志方式记录下来，重启时重放恢复数据，数据更完整，但文件更大、恢复通常更慢。
- 如果业务更看重恢复速度，可以偏 `RDB`；如果更看重数据完整性，可以偏 `AOF`。
- 实际工程里常见是两者同时开启，用 `RDB` 做快速恢复，用 `AOF` 缩小丢失窗口。
- 这题重点是选型思路，不只是背两种持久化的定义。

### 追问答案

- `RDB` 最大的优点是什么？
  文件紧凑、恢复速度快，适合做冷备和快速重启恢复。
- `AOF` 最大的代价是什么？
  文件更大，重写和恢复成本通常更高。
- 持久化是不是开得越全越好？
  不是，持久化越重，对性能和磁盘的影响越大，要看业务容忍度和恢复目标。

---

## 3.6 RDB 和 AOF 的区别怎么答？

### 30秒结论

- `RDB` 是数据快照，`AOF` 是命令日志。前者恢复快、丢失窗口更大；后者数据更完整、恢复通常更慢。

### 3分钟回答

- `RDB` 关注的是某个时间点的数据状态，所以适合做备份和快速恢复。
- `AOF` 关注的是写操作过程，把命令一条条记下来，所以更接近完整变更历史。
- 两者的核心区别不是文件格式，而是一个记录结果，一个记录过程。
- 工程上常见做法是结合使用，而不是只在两者之间做绝对二选一。

### 追问答案

- 为什么 `RDB` 恢复通常更快？
  因为它直接加载快照，不需要逐条重放命令。
- 为什么 `AOF` 数据通常更完整？
  因为它记录的是持续写入的命令日志，丢失窗口通常更小。
- 面试里怎么一句话区分？
  `RDB` 记结果，`AOF` 记过程。

---

## 3.7 AOF 重写是什么？为什么需要它？

### 30秒结论

- AOF 重写的核心是把冗余命令压缩成恢复当前数据所需的最小命令集合，避免 AOF 文件无限膨胀。

### 3分钟回答

- 如果 Redis 一直把每次写命令都原样追加到 AOF，文件会越来越大。
- 但恢复当前数据状态并不需要历史上所有命令，比如一个 Key 被改了很多次，最终只需要最后有效状态。
- AOF 重写就是把这些冗余历史命令折叠，生成新的更精简的 AOF 文件。
- 它不是简单压缩文本，而是以当前内存数据为基础重新生成恢复命令。
- 这题本质上是在回答：为什么 AOF 能保留过程，但又不至于无限膨胀。

### 追问答案

- AOF 重写会不会阻塞业务？
  Redis 会尽量通过后台子进程做重写，降低对主线程影响，但仍然会消耗 CPU 和内存资源。
- 为什么不直接一直追加就行？
  因为文件会越来越大，恢复时间也会越来越长。
- AOF 重写和 RDB 快照有什么相似点？
  都会基于当前某个时间点的数据状态生成新内容，但目标和格式不同。

---

## 3.8 Redis 过期删除策略怎么理解？

### 30秒结论

- Redis 不会单纯依赖某一种方式删过期 Key，而是惰性删除和定期删除结合使用，在内存占用和 CPU 开销之间做平衡。

### 3分钟回答

- 惰性删除是访问 Key 时才检查是否过期，优点是省 CPU，缺点是过期 Key 可能长时间占内存。
- 定期删除是后台周期性抽样扫描过期 Key，优点是能主动清理一部分无效数据。
- 两者结合后，既不会每个 Key 到点就强行扫描，也不会让所有过期 Key 永远留在内存里。
- 但要注意，过期删除和内存淘汰不是一回事，前者是删无效 Key，后者是在内存不够时决定先淘汰谁。
- 这题最容易被答混，所以一定要把“删除已过期 Key”和“淘汰仍有效 Key”分清。

### 追问答案

- Redis 为什么不做定时删除？
  因为给每个 Key 单独维护精确删除调度，成本太高。
- 惰性删除有什么问题？
  如果某些过期 Key 很久都不被访问，就可能一直占内存。
- 过期删除和淘汰策略有什么区别？
  过期删除针对已经失效的 Key，淘汰策略针对内存不够时仍然有效的 Key。

---

## 3.9 Redis 常见淘汰策略有哪些？它们的区别和使用场景是什么？

### 30秒结论

- Redis 常见淘汰策略可以分成 3 类：`noeviction`、随机/TTL 类、`LRU/LFU` 类。它们的核心区别，在于“内存满时优先淘汰谁”；怎么选，要看业务是更关注命中率、热点稳定性，还是更依赖过期时间管理。

### 3分钟回答

- 先按大类分，Redis 常见淘汰策略可以理解成：
- 第一类是 `noeviction`，也就是内存满了不淘汰数据，直接让写入报错。它适合“宁可写失败，也不希望旧数据被自动清掉”的场景。
- 第二类是随机或 TTL 优先类，比如随机淘汰、优先淘汰更快过期的 Key。这类策略实现简单，但通常不如热点感知型策略稳定。
- 第三类是 `LRU` 和 `LFU`。`LRU` 更关注“最近有没有被访问过”，`LFU` 更关注“访问频率高不高”。
- 如果热点变化快，比如首页推荐、短期热点内容，`LRU` 往往更直观；如果热点长期稳定，比如配置项、固定热门商品、稳定高频数据，`LFU` 通常更合适。
- TTL 优先类策略更适合本身大量 Key 就依赖过期时间管理的业务，比如大量短生命周期缓存。
- `allkeys-*` 和 `volatile-*` 的差别也很重要：前者是在所有 Key 里淘汰，后者只在设置了过期时间的 Key 里淘汰。所以如果很多 Key 根本没设过期时间，`volatile-*` 的效果就会受限。
- 真正工程选型不能只背策略名，要结合命中率、热点分布、TTL 设计、业务容错能力和容量规划一起看。
- 如果回到更上层的系统设计视角，这题又会和第 `1.9` 题的“容量治理”连起来理解。

### 追问答案

- `LRU` 和 `LFU` 怎么选？
  热点变化快时更偏 `LRU`，热点长期稳定时更偏 `LFU`。
- `allkeys` 和 `volatile` 的核心区别是什么？
  `allkeys` 是所有 Key 都可能被淘汰，`volatile` 只淘汰设置了过期时间的 Key。
- `noeviction` 是不是最安全？
  不是，它只是内存满了直接拒绝写入，对某些业务反而风险更高。
- 为什么随机淘汰通常不作为优先选择？
  因为它不关心热点，容易把真正有价值的数据提前清掉。
- 为什么淘汰策略和容量规划要一起看？
  因为很多所谓淘汰问题，本质上是容量规划没做好。

## 4.1 Redis 高可用、分布式能力与线上排查，一般从哪些主线理解？

### 30秒结论

- 更稳的回答顺序通常是 4 条线：高可用基础怎么做、分片扩容怎么做、分布式能力边界在哪、线上问题怎么排。

### 3分钟回答

- 第一条线是高可用基础，也就是 Redis 主从复制、哨兵这些能力分别解决什么问题，这会回答“Redis 挂了怎么办”和“高可用靠什么完成”。
- 第二条线是分片扩容，也就是 Cluster 和哈希槽怎么工作，这会回答“单机撑不住时怎么横向扩容”。
- 第三条线是分布式边界，也就是 Redis 分布式锁、Redis 作为消息队列能做什么、不能神化成什么。
- 第四条线是线上排障，也就是实例抖动、慢查询、热 Key、大 Key、主从延迟和数据库回源该怎么联动排查。
- 所以这一组不是单纯讲 Redis 高级特性，而是在回答“Redis 在线上怎么稳定运行、怎么扩、怎么排、哪些地方容易被用过头”。
- 后面的题，基本都可以挂在这 4 条线下面展开。

### 追问答案

- 为什么高可用和分片扩容要分开看？
  因为主从和哨兵主要解决可用性问题，Cluster 主要解决容量和吞吐上限问题。
- 为什么分布式锁和消息队列要单独讲边界？
  因为它们都属于 Redis 能做但容易被过度使用的能力，面试里光讲“能做”不够。
- 这题和后面几题是什么关系？
  这题是总框架，后面的主从、哨兵、Cluster、分布式锁、消息队列和排查，都是它的展开。

---

## 4.2 Redis 主从复制是怎么工作的？

### 30秒结论

- Redis 主从复制的核心是主节点负责写，从节点复制主节点数据用于读扩展和高可用，但复制天然存在延迟窗口。

### 3分钟回答

- 从节点启动后会向主节点发起同步，请求当前数据和后续写命令流。
- 主节点会把当前数据快照和后续增量命令同步给从节点。
- 从节点完成同步后，后续继续接收主节点写入流，保持尽量接近一致。
- 这样能分摊读流量，也为故障切换提供基础，但并不意味着主从绝对实时一致。

### 追问答案

- Redis 主从复制为什么会有延迟？
  因为主节点写入成功不代表从节点已经同步并处理完成。
- 从节点能不能写？
  一般不作为主写入口使用，否则容易引入一致性问题。
- 为什么主从复制不等于高可用已经完成？
  因为复制只是数据同步基础，故障发现和自动切换还需要哨兵或其他机制。

---

## 4.3 Redis 哨兵是什么？解决了什么问题？

### 30秒结论

- 哨兵本质上是 Redis 的高可用监控和故障转移机制，用来监控主从状态、发现主节点故障，并在必要时自动完成主从切换。

### 3分钟回答

- 单纯主从复制只能同步数据，但主节点挂了之后，谁来发现、谁来切换、谁来通知客户端，主从本身并不解决。
- 哨兵会持续监控主节点和从节点状态，多个哨兵协同判断节点是否真的故障。
- 一旦确认主节点不可用，就会从从节点中选一个新的主节点，并通知客户端或上层中间件更新连接信息。
- 所以哨兵解决的是高可用治理问题，不是为了提升缓存读写性能。

### 追问答案

- 为什么需要多个哨兵？
  因为单个哨兵自己也可能故障或误判，需要通过投票和协同降低错误切换风险。
- 哨兵能解决数据丢失吗？
  不能完全解决，它主要解决故障发现和自动切换，复制延迟导致的丢失窗口仍可能存在。
- 哨兵和 Redis Cluster 有什么区别？
  哨兵更偏主从高可用，Cluster 同时解决数据分片和高可用。

---

## 4.4 Redis Cluster 是什么？适合什么场景？

### 30秒结论

- Redis Cluster 是 Redis 的分片集群方案，核心解决单机容量和吞吐瓶颈，同时提供一定高可用能力，适合数据量和流量都上到单机撑不住的场景。

### 3分钟回答

- 单机 Redis 的容量和性能总有上限，主从也只是复制，不解决单点容量问题。
- Cluster 把数据按哈希槽分散到多个节点上，每个节点负责一部分数据。
- 这样可以横向扩容容量和吞吐，同时每个主节点还能挂从节点做高可用。
- 它的代价是运维复杂度更高，一些多 Key 操作、事务和脚本能力会受到分片限制。

### 追问答案

- Cluster 的核心价值是什么？
  核心价值是分片扩容，把单机 Redis 撑不住的数据量和流量拆散到多个节点。
- 为什么多 Key 操作在 Cluster 下更麻烦？
  因为不同 Key 可能落在不同槽位和节点上，跨节点操作天然更复杂。
- 什么场景不一定需要上 Cluster？
  数据量和流量还远没到单机瓶颈、但更看重简单运维和一致性控制的场景。

---

## 4.5 Redis Cluster 的哈希槽怎么理解？

### 30秒结论

- Redis Cluster 用固定数量的哈希槽做数据分片。Key 先映射到槽，再由槽映射到具体节点，这样扩容和迁移更容易管理。

### 3分钟回答

- Cluster 并不是直接把 Key 简单平均分给节点，而是先定义一组固定槽位。
- 每个节点负责一部分槽，Key 通过哈希算法落到某个槽，再由该槽所在节点处理。
- 扩容或迁移时，本质上是把部分槽从一个节点迁到另一个节点，而不是全量重算所有 Key 的归属。
- 这让集群管理和重平衡比直接按节点分 Key 更可控。

### 追问答案

- 为什么要先分槽再分节点？
  因为这样扩容时迁移的是槽，不是所有 Key 的映射规则，更容易做重平衡。
- 哈希槽能解决数据倾斜吗？
  能降低一部分管理复杂度，但如果业务 Key 分布本身极度不均，仍可能出现热点。
- 什么是哈希标签？
  就是通过特定写法让一组 Key 落到同一槽，方便做某些多 Key 操作。

---

## 4.6 Redis 分布式锁怎么实现？要注意什么？

### 30秒结论

- Redis 分布式锁常见做法是 `SET key value NX EX`，核心不是只会加锁，而是要保证加锁、过期、解锁和误删保护都设计完整；同时要明确它只能做轻量并发控制，不能被神化成万能事务方案。

### 3分钟回答

- 最基本思路是用 `SET NX EX` 保证只有一个客户端能成功创建锁，并给锁设置过期时间，避免死锁。
- 每个客户端加锁时要写入唯一标识，解锁时校验这个标识，避免把别人加的锁删掉。
- 这类锁适合做轻量并发控制，但不能神化成万能分布式事务工具。
- 工程上要特别注意锁超时、业务执行时间、续期和误删问题。
- 这题除了讲“怎么做”，还要讲“什么时候别过度使用”，否则很容易把业务原子性问题错误地甩给 Redis 锁。

### 追问答案

- 为什么解锁时不能直接 `DEL`？
  因为锁可能已经过期并被别人重新拿到，直接删会误删别人的锁。
- Redis 锁能完全替代数据库事务吗？
  不能，它只能做分布式并发控制，不能替代业务原子性设计。
- 锁过期了但业务还没执行完怎么办？
  要么合理设置过期时间，要么配合续期机制，但也要控制复杂度。

---

## 4.7 Redis 作为消息队列有什么优缺点？

### 30秒结论

- Redis 可以做轻量消息队列，但它不是专业 MQ。优点是简单、快，缺点是消息可靠性、堆积能力和复杂消费治理通常不如专业消息系统，所以更适合轻量异步，而不适合把所有消息场景都往上搬。

### 3分钟回答

- 用 `List`、`Pub/Sub` 或 `Stream` 都能实现消息传递能力，但能力层次不一样。
- 轻量异步通知、小规模任务分发可以用 Redis 做，但如果涉及强可靠投递、海量堆积、复杂消费组治理，Redis 往往不是最优解。
- 真正工程上要先看业务对可靠性、顺序性、回溯能力和消费确认的要求。
- 所以这题除了讲优点，也要明确边界：Redis 能做轻量消息能力，但不是专业 MQ 的等价替代品。

### 追问答案

- `Pub/Sub` 为什么不适合强可靠消息？
  因为订阅者不在线时消息就可能直接丢掉，不具备可靠堆积能力。
- 为什么 `Stream` 比 `List` 更像消息队列？
  因为它支持消费组、确认、消息 ID 等更接近 MQ 的能力。
- 什么场景适合 Redis 做消息队列？
  轻量异步通知、低复杂度任务分发、对消息堆积和可靠性要求没那么极端的场景。

---

## 4.8 Redis 线上排查一般看哪些指标？

### 30秒结论

- Redis 排查不能只看 QPS，更稳的顺序通常是：先看现象，再看 Redis 自身资源和慢查询，再看热 Key / 大 Key / 主从延迟，最后看数据库回源有没有被放大。

### 3分钟回答

- 第一步先看现象，是 RT 抖动、命中率下降、连接暴涨、慢查询增多，还是业务接口整体变慢。
- 第二步看 Redis 自身资源和执行状态，比如 QPS、RT、慢查询、内存使用率、碎片率、CPU、网络带宽和连接数。
- 第三步看数据层问题，比如大 Key、热 Key、过期 Key 比例、主从延迟、持久化状态。
- 第四步一定要联动下游数据库一起看，因为很多 Redis 问题最终会体现成回源暴增、数据库连接数上升和接口雪崩。
- 所以线上排查不是指标罗列，而是“现象 -> Redis 本身 -> 数据层热点 -> 数据库回源”的完整排障链路。

### 追问答案

- Redis 慢查询最常见意味着什么？
  通常意味着命令本身复杂、Value 太大、热点过高或实例资源紧张。
- 为什么要关注大 Key 和热 Key？
  因为这两类问题最容易同时拖慢 RT、复制和业务稳定性。
- 为什么看 Redis 还要看数据库？
  因为很多 Redis 问题最终会传导到数据库，比如命中率下降或实例故障导致回源暴增。

---

## 4.9 Redis 大 Key 和热 Key 怎么排查？

### 30秒结论

- 大 Key 和热 Key 都会拖垮 Redis，但根因不同：大 Key 更偏“单个数据太重”，热 Key 更偏“单个访问太集中”。排查时要同时看命令耗时、网络流量、主从复制、实例倾斜和业务热点分布。

### 3分钟回答

- 大 Key 常见问题是：单次读写耗时高、网络传输大、主从复制慢、持久化压力大。
- 热 Key 常见问题是：单个 Key QPS 过高、某个分片或实例明显更热、RT 抖动和 CPU 抖动集中在某一点。
- 排查大 Key，一般会看慢查询、内存占用异常、网络流量、主从复制延迟和对象大小分布。
- 排查热 Key，一般会看热点访问统计、实例 QPS 倾斜、命令分布、业务日志和监控埋点。
- 真正线上处理时，大 Key 往往要拆分数据结构，热 Key 往往要做本地缓存、请求合并、静态化或热点隔离。
- 所以这题要讲清楚：大 Key 和热 Key 都危险，但一个是“太大”，一个是“太热”，治理方向不一样。

### 追问答案

- 大 Key 最大风险是什么？
  会同时放大命令耗时、网络传输、复制和持久化成本。
- 热 Key 最大风险是什么？
  会让流量集中打到单个实例或单个分片，造成局部热点雪崩。
- 为什么两者经常一起看？
  因为它们都会直接影响 Redis RT、复制延迟和整体稳定性。

## 4.10 Redis 主从延迟怎么排查？

### 30秒结论

- Redis 主从延迟排查，不能只看“延迟大不大”，更稳的顺序通常是：先确认延迟现象，再看网络和主从同步状态，再看大 Key / 热 Key / 持久化压力，最后再看业务写入峰值。

### 3分钟回答

- 第一步先确认延迟是不是持续存在，还是在某个写入高峰、活动窗口、主从切换后短时放大。
- 第二步看主从同步本身有没有异常，比如复制链路、网络抖动、从节点负载、主节点输出缓冲压力。
- 第三步看是不是有大 Key、热点写入、批量操作、AOF 重写或 RDB 快照把复制链路拖慢。
- 第四步再看业务侧是不是刚好有写入暴增、批量刷新缓存、预热不当等场景。
- 真正排查时，主从延迟不能孤立看，它往往和网络、实例负载、数据形态、业务写入模式一起联动。
- 所以这题最关键的是排查顺序，而不是只回答“主从延迟会有数据不一致”。

### 追问答案

- 主从延迟为什么危险？
  因为会导致读到旧数据，也会影响故障切换时的数据完整性。
- 为什么大 Key 会放大主从延迟？
  因为复制链路同步大对象时，网络和处理成本都会明显增加。
- 排查主从延迟为什么还要看业务写入？
  因为很多复制问题本质上是业务写入模式把主节点压得太重了。
