## 高频易混点

- `volatile` 解决的是可见性和部分有序性，不等于“线程安全”。
- `synchronized` 和 `ReentrantLock` 都能做互斥，但可控能力、使用方式和适用场景不同。
- `CAS` 不是没有代价，它会有 ABA、自旋重试和高竞争下空转问题。
- `AQS` 不是一个具体锁，而是很多同步器共享的底层框架。
- `CountDownLatch`、`CyclicBarrier`、`Semaphore` 都是同步工具，但目标完全不同：等待结束、阶段对齐、限制并发数。
- 线程池参数不是越大越好，线程太多同样会把系统拖慢。

## 关键对比表

### `volatile`、`synchronized`、`ReentrantLock`、CAS

| 对比项         | `volatile`     | `synchronized` | `ReentrantLock`      | CAS               |
| ----------- | -------------- | -------------- | -------------------- | ----------------- |
| 主要解决        | 可见性、部分有序性      | 互斥、可见性、原子性     | 互斥、可见性、原子性，且更灵活      | 无锁更新下的原子性         |
| 能否解决复合操作原子性 | 否              | 是              | 是                    | 单次 CAS 可以，复杂逻辑要小心 |
| 典型优势        | 轻量、语义简单        | 语法直接、JVM 优化成熟  | 支持 `tryLock`、可中断、公平锁 | 避免重量级阻塞           |
| 典型问题        | 容易被误用为“万能线程安全” | 粒度不当会放大竞争      | 使用复杂，容易忘记释放          | ABA、自旋开销、高竞争空转    |

### `CountDownLatch`、`CyclicBarrier`、`Semaphore`

| 工具               | 核心作用         | 更像什么场景        |
| ---------------- | ------------ | ------------- |
| `CountDownLatch` | 等一组任务完成      | 主线程等多个子任务结束   |
| `CyclicBarrier`  | 一组线程到齐后再一起继续 | 多线程阶段性协同      |
| `Semaphore`      | 控制同时进入的线程数   | 限流、连接池、资源配额控制 |

### 线程池排查入口

| 现象    | 第一观察点       | 常见方向            |
| ----- | ----------- | --------------- |
| 队列堆积  | 队列长度、活跃线程数  | 线程太少、任务太慢、下游阻塞  |
| 拒绝增多  | 拒绝次数、流量峰值   | 参数不合理、降级缺失、容量不足 |
| CPU 高 | 线程状态、热点栈    | 线程空转、自旋、任务逻辑过重  |
| RT 抖动 | 队列等待时间、任务耗时 | 下游慢、线程池堆积、锁竞争   |

## 一页速记

- `volatile` 看可见性，锁看互斥，CAS 看无锁原子更新。
- `synchronized` 和 `ReentrantLock` 都能加锁，但一个偏简单，一个偏灵活。
- `AQS` 是并发工具底层骨架，不是单独给业务直接用的一把锁。
- `CountDownLatch` 等结束，`CyclicBarrier` 等对齐，`Semaphore` 控并发。
- 线程池本质上解决的是线程复用、资源控制和任务削峰，不是“线程越多越快”。
- 死锁先看线程栈，线程池问题先看队列和活跃线程，锁竞争先看热点临界区。

## 并发实战入口

- 看到 `线程安全问题`：
  先分清是可见性问题、原子性问题，还是临界区互斥没做好。
- 看到 `CPU 高`：
  先看线程状态和热点栈，再判断是空转、自旋、锁竞争还是线程池里的任务过重。
- 看到 `线程池堆积`：
  先看队列长度、活跃线程数和任务耗时，不要只盯线程数量。
- 看到 `锁竞争严重`：
  先找热点临界区，再判断是锁粒度太粗、持锁时间太长还是竞争模型不合理。

## Java 并发实战案例

### 案例 1：`i++` 为什么会丢数据

- 现象：
  多线程并发累加，最后结果明显小于预期。
- 排查思路：
  先确认这是不是可见性问题，再拆解 `i++` 的读、改、写过程，确认它不是原子操作。
- 常见结论：
  这类问题通常不是“线程没看到最新值”，而是多个线程同时进入了复合操作，导致更新覆盖。

### 案例 2：线程池队列一直堆积

- 现象：
  请求越来越慢，线程池活跃线程数不高，但队列长度持续上涨。
- 排查思路：
  先看任务执行时长和下游依赖，再看是不是队列过大把问题掩盖了，或者核心线程数明显偏小。
- 常见结论：
  线程池问题经常不是“线程数不够”这么简单，而是任务本身太慢、阻塞太多，或者参数配置让问题被延后暴露。

### 案例 3：锁竞争严重导致 RT 抖动

- 现象：
  接口整体吞吐不低，但 P99 很差，线程大量阻塞在某个临界区。
- 排查思路：
  先定位热点锁和持锁代码，再判断是锁范围过大、持锁逻辑过重，还是单个热点资源竞争过于集中。
- 常见结论：
  很多并发性能问题最后不是“锁太差”，而是“锁用得不合理”。

## 1.1 什么叫线程安全？Java 并发里最常见的问题有哪些？

### 核心讲解

- 线程安全可以理解成：多个线程并发访问同一份共享数据或共享对象时，程序结果仍然正确、可预期。
- Java 并发里最常见的问题，通常可以先收成三类：可见性问题、原子性问题和有序性问题。
- 可见性问题是一个线程改了值，另一个线程看不到；原子性问题是复合操作被并发打断；有序性问题是指令重排让执行结果超出预期。
- 后面的 `volatile`、锁、CAS，其实都是在分别解决这几类问题，只是边界不同。

### 深入理解

- 这一题是第一模块最应该先建立的总框架。很多人一上来就背 `volatile`、`synchronized`、CAS，但如果不知道它们分别在补哪一类并发漏洞，后面就容易全混在一起。
- 学 Java 并发时，最稳的主线不是“先记工具”，而是先分清问题类型，再去看工具。比如状态标志位不生效，多半先想到可见性；`i++` 丢数据，多半先想到原子性；双检锁失效，多半先想到有序性。
- 这题学透以后，后面的每道题都更容易归位：`volatile` 偏可见性和部分有序性，锁偏互斥和复合操作原子性，CAS 偏单变量原子更新。
- 所以它不是一题额外补充题，而是整个第 1 模块的总入口。先把这题立住，后面才不会变成纯背概念。

### 追问答案

- 为什么很多并发 bug 平时不容易复现？
  因为线程调度、CPU 时序和竞争窗口都带随机性，问题往往只在特定并发时序下暴露。
- 什么叫“线程安全”里的“结果可预期”？
  就是无论线程切换顺序怎么变化，程序都不会出现脏数据、丢更新或违反业务约束的结果。
- `HashMap` 在多线程下不加保护算线程安全吗？
  不算，因为它对并发读写没有正确的可见性和结构一致性保证。

### 学习提示

- 复习第 1 模块时，先背这题，再去看 `volatile`、锁、CAS，会明显顺很多。
- 面试里这题不需要讲得很长，关键是把“问题分类 -> 工具映射”这条线讲出来。

## 1.2 线程和进程有什么区别？

### 核心讲解

- 进程更像资源分配单位，线程更像 CPU 调度单位。一个进程里可以有多个线程，共享进程内资源。
- 进程之间隔离更强，切换成本更高。
- 线程更轻量，但共享资源也意味着并发安全问题更多。

### 深入理解

- 这题本身不算深，但它是并发学习的入口。真正要理解的不是“进程大、线程小”这种口号，而是资源边界和调度边界的区别。
- 进程更像一层隔离容器，线程更像在这个容器里并发执行的多个执行流。线程轻量，切换更快，但也因为共享资源更多，才会引出后面所有并发安全问题。
- 所以这题的价值不是单独背下来，而是让你知道：Java 并发问题主要发生在线程共享资源这一层，不是在进程隔离这一层。
- 如果你能顺手补一句“线程更适合高并发，但更容易遇到可见性、互斥和原子性问题”，这题就和后面的主线接上了。

### 追问答案

- 为什么线程更适合高并发编程？
  因为切换和协作成本通常比进程低。

### 学习提示

- 这题不用讲太长，重点是给后面的并发问题搭底。
- 最好顺着这题直接过渡到“线程共享资源为什么会有并发安全问题”。

## 1.3 volatile 关键字怎么理解？

### 核心讲解

- volatile 主要解决共享变量的可见性问题，并通过内存屏障限制部分指令重排，但不保证复合操作原子性。
- 一个线程改了 volatile 变量，别的线程更容易及时看到。
- 它也能约束一定的有序性，比如单例双检锁里就常和它一起出现。

### 深入理解

- `volatile` 最容易被误解成“轻量级线程安全关键字”，其实它解决的是共享变量的可见性问题，并通过内存屏障限制部分重排序。
- 真正要区分的是：一个线程写入 `volatile` 变量后，别的线程更容易看到最新值，但这并不等于多个线程对同一个变量做复合操作时就安全了。
- 所以它非常适合做状态标志位、配置开关、双检锁里的引用发布，但不适合拿来保护 `i++`、复合校验后再更新这类逻辑。
- 如果你能把 `volatile` 和后面的 `synchronized`、CAS 做出清晰边界，这题就不会再只是定义题。

### 追问答案

- 为什么双检锁里常加 volatile？
  是为了防止对象初始化过程被重排。

### 学习提示

- 这题一定要和“原子性”分开背，不然最容易答错。
- 复习时最好直接配一个“停止标志位”和一个 `i++` 丢数据的对照例子。

## 1.4 synchronized 怎么理解？

### 核心讲解

- synchronized 本质上是一种内置监视器锁，既能实现互斥，也能保证可见性和有序性。
- 它可以加在实例方法、静态方法和代码块上。
- 加锁后同一时间只有持有锁的线程能进入临界区。

### 深入理解

- `synchronized` 的重点不只是“加锁”，而是它同时提供了互斥、可见性和一定的有序性保证。所以它比很多人想的更完整，不只是“把代码包起来”。
- 这题最容易出现两个误区：一个是把它想得过于沉重，另一个是把它想得过于简单。现代 JVM 对它做了很多优化，但如果锁粒度不合理、持锁时间过长，竞争依然会很明显。
- 工程上理解这题，要关注“锁住的到底是什么对象”“锁范围有多大”“临界区里放了什么操作”。很多性能问题不是 `synchronized` 本身的问题，而是加锁方式不合理。
- 这题学透以后，再看 `ReentrantLock` 和 AQS 会更顺，因为你已经先理解了最基本的互斥模型。

### 追问答案

- synchronized 是不是很重？
  老印象里偏重，但现代 JVM 已做了大量优化。

### 学习提示

- 这题最适合和 `ReentrantLock` 对比着学，不要单独背。
- 面试里最好补一句：问题往往不在关键字本身，而在锁粒度和持锁时间。

## 1.5 synchronized 和 ReentrantLock 有什么区别？

### 核心讲解

- synchronized 是语言级内置锁，写法简单；ReentrantLock 是显式锁，功能更丰富，比如可中断、可轮询、公平锁等。
- 如果只需要简单互斥，synchronized 往往更直接。
- 如果你需要更细粒度控制，比如 `tryLock`、超时等待、公平性，ReentrantLock 更灵活。

### 深入理解

- 这题真正想考的是“你会不会按场景选工具”。`synchronized` 偏语言级、简单直接；`ReentrantLock` 偏框架级、控制力更强。
- 如果业务只是普通互斥，`synchronized` 往往够用，而且代码更清爽；但如果你需要 `tryLock`、可中断、公平性、条件队列等能力，显式锁就会更合适。
- 这题不要答成“哪个更高级”，而要答成“哪个场景更合适”。这比单纯背功能点更有工程感。
- 最稳的表达方式通常是：先说共同点，再说能力差异，最后补使用复杂度和典型场景。

### 追问答案

- 什么时候更适合 ReentrantLock？
  需要高级控制能力时。

### 学习提示

- 复习时别背成功能列表，直接按“简单互斥”和“需要高级控制能力”两类场景去分。
- 这题讲好之后，后面的可重入锁、公平锁会非常自然。

## 1.6 什么是 CAS？有什么问题？

### 核心讲解

- CAS 可以理解成“比较并交换”，它是一种无锁原子更新思路。常见问题有 ABA、自旋开销和高竞争下失败重试。
- CAS 会先比较某个值是不是预期值，是的话才更新。
- 这样很多简单并发更新不用加重锁。

### 深入理解

- CAS 的吸引力在于“看起来不用锁”，但真正要理解的是：它是用比较和重试来换掉阻塞等待，所以不是没有代价，而是把代价从阻塞换成了自旋和失败重试。
- 这题最关键的两个记忆点，一个是 ABA，一个是高竞争下空转。也就是说，CAS 适合低冲突、短临界区、简单原子更新，但不是所有并发写场景都适合用它硬顶。
- 工程上如果竞争非常激烈，CAS 可能反而更浪费 CPU，因为线程一直在重试。
- 所以这题最好的答法不是“CAS 很高性能”，而是“CAS 避免了重量级阻塞，但适用场景有边界”。

### 追问答案

- ABA 怎么理解？
  值看起来没变，但中间其实已经被改过又改回来了。

### 学习提示

- 这题最好和锁做对比记忆：一个偏阻塞互斥，一个偏无锁重试。
- 如果你能顺手补一句“低冲突更适合 CAS”，理解就很到位了。

## 1.7 volatile、synchronized / ReentrantLock、CAS 分别解决什么问题？使用上怎么选？

### 核心讲解

- 这三类工具解决的不是同一层问题。`volatile` 主要解决可见性和部分有序性，`synchronized` / `ReentrantLock` 主要解决临界区互斥和复合操作原子性，CAS 主要解决简单变量的无锁原子更新。
- 如果只是让一个线程写入的状态及时被别的线程看到，比如停止标志位、配置开关，优先考虑 `volatile`。
- 如果要保护一整段临界区逻辑，尤其是“先判断再更新”这类复合操作，通常要用 `synchronized` 或 `ReentrantLock` 这类锁。
- 如果只是对单个共享变量做轻量级原子更新，比如计数器、状态位切换，CAS 或原子类通常更合适。
- 所以更稳的选型思路不是问“哪个更高级”，而是先问：我要解决的是可见性、互斥，还是无锁原子更新。

### 深入理解

- 这题的价值在于把第一模块前面的几题真正串起来。很多并发问题不是因为不会某个关键字，而是因为把不同工具解决的问题混成了一件事。
- `volatile` 最容易被误用成“轻量线程安全”，但它并不能保护复合操作；锁最容易被嫌“太重”，但很多业务临界区本来就应该老老实实加锁；CAS 最容易被神化成“高性能无锁”，但高竞争下它也会有明显重试成本。
- 所以这三者没有绝对谁更好，只有适配场景不同。状态同步偏 `volatile`，临界区保护偏锁，单变量原子更新偏 CAS。
- 如果你能把它们分别对应到“状态标志位”“订单扣减临界区”“原子计数器”这三类场景，这题就真正学透了。

### 追问答案

- 为什么 `volatile` 不能替代锁？
  因为它不能保证复合操作原子性，也不能让多个线程互斥进入临界区。
- 为什么 CAS 不能替代所有锁？
  因为它更适合简单变量更新，复杂业务逻辑往往不是一次比较交换就能解决的。
- 什么时候比起 `synchronized` 更适合 `ReentrantLock`？
  当你需要 `tryLock`、可中断、公平锁或多个条件队列时，显式锁会更灵活。

### 学习提示

- 复习这题时，不要按“语法”分，而要按“解决的问题”分：可见性、互斥、无锁原子更新。
- 面试里最稳的答法，是每类工具都配一个典型业务场景，而不是只背概念定义。

## 2.1 AQS 怎么理解？

### 核心讲解

- AQS 可以理解成 Java 并发包里很多同步器的基础框架，它用一个状态值加等待队列来实现锁和同步器。
- ReentrantLock、CountDownLatch、Semaphore 等很多工具都建立在 AQS 思路上。

### 深入理解

- AQS 这题最容易被答成“底层源码很复杂”，但学习时其实不需要一开始就扎进源码。先抓住一个主线就够：它用一个共享状态值和等待队列，统一抽象了很多锁和同步器的实现思路。
- 这也是为什么 `ReentrantLock`、`CountDownLatch`、`Semaphore` 虽然表面功能不同，底层却能共享一套框架思想。
- 这题的意义不在于让你手写 AQS，而在于帮助你理解“为什么 Java 并发包里这些工具看起来不同，底层逻辑却有共性”。
- 如果你能把 AQS 讲成“并发工具底层骨架”，而不是一堆源码细节，这题就已经很稳了。

### 追问答案

- 为什么 AQS 重要？
  因为它是很多并发工具底层共同的设计思想。

### 学习提示

- 这题不用一开始就深挖源码，先理解“状态值 + 等待队列”这个抽象骨架。

## 2.2 什么是可重入锁？

### 核心讲解

- 可重入锁指的是同一个线程已经拿到锁后，可以再次进入同一把锁保护的代码，而不会把自己锁死。
- synchronized 和 ReentrantLock 都是可重入的。
- 这样设计能让递归调用或内部再次调用受同一把锁保护的方法时更自然。

### 深入理解

- 可重入锁真正解决的是“同一个线程重复进入同一把锁时不要把自己锁死”。如果没有这个特性，递归调用、内部方法层层嵌套调用就会非常难写。
- 这题不要只背“能重复拿锁”这个定义，更要理解它为什么是并发工具里的默认能力。因为真实业务里，方法调用链往往不是单层的，一个已经持锁的方法再调另一个也需要同锁保护的方法，是很常见的。
- 所以可重入锁更像一种“让锁行为贴近真实代码结构”的设计，而不是额外加分项。`synchronized` 和 `ReentrantLock` 都支持它，也说明这不是某个实现的偶然特性。
- 学这题时，最好把它和后面的公平锁分开：可重入锁解决的是“同线程再次进入”的问题，公平锁解决的是“不同线程排队拿锁”的问题。

### 追问答案

- 如果锁不可重入会怎样？
  同一个线程再次进入时可能把自己阻塞住。

### 学习提示

- 最好直接拿“持锁方法里再调用另一个持锁方法”这个例子来记，比抽象定义更牢。
- 复习时要刻意和公平锁区分，不然这两个概念很容易混。

## 2.3 公平锁和非公平锁怎么理解？

### 核心讲解

- 公平锁更强调先来先得，非公平锁更强调吞吐和抢占效率。
- 公平锁可以减少线程长期饥饿，但调度成本通常更高。
- 非公平锁吞吐往往更好，所以默认实现里更常见。

### 深入理解

- 这题的关键不是“公平听起来更好”，而是先来先得往往是以吞吐下降为代价换来的。公平锁要更严格地照顾等待队列里的顺序，所以调度开销通常更高。
- 非公平锁更常见，不是因为它“不讲道理”，而是因为很多业务目标首先是整体吞吐和响应时间，而不是绝对排队公平。只要没有明显饥饿问题，默认偏非公平往往更划算。
- 所以这题真正考的是取舍意识：系统到底更看重“谁先来先服务”，还是“整体跑得更快”。一旦能从这个角度说，就比单背定义强很多。
- 学习时也要注意，公平锁不是万能的“更优选项”。如果没有明确公平性诉求，盲目追公平反而可能让系统更慢。

### 追问答案

- 面试里怎么回答选型？
  一般先说默认偏非公平，因为整体吞吐更好，只有明确需要公平性时才考虑公平锁。

### 学习提示

- 这题最适合配“吞吐优先”和“排队公平优先”两个场景来记。
- 面试里最好补一句：默认偏非公平，不是因为公平没用，而是因为大多数系统先看吞吐。

## 2.4 CountDownLatch、CyclicBarrier、Semaphore 怎么理解？

### 核心讲解

- CountDownLatch 更适合等待一组任务完成，CyclicBarrier 更适合一组线程彼此等待，Semaphore 更适合控制并发访问数量。
- 这几个工具都很高频，重点是分清它们解决的是“等待完成”“阶段对齐”还是“限量进入”。

### 深入理解

- 这题最容易混，因为三个工具都在“协调线程”，但协调方式完全不同。`CountDownLatch` 更像“等大家干完”，`CyclicBarrier` 更像“等大家到齐再一起走”，`Semaphore` 更像“同一时间只允许这么多人进去”。
- 真正学透这题，不是死背名字，而是能把它们和业务动作挂钩。比如主线程等多个子任务结果，用 `CountDownLatch`；多线程分阶段计算、每一轮都要对齐，用 `CyclicBarrier`；控制连接池或外部资源并发访问量，用 `Semaphore`。
- 这题也是模块里最适合场景化记忆的一题。因为它们的区别主要不是底层实现差异，而是线程协作目标不同。
- 如果能把这三者讲清楚，后面看到各种“多线程协同”题时，就更容易快速归类。

### 追问答案

- 为什么 Semaphore 常被叫做限流工具？
  因为它能限制同时进入的线程数。

### 学习提示

- 这题建议直接背三句话：等结束、等到齐、控名额。
- 只背定义很容易混，最好每个工具都配一个一句话业务例子。

## 2.5 ConcurrentHashMap 为什么适合并发场景？

### 核心讲解

- 因为它在保证并发安全的同时，尽量减少了全表锁带来的性能损失。
- 面试里重点不是背源码细节，而是理解它相比 `Hashtable` 和外部加大锁方案，在吞吐和并发度上更有优势。

### 深入理解

- 这题的核心不是背实现细节，而是理解它为什么比“整张表一把大锁”更适合高并发。它的设计目标就是在保证并发安全的同时，尽量缩小竞争范围，让更多线程可以并行读写。
- 所以学习时要把它和 `Hashtable`、`synchronizedMap` 放在一起比较。后两者更像“先保证安全，再说性能”，而 `ConcurrentHashMap` 更像“在安全前提下，尽量把并发度做高”。
- 这题也提醒你一个常见误区：并发容器不等于一切复合操作都自动安全。单次读写可以更稳，但“先判断再更新”这类复合逻辑仍然可能需要额外控制。
- 如果你能把它讲成“高并发共享数据访问场景下，比整张表加锁更平衡的方案”，这题就已经有工程味了。

### 追问答案

- ConcurrentHashMap 就一定完全无锁吗？
  不完全是，它是尽量减少锁竞争，而不是绝对不用锁。

### 学习提示

- 最好和 `Hashtable`、`synchronizedMap` 做对照记忆，不要单独背它自己。
- 这题面试里最容易加分的一句是：并发容器提高了并发度，但不代表复合业务操作天然安全。

## 3.1 线程池为什么重要？

### 核心讲解

- 线程池的核心价值，是复用线程、控制并发数量、限制资源消耗和统一任务调度。
- 如果每来一个任务就新建线程，开销会很大，也容易把系统打爆。
- 线程池本质上是在做线程复用和并发控制。

### 深入理解

- 线程池真正重要，不是因为“看起来专业”，而是因为线程本身就是昂贵资源。线程池把线程创建、销毁和并发数量控制统一收口，避免系统在高并发下无限制地起线程把自己拖垮。
- 所以这题不要只答“复用线程”，还要补一句“线程池同时在控制资源上限和任务承载方式”。这是它和“直接 new 线程”最大的差别。
- 工程上很多线上问题，本质上都是线程池参数和业务任务模型不匹配：任务太慢、队列太大、线程太少、拒绝策略不合理。也就是说，线程池不是可选优化，而是服务端稳态运行的重要基础设施。
- 如果能把它讲成“线程资源治理工具”，而不是单纯的性能优化工具，这题就会更有层次。

### 追问答案

- 线程池是不是越大越好？
  不是，太大容易带来上下文切换和资源争抢。

### 学习提示

- 复习这题时一定别只背“线程复用”，要把“资源控制”和“任务缓冲”也一起带上。
- 最好直接配一个“请求暴涨时为什么不能无限建线程”的场景。

## 3.2 ThreadPoolExecutor 核心参数有哪些？

### 核心讲解

- 核心参数通常有：核心线程数、最大线程数、存活时间、阻塞队列、线程工厂和拒绝策略。
- 这些参数一起决定了任务是直接执行、入队等待、扩线程还是被拒绝。

### 深入理解

- 这题最容易被答成参数列表，但真正高频的是“这些参数怎么一起工作”。线程池接到任务后，不是简单看某一个参数，而是按“核心线程 -> 队列 -> 最大线程 -> 拒绝策略”这条链依次决策。
- 所以这题真正要记住的是联动关系，而不是孤立定义。单独背 `corePoolSize` 没太大意义，必须知道它和队列、最大线程数一起决定系统在高峰下是排队、扩容还是拒绝。
- 这也是为什么线程池问题通常不会出在单个参数上，而是出在参数组合和任务特征不匹配。
- 如果你面试时能画出这条任务流转路径，这题就会比普通回答更稳。

### 追问答案

- 面试里最常被追问哪几个？
  核心线程数、最大线程数、队列和拒绝策略。

### 学习提示

- 这题最适合按“任务来了之后会发生什么”来讲，不要只背参数名。
- 如果能顺手讲出“核心线程满了先入队，队列满了再扩线程”，基本就稳了。

## 3.3 为什么线程池参数不能乱配？

### 核心讲解

- 因为线程池参数直接决定系统吞吐、等待时间、内存占用和拒绝行为。乱配很容易导致任务堆积、线程爆炸或请求雪崩。
- 队列太大，问题会被拖延但更难发现。
- 线程太多，会增加上下文切换和资源争抢。
- 线程太少，又可能让请求大量排队或超时。

### 深入理解

- 线程池参数不能乱配，是因为它们直接决定系统高峰时的退化方式。线程太少会让请求排队，线程太多会把 CPU 和上下文切换拖垮，队列太大会掩盖问题并放大 RT，队列太小又可能过早触发拒绝。
- 所以调线程池不是“找一个万能参数”，而是先看任务类型，再看机器资源，再看压测和监控反馈。CPU 密集和 IO 密集的线程池策略，本来就不该一样。
- 这题里最有工程感的一句通常是：线程池调参的目标不是让参数好看，而是让系统在正常流量和峰值流量下都能以可预期的方式退化。
- 如果你能把“线程池问题”讲成“容量治理问题”，这题层次会更高。

### 追问答案

- 更稳的调参思路是什么？
  结合任务类型、CPU/IO 特征、压测和监控数据来定。

### 学习提示

- 这题最适合配“队列堆积”“线程爆炸”“拒绝增多”三个现象来记。
- 面试里不要直接报参数经验值，先说任务类型和调参思路更稳。

## 3.4 线程池参数一般怎么配置？有哪些常见思路？

### 核心讲解

- 线程池参数配置没有万能模板，更稳的思路是先看任务类型，再看机器资源，再看系统目标，最后结合压测和监控去定。
- CPU 密集型任务，线程数通常更接近 CPU 核数；IO 密集型任务，因为大量时间在等待外部资源，线程数通常可以更高一些。
- `corePoolSize` 更像系统平时的常态承载能力，通常先按稳定流量和常驻任务量来定；`maximumPoolSize` 更像流量高峰时允许扩到多大，通常要结合机器资源上限和可接受的退化方式来定。
- 更常见的经验是：CPU 密集型任务，核心线程数通常更接近 CPU 核数，最大线程数也不会放大太多；IO 密集型任务，因为线程经常阻塞等待，核心线程数和最大线程数都可以适当放大，但必须靠压测验证。
- 队列不要一味配很大，队列太大容易掩盖问题并放大等待时间；队列太小又可能过早触发扩线程或拒绝。
- 拒绝策略也不是最后才想的补丁，而是配置线程池时就要一起考虑的止血方案。

### 深入理解

- 这题真正要讲的是“线程池怎么按场景落地”，而不是背几个经验数字。线程池参数配置本质上是在做容量治理，要回答系统在正常流量、高峰流量和异常流量下分别怎么表现。
- 所以最稳的顺序通常是：先分 CPU 密集还是 IO 密集，再判断你更看重吞吐、低延迟还是平稳退化，然后再去定核心线程数、最大线程数、队列和拒绝策略。
- 核心线程数一般回答“平时准备养多少工人”，最大线程数一般回答“高峰时最多再临时扩到多少”。如果 `corePoolSize` 太小，系统平时就容易排队；如果 `maximumPoolSize` 太大，高峰时虽然看起来抗住了，但也可能把 CPU、内存和上下文切换一起打爆。
- 如果任务是 CPU 密集型，线程太多通常只会增加切换开销；如果任务是 IO 密集型，线程适当放大往往更能把等待时间利用起来。也就是说，线程池配置一定要跟任务模型绑在一起，不能脱离业务谈。
- 工程上真正靠谱的配置，不是某个“网上推荐值”，而是“有分类思路 + 有压测验证 + 有运行期监控”。如果你能把这三个点讲出来，这题就很完整了。

### 追问答案

- CPU 密集型任务线程池一般怎么配？
  通常更接近 CPU 核数，避免线程过多带来的上下文切换。
- IO 密集型任务线程池为什么可以更大？
  因为线程大量时间在等待外部资源，适当增加线程能提高资源利用率。
- 核心线程数和最大线程数一般怎么分工？
  核心线程数负责日常稳定承载，最大线程数负责高峰期临时扩容，它们要和队列、拒绝策略一起看。
- 为什么最大线程数不能一味调大？
  因为它只是把压力换成更多线程竞争，可能导致 CPU 抖动、上下文切换变多，最后把系统整体拖慢。
- 为什么不建议把队列配得特别大？
  因为这会把问题延后暴露，导致堆积更深、RT 更差、排查更难。

### 学习提示

- 这题不要上来就报数字，先说任务类型和配置思路会更稳。
- 最好用一句话记住：先分任务类型，再定线程数，再看队列和拒绝策略。

## 3.5 线程池常见拒绝策略怎么理解？

### 核心讲解

- 拒绝策略本质上是在说“系统扛不住时，任务怎么办”。常见有抛异常、调用者执行、丢弃当前、丢弃最旧任务。
- 不同拒绝策略影响故障传播方式和用户感知方式。

### 深入理解

- 拒绝策略真正回答的是“系统已经扛不住了，接下来是快速失败、让上游变慢，还是容忍任务丢失”。所以这题本质上不是 API 题，而是系统保护题。
- `AbortPolicy` 更像及时报警，`CallerRunsPolicy` 更像反压调用方，`DiscardPolicy` 和 `DiscardOldestPolicy` 更像在可丢任务场景里做止血。
- 这题不能脱离业务讲。比如日志上报、埋点任务和核心下单请求，对拒绝的容忍度完全不一样。
- 如果你能把拒绝策略和“故障传播方式”联系起来，这题就会非常有工程味。

### 追问答案

- CallerRunsPolicy 为什么有时有价值？
  因为它能把压力反向传递给调用方。

### 学习提示

- 这题别背成枚举题，最好直接按“快速失败 / 反压 / 可丢弃”三类去记。
- 最好给 `CallerRunsPolicy` 配一个“把压力传回上游”的例子。

## 3.6 常见并发工具类有哪些？

### 核心讲解

- 常见高频并发工具类可以按三类记：线程协作工具、并发容器和任务队列。
- 线程协作工具有 `CountDownLatch`、`CyclicBarrier`、`Semaphore`；并发容器有 `ConcurrentHashMap`；任务队列常见有 `BlockingQueue`。
- 这题重点不是罗列得越多越好，而是先把工具按用途分组，再说代表场景。

### 深入理解

- 这题如果只答成“我知道很多类名”，学习价值其实不高。更好的答法是先分类，再说每类工具在系统里到底扮演什么角色。
- 比如线程协作工具负责让线程按规则配合，并发容器负责在多线程访问下兼顾安全和吞吐，阻塞队列负责承接任务生产和消费。
- 这样你就不会把这题答成杂项罗列，而是能形成一个小型工具地图。
- 这题也适合和第 2 模块联动看，因为很多并发工具其实就是同步器和容器在不同业务场景下的具体落地。

### 追问答案

- 为什么 Semaphore 常被叫做限流工具？
  因为它能限制同时进入的线程数。

### 学习提示

- 这题最好按“协作工具 / 容器 / 队列”三组来记，不要散着背类名。
- 面试里列 3 到 5 个高频工具并讲用途，比一口气报十几个名字更好。

## 3.7 Future 和 CompletableFuture 怎么理解？

### 核心讲解

- Future 更偏“拿结果”，CompletableFuture 更偏“把异步任务编排起来”。
- Future 解决的是异步结果获取问题，但编排能力弱。
- CompletableFuture 可以更自然地做串联、并联、回调和异常处理。

### 深入理解

- `Future` 和 `CompletableFuture` 的核心区别，不只是“新旧 API”，而是前者更像“异步提交后等结果”，后者更像“把异步步骤编排成一条流程”。
- 所以 `Future` 更适合简单异步任务，而 `CompletableFuture` 更适合并发查多个下游、结果汇总、异常兜底和链式回调。
- 这题最容易答偏成“CompletableFuture 更高级”，更稳的讲法是：如果只是拿结果，`Future` 就够；如果要做串联、并联、回调和异常组合，`CompletableFuture` 更自然。
- 学这题时最好直接配一个“并发查商品、库存、价格，再合并返回”的例子，会非常顺。

### 追问答案

- CompletableFuture 就等于线程池吗？
  不是，它是异步编排工具，底层执行通常仍然依赖线程池。

### 学习提示

- 这题最好配一个“并发调用多个下游后汇总返回”的业务例子。
- 记住一句话就够：`Future` 偏拿结果，`CompletableFuture` 偏异步编排。

## 4.1 什么是死锁？一般怎么排查？

### 核心讲解

- 死锁是多个线程互相等待对方持有的资源，导致谁也走不下去。
- 面试里回答死锁时，重点说清楚“互相等待资源”这个本质。
- 排查时通常结合线程栈、锁信息和代码加锁顺序来看。

### 深入理解

- 死锁这题最有价值的地方，不是定义本身，而是它代表了一类“线程彼此等着不动”的极端问题。线上看到这种问题时，最稳的入口通常不是猜代码，而是先看线程栈里有没有互相等待同一组锁资源。
- 所以排查死锁的主线一般是：先看线程状态和线程栈，再确认锁持有关系，最后回到代码里的加锁顺序。真正高频的问题也不是“死锁这个词”，而是为什么这些线程会走到互相等待这一步。
- 这题也很适合顺手补一句预防思路：统一加锁顺序、减少嵌套锁、缩小锁范围、避免长时间持锁。这样回答就会从“会排查”提升到“会治理”。
- 如果你能把它讲成“线程彼此卡住，先看线程栈，再看锁顺序”，这题就已经很像真实排障了。

### 追问答案

- 怎么预防死锁？
  统一加锁顺序、缩小锁范围、尽量避免嵌套锁和长时间持锁。

### 学习提示

- 这题最好按“线程卡死 -> 看线程栈 -> 找互相等待 -> 回代码”来记。
- 面试里别只背四个必要条件，最好再补一句真实排查顺序。

## 4.2 线程池问题一般怎么排查？

### 核心讲解

- 更稳的思路通常是：先看线程池活跃线程数、队列长度、拒绝次数，再看任务执行时间和调用链。
- 如果队列堆积严重，要判断是线程太少、任务太慢，还是下游卡住了。
- 如果拒绝次数高，就看参数配置、流量突增和降级策略是否缺失。

### 深入理解

- 线程池问题最怕“只盯线程数”。真实排查时，更稳的顺序通常是先看队列有没有堆、拒绝有没有涨、活跃线程是不是打满，再去判断是线程池本身配置不合理，还是任务逻辑太慢、下游调用卡住。
- 如果队列在涨但活跃线程不高，往往说明问题不只是“线程不够”，还可能是核心线程数偏小、任务执行阻塞严重，或者线程池配置把问题掩盖了。反过来，如果线程打满且拒绝增加，就要看是不是峰值流量超出设计容量。
- 这题要有工程感，就一定要补一句：线程池问题常常是业务任务模型的问题，不只是线程池参数本身的问题。
- 所以最稳的表达通常是“先看线程池指标，再看任务耗时，再看下游依赖”，这样排查链路就立住了。

### 追问答案

- 为什么不能只盯着线程数？
  因为真正的问题往往在任务本身太慢，线程数只是结果表现。

### 学习提示

- 这题最好按“队列、活跃线程、拒绝次数、任务耗时”四个点去背。
- 面试里最容易加分的一句是：线程池问题很多时候只是表象，真正慢的是任务本身或下游依赖。

## 4.3 锁竞争严重时一般怎么看？

### 核心讲解

- 更稳的思路通常是：先看线程是不是大量阻塞在同一把锁附近，再看热点临界区做了什么，最后判断是锁粒度太粗、持锁时间太长，还是热点资源竞争过于集中。
- 锁竞争严重不代表一定要“去锁化”，很多时候问题只是锁范围不合理，或者把耗时逻辑放进了临界区。
- 这题的核心不是先想怎么改，而是先定位到底是哪一把锁、哪一段代码、哪一种业务流量在竞争。

### 深入理解

- 锁竞争问题最容易被答成“那就换成 CAS 或无锁”，但真实线上排查不是先想技术升级，而是先找热点。很多时候真正的问题是锁粒度过大、临界区里塞了慢逻辑，或者所有流量都在抢同一份热点资源。
- 所以这题排查顺序通常是：先看线程栈和热点锁位置，再看锁保护的代码范围，最后判断是拆锁、缩锁、移出耗时逻辑，还是做业务侧分片和削峰。
- 这题真正考的是“你能不能把锁问题讲成性能瓶颈定位题”。因为锁竞争通常不是功能错误，而是吞吐和延迟问题。
- 如果你能补一句“先定位热点，再决定是否换方案”，这题就会比“直接上无锁”更成熟。

### 追问答案

- 面试里怎么体现工程感？
  不只说“换成无锁”，而是先定位热点，再结合业务特征做取舍。

### 学习提示

- 这题最好按“看线程阻塞点 -> 找热点锁 -> 看临界区内容 -> 决定优化方向”来记。
- 最容易答错的地方是直接说“换成无锁”，却没先解释为什么会竞争严重。
