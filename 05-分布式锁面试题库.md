## 1.1 什么是分布式锁？它解决什么问题？

### 核心讲解

- 分布式锁可以理解成“在多个进程、多个节点之间共享的一把锁”。它的核心作用，是在分布式环境下保护某段临界区代码或某个共享资源，避免被多个实例同时处理。
- 在单机程序里，我们可以用 `synchronized`、`ReentrantLock` 这类本地锁控制并发。
- 但在微服务或多实例部署场景下，请求可能同时落到不同机器上，本地锁已经管不到别的节点。
- 这时候就需要一个所有实例都能共同感知的协调机制，这就是分布式锁。
- 它常用来保护共享资源，比如同一份库存、同一个任务、同一个订单状态变更。
- 所以分布式锁本质上解决的是“多节点并发访问同一资源时，怎么保证同一时刻只有一个执行者进入临界区”。

### 深入理解

- 这题真正要理解的，不只是结论本身，还要理解“什么是分布式锁？它解决什么问题？”背后的原理、适用边界和工程取舍。
- 学习时建议把它和本模块前后题串起来看：它解决什么问题、在什么场景下最有价值、又会带来什么代价或限制。
- 如果这题偏原理，就重点理解数据流转、执行路径或状态变化；如果这题偏设计，就重点理解业务约束、方案选择和失败场景。
- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- 分布式锁是不是为了替代数据库事务？
  不是，它主要解决并发访问顺序，不直接替代事务和回滚能力。
- 分布式锁是不是只有 Redis 能做？
  不是，数据库、ZooKeeper 这类协调组件也可以做。
- 分布式锁最大价值是什么？
  是在多实例环境下保护共享资源，避免重复执行和并发冲突。

### 学习提示

- 学这题时要先想清楚“本地锁为什么不够用”。
- 真正的关键词不是“锁”，而是“跨节点共享的一致感知”。

## 1.2 为什么微服务和分布式系统里会需要分布式锁？

### 核心讲解

- 因为分布式环境下，同一份资源可能被多个服务实例同时操作，而本地锁只对单个进程有效。只要存在跨节点并发竞争，就可能需要分布式锁这类协调机制。
- 微服务部署后，一个服务通常会有多个实例，请求可能被负载均衡到不同节点。
- 如果这些实例都可能同时处理同一个资源，就会出现并发竞争问题。
- 比如多个实例同时扣减同一库存、重复执行同一个任务、重复发起同一业务操作。
- 这类问题在单机场景下靠本地锁还能管住，但一旦跨节点，本地锁就失效了。
- 所以分布式锁通常是分布式并发控制里比较直观的一种手段。

### 深入理解

- 这题真正要理解的，不只是结论本身，还要理解“为什么微服务和分布式系统里会需要分布式锁？”背后的原理、适用边界和工程取舍。

- 如果这题偏原理，就重点理解数据流转、执行路径或状态变化；如果这题偏设计，就重点理解业务约束、方案选择和失败场景。
- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- 是不是只要是微服务就一定要用分布式锁？
  不是，只有存在跨节点共享资源竞争时才有必要。
- 为什么单机锁在分布式环境里不够？
  因为它只能约束当前进程，看不到其他实例。
- 微服务里最常见的锁场景是什么？
  共享库存、全局任务、状态迁移和重复执行控制。

### 学习提示

- 学这题别把“微服务”和“必须上锁”画等号。
- 真正决定要不要上锁的是“有没有跨节点共享资源竞争”。

## 1.4 什么场景适合用分布式锁？什么场景不适合？

### 核心讲解

- 适合上分布式锁的，是跨节点共享资源竞争明显、需要串行化执行的场景；不适合的，是能靠幂等、唯一约束、状态机、消息削峰解决的问题，或者高频热点到锁本身会变瓶颈的场景。
- 适合用锁的场景通常有：多个实例争抢同一资源、同一个全局任务只能执行一次、某类状态迁移必须严格串行。
- 不适合用锁的场景也很多，比如只想防止重复提交，往往幂等更直接。
- 如果只是防止插入重复数据，数据库唯一约束通常更稳。
- 如果问题是瞬时高并发，很多时候消息队列削峰比大家一起抢锁更优。
- 还有一些超高频热点场景，如果所有请求都抢同一把锁，锁本身会先成为瓶颈。

- 这题真正要理解的，不只是结论本身，还要理解“什么场景适合用分布式锁？什么场景不适合？”背后的原理、适用边界和工程取舍。

## 1.5 分布式锁和本地锁有什么区别？

### 核心讲解

- 本地锁控制的是单进程内的并发，分布式锁控制的是多进程、多节点之间的并发。两者的核心区别，不在 API，而在约束范围和失败模式。
- 本地锁只存在于当前 JVM 或进程里，性能高、实现简单、失败模式相对单纯。
- 分布式锁需要借助外部共享存储或协调组件，让多个节点都能感知同一把锁的状态。
- 一旦跨节点，问题就复杂很多，比如网络超时、锁过期、误删、脑裂、时钟漂移、节点宕机。
- 所以分布式锁不是“更高级的锁”，而是“更复杂、代价更高的跨节点锁”。

### 深入理解

- 这题真正要理解的，不只是结论本身，还要理解“分布式锁和本地锁有什么区别？”背后的原理、适用边界和工程取舍。

- 如果这题偏原理，就重点理解数据流转、执行路径或状态变化；如果这题偏设计，就重点理解业务约束、方案选择和失败场景。
- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- 为什么本地锁不能直接用于微服务？
  因为不同实例上的锁彼此看不见。
- 分布式锁是不是一定比本地锁慢？
  通常是的，因为它需要网络通信和外部协调。
- 两者最本质的区别是什么？
  是约束范围不同，本地锁管进程内，分布式锁管跨节点。

### 学习提示

- 学这题时要抓住“失败模式升级”这个关键词。
- 很多分布式锁问题，本质上都是本地锁里不存在的新失败模式。

## 1.6 分布式锁和幂等、防重、事务有什么区别？

### 核心讲解

- 分布式锁解决的是“同一时间谁能执行”，幂等解决的是“重复执行结果是否可控”，防重更偏“别让重复请求再进来”，事务解决的是“一组操作要么都成功要么都失败”。它们相关，但不是一回事。
- 分布式锁更偏执行时序控制，强调某个资源同一时刻只给一个执行者处理。
- 幂等更偏结果控制，强调同一个请求重复执行多次，最终结果仍然只生效一次。
- 防重更偏入口拦截，强调别让同一个请求反复进入系统。
- 事务更偏状态一致性，强调一组操作整体成功或失败。
- 所以很多问题不是“上个锁就完了”，还要看结果能不能重复执行、状态能不能收敛。

### 深入理解

- 这题真正要理解的，不只是结论本身，还要理解“分布式锁和幂等、防重、事务有什么区别？”背后的原理、适用边界和工程取舍。
- 学习时建议把它和本模块前后题串起来看：它解决什么问题、在什么场景下最有价值、又会带来什么代价或限制。
- 如果这题偏原理，就重点理解数据流转、执行路径或状态变化；如果这题偏设计，就重点理解业务约束、方案选择和失败场景。
- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- 上了分布式锁还需要幂等吗？
  很多时候还是需要，因为锁不能覆盖所有异常时序和重复请求。
- 唯一键是不是也能替代一部分锁？
  是的，很多“防重复写入”问题用唯一约束更简单稳妥。
- 为什么说锁和事务不是一回事？
  锁控制的是并发进入，事务控制的是状态提交和回滚。

### 学习提示

- 这题特别适合建立“工具边界感”。
- 如果你能把这几个概念讲清楚，分布式锁就不会被你滥用。

## 2.1 Redis 分布式锁一般怎么实现？

### 核心讲解

- 最常见的 Redis 锁实现思路是：用 `SET key value NX EX` 原子加锁，value 放唯一标识，释放时先校验 value 再删除。核心是“只加自己的锁、只释放自己的锁”。
- Redis 锁通常会用一个固定 key 代表共享资源，比如订单、库存或任务。
- 加锁时用 `SET key uniqueValue NX EX ttl`，其中 `NX` 表示不存在才设置，`EX` 表示顺便带上过期时间，避免死锁。
- `uniqueValue` 一般是请求 ID、线程 ID 或实例 ID 组合，用来标识锁的拥有者。
- 业务执行完成后，释放锁时不能直接删，而是要先判断当前 value 还是不是自己，再删除。
- 所以 Redis 锁的标准思路其实就三步：原子加锁、唯一标识、原子释放。

### 追问答案

- 为什么加锁和过期时间要一起设置？
  因为分开做会有中间失败窗口，容易导致锁没有过期时间。
- 为什么一定要带唯一 value？
  因为后面释放锁时要区分“是不是自己加的锁”。
- Redis 锁最基础的正确性是什么？

## 2.2 为什么分布式锁要设置唯一 value？

### 核心讲解

- 因为锁 key 只能告诉你“现在有人持有这把锁”，但不能告诉你“这个人是不是我”。唯一 value 的作用，就是标识锁的拥有者，避免后续误删别人的锁。
- 如果加锁时只是写一个固定值，比如 `locked`，那释放锁时你无法判断这把锁是不是自己加的。
- 在高并发和超时场景下，锁可能已经过期并被别人重新拿到，这时候如果你还按老逻辑去删，就会把别人的锁删掉。
- 所以 value 一般要是全局唯一的，比如请求 ID、实例 ID、线程 ID 等组合。
- 这样释放时先比较 value，再决定删不删，才能避免误删。

### 追问答案

- key 已经代表资源了，为什么还不够？
  因为 key 只代表资源，不代表持有者。
- value 一般放什么？
  常见是请求唯一 ID，或者实例 ID 加线程 ID 之类的组合。
- 唯一 value 解决的核心问题是什么？
  是释放锁时的归属判断问题。

## 2.3 为什么释放锁时不能直接 delete？

### 核心讲解

- 因为锁可能已经过期并被别人重新拿到了，这时候如果你直接 `delete`，删掉的就可能不是自己的锁，而是别人的锁。
- 一个典型场景是：线程 A 加了锁，但业务执行太久，锁过期了。
- 过期后线程 B 抢到了同一个 key，成了新的锁持有者。
- 这时如果线程 A 执行完后直接 `delete key`，删掉的其实是线程 B 的锁。
- 所以正确做法通常是用 Lua 脚本把“比较 value”和“删除 key”做成原子操作，保证只有持有者自己能释放。

  因为它能把比较和删除放在 Redis 里一次性完成。

## 2.4 锁过期了但业务还没执行完怎么办？

### 核心讲解

- 这说明锁的 TTL 设计和业务执行时间不匹配。常见处理思路有两种：要么把 TTL 设计得更合理，要么引入续期机制，让锁在任务还活着时自动延长。
- 分布式锁通常会设置过期时间，避免持有者异常退出后锁永远不释放。
- 但如果业务执行时间超过 TTL，锁就会提前过期，其他线程可能重新拿到锁，造成并发执行。
- 一种思路是按业务预估时间设置更合理的 TTL，但这通常不够稳，因为执行时间可能波动很大。
- 更常见的工程方案是续期，也叫看门狗机制，只要持有锁的线程还活着，就周期性延长过期时间。
- 所以这题本质上是在回答：如何平衡“避免死锁”和“业务执行时间不稳定”。

- 续期机制核心解决什么问题？

## 2.5 Redis 分布式锁为什么会误删？

### 核心讲解

- 误删的根因通常是：原持有者执行太久导致锁过期，新的持有者已经拿到了锁，而原持有者结束后还按旧逻辑去删 key，结果把别人的锁删掉了。
- 误删场景几乎都是围绕“锁过期 + 持有者变化 + 非原子释放”展开的。
- 原线程以为这把锁还是自己的，但实际上锁已经过期并被别人重建了。
- 如果这时候释放逻辑没有校验唯一 value，或者比较和删除不是原子操作，就会出现误删。
- 所以误删问题本质上是在提醒我们：锁状态在分布式环境下是会变化的，不能用单机思维来处理。

## 2.8 Redlock 怎么理解？争议点在哪里？

- 单节点 Redis 锁最大的担忧之一，是 Redis 宕机或主从切换带来的锁语义风险。
- Redlock 试图通过在多个独立节点上分别加锁，并要求在限定时间内拿到多数节点的锁，来降低单点风险。

## 3.1 分布式锁、一致性与并发控制问题，一般从哪些主线理解？

## 3.2 分布式锁能保证强一致吗？

### 核心讲解

- 通常不能简单把分布式锁等同于强一致。它能帮助你减少并发冲突，但在网络异常、锁过期、底座故障和业务多步骤执行时，仍然需要别的机制补足一致性。
- 分布式锁主要控制“谁能先执行”，但并不天然保证“执行结果一定全局强一致”。
- 比如锁保护住了入口，但业务内部仍可能跨库、跨服务、多步骤执行，这时候状态一致性还得靠事务、补偿、幂等等机制。

## 3.3 Redis 锁、数据库锁、ZooKeeper 锁分别适合什么场景？

### 核心讲解

- Redis 锁通常更轻、更快，适合高性能但可接受一定工程边界的场景；数据库锁适合和数据强绑定、规模不大、实现简单的场景；ZooKeeper 锁更偏强协调、顺序性和一致性要求更高的场景。
- Redis 锁的优势是实现简单、性能较高、接入门槛低，很多互联网业务都爱用。
- 数据库锁的优势是和数据更新天然靠近，特别是本来就要落库的场景，有时用唯一键或原子更新更直接。
- ZooKeeper 这类协调组件更擅长做严格协调和有序感知，但性能和复杂度通常高于 Redis。

## 3.4 为什么很多场景其实不该先想到分布式锁？

### 核心讲解

- 因为很多问题并不是“并发进入”导致的，而是“结果重复”“数据重复”“流量过高”导致的。这类问题往往用幂等、唯一约束、状态机或消息削峰更直接、更稳。
- 很多人一看到“并发”就先想上锁，但并发问题的根因并不总是同一时刻多人进入临界区。
- 如果问题是防重复下单，幂等号和唯一键可能更直接。
- 如果问题是库存热点，数据库原子扣减或队列串行化可能比所有人抢锁更稳。
- 如果问题是任务重复调度，调度中心的去重和状态控制可能比锁更自然。

## 3.5 分布式锁和幂等怎么配合？

### 核心讲解

- 更稳的设计通常不是“锁和幂等二选一”，而是让锁控制并发进入，让幂等控制重复结果。这样即使锁边界失守，结果也更容易守住。
- 锁主要解决“同一时刻谁先处理”，适合入口并发控制。
- 幂等主要解决“重复处理会不会把结果做乱”，适合结果兜底。
- 在真实系统里，仅靠锁往往不够，因为锁可能过期、误删、失效或被绕过。
- 如果同时有幂等机制，即使同一个请求真的重复执行，结果也更容易收敛。
- 所以锁和幂等通常不是对立关系，而是前后两道防线。

### 深入理解

- 这题真正要理解的，不只是结论本身，还要理解“分布式锁和幂等怎么配合？”背后的原理、适用边界和工程取舍。
- 学习时建议把它和本模块前后题串起来看：它解决什么问题、在什么场景下最有价值、又会带来什么代价或限制。
- 如果这题偏原理，就重点理解数据流转、执行路径或状态变化；如果这题偏设计，就重点理解业务约束、方案选择和失败场景。
- 真正掌握这题的标志，不是能背定义，而是能用一个具体业务场景把它讲顺。

### 追问答案

- 只做幂等不做锁行不行？
  有些场景可以，但如果临界区本身就不想被并发进入，还是要考虑锁。
- 只做锁不做幂等行不行？
  风险较大，因为锁不能覆盖所有异常和重复执行。
- 更稳的组合方式是什么？
  锁做入口并发控制，幂等做结果兜底。

### 学习提示

- 这题特别适合你建立“多道防线”的设计意识。
- 真正成熟的系统，通常不会只靠一道锁来赌正确性。

## 3.6 分布式锁和消息队列怎么配合？

### 核心讲解

- 消息队列更擅长削峰、解耦和串行化某类处理；分布式锁更擅长保护临界区。两者可以配合，但很多热点高并发场景里，队列往往比大家抢锁更自然。
- 如果问题是瞬时高并发都来争一个资源，直接上锁往往会导致大量请求竞争、等待和超时。
- 这时候消息队列可以先把流量削下来，再由消费者按一定顺序处理。
- 锁在这里更适合保护某个临界步骤，而不是承担全部并发收敛压力。
- 所以消息队列和锁不是替代关系，而是可以组合：队列负责平滑流量，锁负责关键步骤串行化。

- 这题真正要理解的，不只是结论本身，还要理解“分布式锁和消息队列怎么配合？”背后的原理、适用边界和工程取舍。

### 追问答案

- 为什么有些秒杀场景更爱队列？
  因为它先把并发峰值平滑掉，比所有请求一起抢锁更稳。
- 锁和队列谁更重？
  不好一概而论，但在高并发热点场景下，队列通常更适合做流量整形。
- 面试里怎么答更实战？

### 学习提示

- 学这题时可以把锁和队列分别理解成“并发控制”和“流量控制”。
- 很多系统设计题里，这两个会同时出现。

## 3.7 为什么“有锁”不等于“系统就安全了”？

### 核心讲解

- 因为锁只能保护一部分并发进入路径，不能自动保证业务结果正确、状态一致、系统高可用和异常恢复完整。真正安全的系统，通常还要靠幂等、事务、监控和补偿一起配合。

- 再者，很多问题根本不是锁的问题，比如业务重复、消息重复、发布配置错误、数据库状态异常。

## 4.2 秒杀、库存扣减场景一定要用分布式锁吗？

## 4.3 如何避免把分布式锁用成性能瓶颈？

### 核心讲解

- 关键思路通常有 4 个：降低锁粒度、减少持锁时间、避免热点集中、尽量让锁只保护真正必要的临界区。
- 第一，锁粒度不能太粗，不然很多本来可以并行的请求都会被串行化。
- 第二，持锁逻辑要尽量短，不要把 RPC、复杂查询、慢 I/O 都放在锁里。
- 第三，要尽量避免所有请求争同一把锁，比如按资源拆分锁 key、按分片隔离。
- 第四，如果场景本来更适合队列、幂等或数据库原子更新，就别用锁去扛全部压力。
- 所以避免锁成瓶颈，核心不是“换一个更快的锁”，而是减少不必要的锁竞争。

### 深入理解

- 这题真正要理解的，不只是结论本身，还要理解“如何避免把分布式锁用成性能瓶颈？”背后的原理、适用边界和工程取舍。

### 追问答案

  因为持锁越久，后面的等待和竞争就越严重。
- 锁粒度太粗会怎样？
  会把大量本来能并行的请求强行串行。
- 优化锁最常见的第一步是什么？
  先看临界区里到底有没有不该放进去的逻辑。

### 学习提示

## 4.4 锁竞争激烈时怎么优化？

### 核心讲解

- 更稳的优化方向通常是：先判断是不是误用锁，再看能不能拆分资源、分片、削峰、降级或改成幂等和状态控制，而不是一味调参数。
- 如果锁竞争激烈，第一步不是先调 Redis，而是先确认这个场景是不是真的必须上锁。
- 如果必须上锁，再看资源是否能拆分，比如按用户、订单、商品、分片粒度拆 key。
- 如果流量太集中，队列削峰、限流和预扣模型往往比大家一起抢锁更稳。
- 如果只是防重问题，幂等和唯一约束通常更直接。
- 所以锁竞争优化，本质上是系统设计优化，不只是调 `ttl` 或重试参数。

## 4.5 如何排查分布式锁导致的线上问题？

### 核心讲解

- 更稳的排查顺序通常是：先看现象，再看锁竞争和等待情况，再看 Redis 状态和业务时序，最后确认是不是锁误用或锁释放异常。
- 第一步先看业务现象，是 RT 变高、吞吐下降、重复执行，还是任务不执行。
- 第二步看锁相关指标，比如获取锁失败率、等待时间、重试次数、持锁时间。
- 第三步再看 Redis 本身有没有慢查询、CPU 飙高、热 key、网络抖动或主从切换。
- 第四步回到业务代码，确认临界区是不是太大、TTL 是否合理、释放逻辑有没有误删风险。
- 很多线上锁问题，最终都能归到三类：竞争太激烈、锁实现有边界、场景本来不该用锁。

## 4.6 锁服务本身挂了怎么办？

### 核心讲解

- 这要看你用的锁底座是什么。核心思路通常是：先接受锁服务也可能故障，再设计降级、兜底和恢复策略，而不是假设锁服务永远可用。
- 如果锁基于 Redis，那 Redis 故障、主从切换、网络抖动都可能影响锁语义。
- 这时候系统要先明确策略：是短期失败直接拒绝、降级、排队，还是允许某些业务退化执行。
- 对关键场景，还要设计多层防线，比如锁失败后仍然有幂等、唯一约束和状态机兜底。
